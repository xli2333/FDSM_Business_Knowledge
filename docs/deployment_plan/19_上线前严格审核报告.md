## 1. P0 阻塞问题（必须修复，否则禁止上线）

### P0-1. Redis 无密码、无 ACL、无 TLS

**位置**：`docker-compose.prod.yml:117-130`

```yaml
redis:
  image: redis:7.4-alpine
  command: ["redis-server", "--appendonly", "yes", "--maxmemory", "${REDIS_MAXMEMORY:-512mb}", "--maxmemory-policy", "allkeys-lru"]
```

**问题**：
- 无 `--requirepass` 或 ACL 配置，容器网络内任意服务 / 同宿主机容器 / 同 network 入侵的容器均可无密码连 Redis。
- `REDIS_URL: redis://redis:6379/0` 明文无鉴权（`backend/services/async_task_service.py:46`）。
- 异步任务队列（含可能含有用户内容 payload）可被未授权清空/污染。

**修复建议**：
```yaml
redis:
  environment:
    REDIS_PASSWORD: ${REDIS_PASSWORD}
  command: [
    "redis-server", "--appendonly", "yes",
    "--requirepass", "${REDIS_PASSWORD}",
    "--maxmemory", "${REDIS_MAXMEMORY:-512mb}",
    "--maxmemory-policy", "allkeys-lru"
  ]
```
并修改 `REDIS_URL=redis://:${REDIS_PASSWORD}@redis:6379/0`，密码由 secrets 注入。

### P0-2. Dockerfile 前端构建把 Supabase 密钥写入了镜像层

**位置**：`frontend/Dockerfile:12-22`、`docker-compose.prod.yml:97-101`

```
ARG VITE_SUPABASE_URL=
ARG VITE_SUPABASE_ANON_KEY=
ENV VITE_SUPABASE_URL=${VITE_SUPABASE_URL}
ENV VITE_SUPABASE_ANON_KEY=${VITE_SUPABASE_ANON_KEY}
RUN npm run build
```

**问题**：
- `ARG` + `ENV` 把敏感变量写进镜像 history；即使 build 后删除 `node_modules`，`docker history` 和 layer 仍可提取。
- Supabase anon key 虽然是公开 key，但 `VITE_ENABLE_DEBUG_AUTH` 也经相同通道，若误注入 `1` 会在生产前端开启 debug 头旁路（见 P0-3）。

**修复建议**：
- 只用 `ARG`，不用 `ENV`，Vite 构建时自然读取 `ARG` 经 `process.env`。
- 或改用 build-time 文件（`.env.production.local`）挂载到构建容器、不进 final image。
- 前端 `dist/` 里已经包含这些值（SPA 必然），但镜像不应再多留一份。

### P0-3. 前端生产构建未强制关闭 `VITE_ENABLE_DEBUG_AUTH`

**位置**：`frontend/src/api/index.js:7`

```js
const DEBUG_AUTH_HEADERS_ENABLED = import.meta.env.DEV && import.meta.env.VITE_ENABLE_DEBUG_AUTH !== '0'
```

**问题**：
- 虽然 `import.meta.env.DEV` 在 `vite build` 时为 `false`，依赖构建环境正确。
- 但 compose 的 `VITE_ENABLE_DEBUG_AUTH: ${VITE_ENABLE_DEBUG_AUTH:-0}` 允许值为 `1` 通过环境透传。若运维误设 `VITE_ENABLE_DEBUG_AUTH=1` 并 rebuild，生产仍会在 `import.meta.env.DEV` 为 false 时被短路保护——但代码只检查 `!== '0'`，并未显式校验生产模式。
- 同时后端 middleware 在 `APP_ENV=production` 才会拒绝 debug header。任一环节 `APP_ENV` 错配即导致越权。

**修复建议**：
- 在 `Dockerfile` / CI 中硬编码生产构建使用 `--mode production` 且 `VITE_ENABLE_DEBUG_AUTH` 强制为 `0`，或在 `main.jsx` 运行时检查 `import.meta.env.PROD` 并忽略 debug 头读取。
- 后端对所有 `/api/admin/*` 路由叠加独立 assert：`assert IS_PRODUCTION == (APP_ENV == 'production')`，启动时失败。

### P0-4. 业务数据库与前端容器共享 `./data` 卷，前端容器为 `ro` 但仍直接暴露 FAISS/上传文件

**位置**：`docker-compose.prod.yml:107-108`、`deploy/nginx/default.conf:40-56`

```yaml
frontend:
  volumes:
    - ./data:/data:ro
```
Nginx `alias /data/audio/`, `/data/uploads/editorial/`, `/data/uploads/media/`。

**问题**：
- `./data/` 整个目录挂载进前端容器（只读），**包括 `fudan_knowledge_base.db`、`fudan_knowledge_base.db-wal`、`Fudan_Business_Knowledge_Data/`、`faiss_index_business/`**。只读防篡改，但**容器逃逸或配置错误时，整库可被下载**。
- 更重要：Nginx 配置 `root /usr/share/nginx/html;` + `alias /data/xxx/`，若 `location` 未精确匹配，误配会导致 `/data/..` 穿越读取 704 MB 数据库。

**修复建议**：
- 拆分 `./data/uploads`、`./data/audio` 单独挂载为 `ro`，不挂整个 `./data`。
- `location` 显式用 `internal;` + `X-Accel-Redirect` 对付费内容鉴权，而不是裸 `alias`（目前 `/editorial-uploads/`、`/media-uploads/` **完全公开无鉴权**，付费会员专属音视频可被直链绕过）。

### P0-5. 付费音视频/编辑稿可绕过会员鉴权直链下载

**位置**：`deploy/nginx/default.conf:40-56`

```
location /audio-files/ { alias /data/audio/; }
location /editorial-uploads/ { alias /data/uploads/editorial/; }
location /media-uploads/ { alias /data/uploads/media/; }
```

**问题**：
- 会员鉴权在 API 层（`backend/routers/media.py:192`），但文件 URL 一旦泄漏即公开下载。
- `MediaPublicDetail` 返回 `media_url` 字段为 `/media-uploads/<kind>/media/<name>`，浏览器一次请求 API 拿到 URL 后，后续下载不过网关鉴权。
- 商用会员订阅（`paid_member_monthly=¥30`、`paid_member_yearly=¥300`）的付费壁垒实际形同虚设。

**修复建议**：
- 全部私有资源走 `X-Accel-Redirect`：前端请求 `/api/media/stream/<id>`，FastAPI 鉴权通过后返回 `X-Accel-Redirect: /internal-media/...`，Nginx 用 `internal;` location 真正出文件。
- 短期 mitigations：`media_url` 签名化（HMAC + 过期时间），每次播放重新签。

### P0-6. 无 HTTPS / Nginx 宿主机证书为占位示例

**位置**：`deploy/nginx/fdsmarticles-https.conf:11, 26-27`

```
server_name knowledge.example.edu;
ssl_certificate /etc/letsencrypt/live/knowledge.example.edu/fullchain.pem;
```

**问题**：
- 示例配置，无实际域名、无证书、无自动续签（`certbot renew` 定时任务）。
- 无 HSTS 头（`Strict-Transport-Security`）。
- CAS 协议要求严格 HTTPS，HTTP 回调 ticket 直接泄漏。
- TLS 协议白名单只写 `TLSv1.2 TLSv1.3`，但没有明确 cipher suite 限制，默认可能含弱密码。

**修复建议**：参见本报告 §3 HTTPS 完整清单。

### P0-7. 没有任何速率限制 / WAF / CC 防护

**位置**：全局缺失

**问题**：
- Nginx 配置无 `limit_req`、`limit_conn`、`limit_req_zone`。
- FastAPI 无 `slowapi` 或任何应用层限流。
- CAS 登录接口 `/api/auth/cas/login` 无限流：被刷会给 CAS 上游服务器造成灾难（学校 CAS 运维会先封我方 IP）。
- `/api/chat`（单次 10-90 秒 Gemini 调用、占 gunicorn worker）无限流：10 个并发请求即耗尽全部 worker。
- `/api/home/feed` 有 30 秒缓存，但**首次预热**时仍会穿透到 DB。
- 登录 `/api/auth/login` 明文密码验证无频率限制，可枚举撞库。

**修复建议**（Nginx 层）：
```nginx
limit_req_zone $binary_remote_addr zone=api_general:10m rate=60r/m;
limit_req_zone $binary_remote_addr zone=api_login:10m rate=5r/m;
limit_req_zone $binary_remote_addr zone=api_ai:10m rate=10r/m;
limit_conn_zone $binary_remote_addr zone=addr:10m;

location /api/ { limit_req zone=api_general burst=30 nodelay; limit_conn addr 20; ... }
location ~ ^/api/(auth|auth/cas/login) { limit_req zone=api_login burst=3 nodelay; ... }
location ~ ^/api/chat { limit_req zone=api_ai burst=5 nodelay; ... }
```

## 2. P1 高危问题（必须修复，否则首周必出事）

### P1-1. 容器无 CPU / 内存限制（OOM 会拖死整机）

**位置**：`docker-compose.prod.yml`（整个文件无 `deploy:` 块）

**问题**：
- 无 `deploy.resources.limits.cpus/memory`、无 `mem_limit`。
- FAISS 首次加载 + Gemini 调用 + 批处理容器（`backend-worker` 里还有 `backend/scripts/*`）一起吃内存，极易触发 OOMkiller 杀掉 `backend-web`。
- 16GB 机器：3 gunicorn worker × (200MB FAISS + ~400MB Python 模块) + worker ≈ 2.5GB，剩下给 SQLite 页缓存。但无限制时 Gemini SDK 偶发内存泄漏（`langchain-google-genai` 历史记录）会吃光所有剩余。

**修复建议**：
```yaml
backend-web:
  deploy:
    resources:
      limits: { cpus: '4.0', memory: 6G }
      reservations: { cpus: '1.0', memory: 2G }
backend-worker:
  deploy:
    resources:
      limits: { cpus: '2.0', memory: 4G }
redis:
  deploy:
    resources:
      limits: { cpus: '1.0', memory: 1G }
frontend:
  deploy:
    resources:
      limits: { cpus: '1.0', memory: 512M }
```

### P1-2. Gunicorn 进程数和超时配置不适合生产

**位置**：`Dockerfile.backend:45`

```
CMD ["gunicorn", "backend.main:app", "-k", "uvicorn.workers.UvicornWorker",
     "--workers", "3", "--timeout", "180", "--max-requests", "1000",
     "--max-requests-jitter", "100", ...]
```

**问题**：
- `--workers 3` 硬编码；应 `--workers ${GUNICORN_WORKERS:-$(2*CPU+1)}` 或按 `WEB_CONCURRENCY`。
- `--timeout 180` 对 AI 同步调用（未走异步的路由如 `/api/chat`）偏低，对普通 API 偏高（应 60s）。混用导致慢请求占 worker。
- 无 `--preload`：FAISS 和 langchain 模块每个 worker 独立加载 → 首次请求慢 5-10 秒、内存多占 2×200MB。
- 无 `--worker-tmp-dir /dev/shm`：容器 `/tmp` 在磁盘上，高并发下 heartbeat 文件 I/O 成为瓶颈。
- 无 `--access-logformat` 自定义，默认格式丢失 `X-Forwarded-For`、`X-Real-IP`，日志里全是 `172.x` 容器内网地址。

**修复建议**：
```
CMD ["gunicorn", "backend.main:app",
     "-k", "uvicorn.workers.UvicornWorker",
     "--bind", "0.0.0.0:8000",
     "--workers", "${GUNICORN_WORKERS:-5}",
     "--worker-connections", "1000",
     "--timeout", "90",
     "--graceful-timeout", "30",
     "--keep-alive", "5",
     "--max-requests", "500",
     "--max-requests-jitter", "50",
     "--worker-tmp-dir", "/dev/shm",
     "--preload",
     "--access-logformat", "%(h)s %({x-forwarded-for}i)s %(l)s %(u)s %(t)s \"%(r)s\" %(s)s %(b)s %(D)s \"%(f)s\" \"%(a)s\""
]
```

### P1-3. SQLite 在 4-8 并发写场景下仍然会成瓶颈

**位置**：`backend/database.py:40-47`

```python
def get_connection() -> sqlite3.Connection:
    connection = sqlite3.connect(SQLITE_DB_PATH, timeout=60)
    connection.execute("PRAGMA busy_timeout = 60000")
    connection.execute("PRAGMA journal_mode = WAL")
    connection.execute("PRAGMA synchronous = NORMAL")
    connection.execute("PRAGMA foreign_keys = ON")
```

**问题**：
- WAL 允许"N 读 + 1 写"，但 5 个 gunicorn worker + backend-worker + backup 容器全部并发写 → lock 重试（`run_write_transaction` 最多 5 次重试 × 2 秒，合计最高延迟 60 秒+）。
- 缺少 `PRAGMA cache_size` / `PRAGMA mmap_size` / `PRAGMA temp_store=MEMORY`，704MB DB 冷启动慢。
- backup 容器用 `sqlite3.backup()` API 会在大表上持锁（尽管是 read-only URI）。
- 压测报告（`17_最终压测性能预算与交付清单.md`）宣称 `80/80 完成，0 错误`，但 concurrency_write_probe 的 `hold-seconds 0.02` 是无业务逻辑的裸写，不代表 AI 总结/摘要并发写的真实情况。
- 文档声明 "SQLite 数据库超过 5 GB 或持续写 QPS 超过 50 再迁 PostgreSQL"，但目前 DB 已 704 MB 且含超大 `editorial_articles`，**半年内必超 5GB**。

**修复建议**：
```python
connection.execute("PRAGMA cache_size = -65536")      # 64 MB 页缓存
connection.execute("PRAGMA temp_store = MEMORY")
connection.execute("PRAGMA mmap_size = 536870912")    # 512 MB mmap
connection.execute("PRAGMA wal_autocheckpoint = 1000")
```
**强烈建议**：正式商用前规划 PostgreSQL 切换路径——`DATABASE_BACKEND=postgres` 字段已预留，但没有 Postgres adapter 代码，`backend/database.py:1794` 全部写的是 `sqlite3.connect`。

### P1-4. 上传文件类型只校验扩展名、不校验 MIME 真实类型

**位置**：`backend/services/media_service.py:2122`、`backend/services/image_upload_service.py:30-37`

**问题**：
- 只比对 `Path(safe_name).suffix.lower()` 与白名单；`content_type` 只记录不校验。
- 攻击者上传 `evil.mp3`（实际是 PHP Shell 内容）绕过。虽然服务不解释 `.mp3`，但：
  - 被 Nginx 直出 + 用户点击下载后，如果 OS 关联了同名扩展，仍可 RCE。
  - 更严重：`.html/.htm/.docx` 被 `ALLOWED_TEXT_UPLOAD_EXTENSIONS` 允许上传；`.html` 由浏览器访问直接执行 JS → **存储型 XSS，盗取 admin token**（token 存 localStorage `fdsm-cas-token`）。
- 无 `Content-Disposition: attachment`、无文件大小上限、无幻数（magic bytes）校验。

**修复建议**：
```python
import magic
MAX_UPLOAD_BYTES = {"audio": 200*1024*1024, "video": 2*1024*1024*1024, "cover": 10*1024*1024}

async def _validate_upload(upload: UploadFile, kind: str) -> bytes:
    head = await upload.read(2048)
    await upload.seek(0)
    detected = magic.from_buffer(head, mime=True)
    allowed_mimes = ALLOWED_MIMES[kind]
    if detected not in allowed_mimes:
        raise HTTPException(400, f"MIME mismatch: detected={detected}")
    return head
```
并在 Nginx 静态资源 location 加 `add_header Content-Disposition 'attachment';` 对 `.html` 等高危扩展强制下载而非解释。禁用 `.html/.htm` 在 upload 中。

### P1-5. 上传没有总大小限制，可被刷爆磁盘

**位置**：`backend/services/media_service.py:2084-2096`（流式 chunk 写入无累计检查）

**问题**：
- `_write_upload_stream` 写入时只累加 `total_bytes`，不在中途 break。
- Nginx `client_max_body_size 200m` 是单次上限，但 admin 上传视频（mp4）实际需要超过 200m → 若放宽，就失去保护。
- 单用户可连续上传 N 次，磁盘 200GB 半小时刷满。
- `ENOSPC` 会同时拖死 SQLite WAL checkpoint（磁盘满时写 commit 会失败）和 Redis append-only（断电安全丢失）。

**修复建议**：
- Nginx 差异化限制：`/api/media/admin/upload` 放宽到 2GB，其他 API 10MB。
- 应用层总配额表（per-user）：`SELECT SUM(size) FROM uploads WHERE user_id=? AND created_at > now()-1day`，超过阈值拒绝。
- backup 容器 `df` 监控 + Grafana 告警。

### P1-6. 无密钥轮换机制、`.env.production` 文件级明文存放

**位置**：`.env.production`（实际文件存在于仓库 git status 内，未 commit 但在工作区）

**问题**：
- 所有 Gemini API key（5 个轮询）、CAS 凭证、Supabase anon key 都在单个文件中，无 Docker secrets / Vault / SOPS 加密。
- 一旦服务器被入侵（SSH 弱密码 / 容器逃逸），**全部密钥一把抓**。
- 无密钥版本号、无过期时间、无调用审计。

**修复建议**：
- 使用 `docker compose secrets:`（file-based）或迁 HashiCorp Vault / 阿里云 KMS。
- Gemini key 每月轮换，调用失败自动切备 key 并告警。
- `.env.production` 文件权限必须 `chmod 600` + `chown root:root`。
- Git 仓库增加 `git-secrets` / `trufflehog` pre-commit hook 拦截误提交。

### P1-7. 日志无结构化、无轮转、无集中采集

**位置**：`Dockerfile.backend:45`（`--access-logfile - --error-logfile -`）

**问题**：
- Gunicorn 日志打 stdout，由 Docker json-log driver 直接落盘到宿主机 `/var/lib/docker/containers/*/*.log`。
- **无 `--log-opt max-size=10m --log-opt max-file=5`**：单容器日志可无限增长直到占满磁盘。
- 无 structured JSON logging，排障只能靠 grep。
- 无集中日志（ELK / Loki / 阿里云 SLS / 腾讯云 CLS），服务器一旦漂移，故障现场无法回溯。
- Python 代码里基本没有 `logging.getLogger(__name__)` 使用（只有 6 个文件），都是 `print(..., flush=True)` → 生产定位难。

**修复建议**：
```yaml
backend-web:
  logging:
    driver: json-file
    options:
      max-size: "10m"
      max-file: "5"
      tag: "fdsm-web"
```
- 采 `python-json-logger` 统一 JSON 格式，字段含 `request_id / user_id / route / latency_ms`。
- 接 Loki + Promtail（轻量）或阿里云 SLS agent。

### P1-8. 无 Prometheus / APM / Sentry，生产故障全凭肉眼

**位置**：`requirements.txt`（完全没有相关依赖）

**问题**：
- 无 `/metrics` 暴露（`prometheus-fastapi-instrumentator`）。
- 无 APM（OpenTelemetry / Datadog / Sentry SDK）。
- `backend/database.py` 有 slow query 观测（`DB_SLOW_QUERY_MS=750`）但只 `print` 到 stdout。
- CPU/内存/磁盘/Redis/连接数 → 全部无监控。
- 第一次客户投诉"网站慢"时，你没有任何手段定位瓶颈。

**修复建议**：
```python
from prometheus_fastapi_instrumentator import Instrumentator
Instrumentator().instrument(app).expose(app, endpoint="/metrics")
```
- 部署 `cadvisor` + `node_exporter` + `redis_exporter` + Grafana dashboard。
- 免费版 Sentry 接错误上报（免费额度 5k events/月足够单机用）。
- Nginx 启用 `stub_status`，Prometheus nginx_exporter 抓取。

### P1-9. FastAPI CORS `allow_headers=["*"]` + `allow_credentials=True` 是 CVE 级错误

**位置**：`backend/main.py:108-114`

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS or ["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**问题**：
- `allow_origins or ["*"]`：当 `ALLOWED_ORIGINS` 为空（配置错误），**自动降级为 `*`**。
- `allow_credentials=True` + `allow_origins=["*"]`（即使浏览器拒绝也破坏策略意图）。
- `allow_methods=["*"]` 包括 `DELETE`、`PUT`，CSRF 面变大。

**修复建议**：
```python
if not ALLOWED_ORIGINS:
    raise RuntimeError("ALLOWED_ORIGINS must be set in production")
app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS,
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "PATCH", "DELETE"],
    allow_headers=["Authorization", "Content-Type", "X-Visitor-Id", "X-Requested-With"],
    max_age=600,
)
```

### P1-10. 数据库备份没有异地 / 加密 / 恢复演练

**位置**：`deploy/backup_sqlite.py`、`docker-compose.prod.yml:132-146`

**问题**：
- 只在本机 `./backups` 目录复制 db 文件；无 S3/OSS 异地；服务器硬盘坏 = 数据归零。
- 备份文件明文，泄漏即整库泄漏（含所有用户邮箱、手机号、鉴权 session hash）。
- 保留 14 天；商用场景应至少 30 天 + 每周快照保留 6 个月。
- 无"恢复演练"流程，没有验证过备份文件真的能还原（`sqlite3 backup.db '.schema'` + 数据条数比对）。

**修复建议**：
- 备份 → gzip → GPG 加密（公钥加密，私钥异地） → rclone 推到阿里云 OSS / S3 冷归档。
- 每周自动化恢复脚本：拉最新备份 → 跑在 staging 容器 → `SELECT COUNT(*)` 比对生产 → 日志归档。
- RPO（数据丢失容忍）目标：1 小时；RTO（恢复时长）目标：2 小时。当前 `BACKUP_INTERVAL_SECONDS=86400`（24 小时）= RPO 24 小时，远未达商用。

### P1-11. `secrets/` 目录先创建再不用，目录本身就是坑

**位置**：`deploy/preflight.sh:19`

```bash
mkdir -p "$APP_DIR/data" "$APP_DIR/backups" "$APP_DIR/logs" "$APP_DIR/secrets"
```

**问题**：
- preflight 创建 `secrets/`，但 `docker-compose.prod.yml` 没有 `secrets:` 块引用；目录空置或人为乱塞。
- 整个 compose 用 `env_file: .env.production` 一把梭哈，没有 `secrets: [cas_ticket_cert, gemini_api_key]` 分项挂载。

**修复建议**：启用 compose secrets：
```yaml
secrets:
  gemini_api_keys:
    file: ./secrets/gemini_api_keys.txt
services:
  backend-web:
    secrets: [gemini_api_keys]
```
代码改从 `/run/secrets/gemini_api_keys` 读，不再依赖环境变量。

## 3. P2 商用强烈建议（不修复风险显著）

### P2-1. 无蓝绿 / 金丝雀 / 滚动升级，发布即停服

- 当前流程：`docker compose build` → `up -d` 重启容器 → 服务中断 10-40 秒。
- 建议：
  - 最简：利用 `deploy.update_config` 滚动。
  - 进阶：两组 compose（blue/green）+ Nginx upstream 切换。
  - 专业：Kubernetes + Ingress + Argo Rollouts（目标规模到了再上）。

### P2-2. 没有 CI/CD（目前完全手工）

- 仓库根目录无 `.github/workflows/`、无 `.gitlab-ci.yml`、无 `Jenkinsfile`。
- `deploy/create_release_package.ps1` 是 PowerShell，只能 Windows 本机跑。
- 无"push main → 自动构建 → 推镜像仓库 → 服务器 pull → 滚动重启 → 跑 acceptance_check"的完整流水线。
- 用户期望的"一键上传到云服务器的 docker" → 当前是"手动打包 zip → scp → ssh → 手动 build"。

**建议最小 CI**（GitHub Actions）：
```yaml
name: deploy-prod
on: { push: { tags: ['v*'] } }
jobs:
  build:
    runs-on: ubuntu-22.04
    steps:
      - uses: actions/checkout@v4
      - run: docker build -t registry.cn-shanghai.aliyuncs.com/fdsm/backend:${{github.ref_name}} -f Dockerfile.backend .
      - run: docker build -t registry.cn-shanghai.aliyuncs.com/fdsm/frontend:${{github.ref_name}} -f frontend/Dockerfile .
      - run: docker push ...
  deploy:
    needs: build
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.PROD_HOST }}
          script: |
            cd /srv/fdsm/app
            docker compose pull
            docker compose up -d
            bash deploy/acceptance_check.sh
```

### P2-3. Frontend Dockerfile 镜像不是多阶段最优

**位置**：`frontend/Dockerfile:3`（`node:20-bookworm-slim` 基础镜像 600MB+）
- builder 阶段 OK（多阶段已用）。
- runtime 阶段是 `nginx:1.27-alpine`（20MB），这部分 ✓。
- 但 builder 阶段未锁死 npm 版本，`npm ci` 拉最新包，构建不可重现。建议 `engines` 指定 Node 版本 + `package-lock.json` 已入仓（✓）。

### P2-4. 后端 Dockerfile 装了完整 Node + npm（仅为 wechat_runtime）

**位置**：`Dockerfile.backend:3, 17-21`

- 从 `node:20-bookworm-slim` copy node binary + 全部 `node_modules/npm` 到 Python 镜像，镜像膨胀 ~200MB。
- `wechat_runtime` 只被 `backend/scripts/wechat_fudan_bridge.mjs` 调用，且 `sharp@0.34.5` 本身就占 ~80MB。
- 建议：把 `wechat_runtime` 拆到独立容器（微信发布时才启动），或使用 `distroless/nodejs` 组合。

### P2-5. 健康检查只检查"进程活"不检查"依赖活"

**位置**：`docker-compose.prod.yml:41-46`

```yaml
healthcheck:
  test: ["CMD", "python", "-c", "import urllib.request; urllib.request.urlopen('http://127.0.0.1:8000/api/health', timeout=5).read()"]
```

- `/api/health` 只在 `?details=1` 时才查 DB。默认模式只返回 200。
- Redis 挂了 / FAISS 索引损坏 / 磁盘只读 → backend-web 仍然 "healthy"。
- 建议改 `healthcheck` URL 为 `/api/health?details=1` 或新增 `/api/ready`（readiness）端点：db_ok + redis_ok + faiss_ok 三项全绿才 200。

### P2-6. Nginx 静态缓存策略未按 hash 区分

**位置**：`deploy/nginx/default.conf:58-62`

```
location ~* \.(?:js|css|png|jpg|...)$ {
    add_header Cache-Control "public, max-age=2592000, immutable";
}
```

- Vite 默认产物带 hash（`index-abc123.js`），可 `immutable`。但 `index.html` 没被排除，上面 `try_files` 命中 `/` 的 `index.html` 默认不缓存 ✓ 但未显式声明。
- 建议 `index.html` 显式 `Cache-Control: no-cache`，否则客户端缓存一小时版本错位。

### P2-7. 异步任务队列无死信 / 毒丸处理

**位置**：`backend/services/async_task_service.py:274-305`

- `run_worker_loop` 只在 `exception` 时 `time.sleep(1)` 继续。
- 无 `max_attempts` 重试后入死信队列；失败任务直接 `fail_async_task` 不重试。
- 毒丸任务（比如触发 OOM 的任务）每次 worker 重启都会被 claim → 又 crash → 无限循环。

**修复建议**：
- Redis `rpoplpush` 搬 `fdsm:async-tasks:processing`，完成才删；超时未完成回归主队列，3 次失败入死信。
- 死信队列有告警。

### P2-8. CAS `auth_sessions` 表无清理 job

**位置**：`backend/database.py:284-300`（建表 + 索引 OK）、`cas_auth_service.py`（无定期清理）

- 会话表只增不删，expired session 永远留在表里。
- 访问量 1000 DAU × 每天 3 次登录 × 365 天 = 100 万行，SQLite 查询变慢。
- 建议 backup 容器或新增 cron 容器每天 `DELETE FROM auth_sessions WHERE expires_at < datetime('now', '-30 days')`。

### P2-9. Supabase 仍未退场，双鉴权并存增加攻击面

**位置**：`.env.production.example:32-33`（Supabase 字段保留）、`backend/services/auth_service.py`（`dual` 模式）

- 上线目标是复旦 CAS，但代码中 Supabase 路径完整保留且 `AUTH_BACKEND=auto` 在某些配置下会回退 Supabase。
- 双鉴权 = 双漏洞面。建议 CAS 上线首周即硬编码 `AUTH_BACKEND=cas`，注释掉 `supabase_auth_service` 路径（至少 import guard）。

## 5. 高并发支撑能力评估

| 指标 | 当前能力 | 商用目标 | 差距 |
|---|---|---|---|
| 并发用户（读为主） | ~50（单机 3 worker + 30s 缓存） | 500 | 10× |
| 并发用户（写） | ~8-10（SQLite WAL 单写） | 100 | 10× |
| AI 任务 QPS | 异步队列 1 worker × 10-90s/任务 = 0.02 QPS | 1 QPS | 50× |
| RPO（数据丢失） | 24 小时 | ≤1 小时 | 24× |
| RTO（恢复时长） | 未演练 | ≤2 小时 | 未知 |
| 可用性 SLA | 无 | 99.5% | 无法承诺 |

**结论**：当前架构设计只能扛日 UV < 500、峰值并发 < 30 的内测场景。要商用高并发必须：
1. **数据库**：SQLite → PostgreSQL（或至少只读副本 + FAISS 外置到 pgvector / Qdrant）。
2. **应用层**：水平扩容（backend-web × 3 实例 + Nginx upstream 负载均衡），Redis 主从。
3. **CDN**：静态资源 + 音视频 → 阿里云 OSS + CDN。
4. **AI 推理**：Gemini 调用走异步 + 超时分级 + 结果 Redis 缓存（同题目不重复调用）。
