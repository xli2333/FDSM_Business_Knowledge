## 背景与目标

### 为什么要改

#### 问题 1：每次鉴权都远程调用

`backend/services/supabase_auth_service.py:55-63` 现在这样验证 token：
```python
response = requests.get(
    _user_endpoint(),
    headers={"apikey": SUPABASE_ANON_KEY, "Authorization": f"Bearer {access_token}"},
    timeout=SUPABASE_AUTH_TIMEOUT_SECONDS,  # 8 秒
)
```

每次带 token 的 API 都要多 100-300ms 的远程调用，且有 8 秒超时风险。多 worker + 高并发下 Supabase 还可能限流。

#### 问题 2：业务代码强耦合 Supabase UUID

当前 `get_authenticated_user` 返回 `{"id": supabase_user_uuid, "email": ...}`，业务代码里 `user_memberships`、`user_saved_articles`、`user_follows` 等表都用这个 UUID 当外键。

**这对 CAS 迁移是个灾难**：
- CAS 用户根本没有 Supabase UUID
- 已有用户切换到 CAS 后身份 ID 会变，关联数据全丢
- 数据库 schema 要大改

**正确姿势**：所有业务表的 user_id 都引用**本地 `business_users.user_id`**（随机 UUID）。`business_users` 里存外部 ID 的映射列（`supabase_user_id` / `cas_employee_number` / `cas_username`）。未来切 CAS 只是换一种方式找到本地 user 记录，业务外键零改动。

#### 问题 3：注册用户没有落地记录

用户通过 Supabase 注册/登录后，**不会自动在我们 `business_users` 表里有记录**。项目的 `get_membership_profile`、`get_business_profile` 等函数遇到没见过的 user_id 会走 fallback 路径（返回默认免费会员的假数据）。这导致：
- 无法 UPDATE 用户的 tier（admin 升级失败）
- 无法查"有哪些用户已注册"
- 统计指标失真

必须在首次登录时写一条 `business_users` 记录，后续所有操作走本地主键。

## 原子步骤

### 步骤 1 · 在 `database.py` 新增补列函数

**打开 `backend/database.py`**，找到 `ensure_runtime_tables` 函数（第 64 行左右开始）。

**在 `ensure_runtime_tables` 函数外面（但在文件里）新增一个辅助函数**。建议放在 `_table_has_column` 之后：

```python
def _ensure_business_users_columns(connection: sqlite3.Connection) -> None:
    """为未来迁移 CAS 预留 business_users 列。幂等：重复调用安全。

    新增列：
      - supabase_user_id: 存 Supabase auth.users.id（UUID），建唯一索引
      - cas_employee_number: 存复旦 CAS 工号，建唯一索引
      - cas_username: 存 CAS 返回的 username（备份，不建索引）

    这些列都允许 NULL。同一个用户可以既有 supabase_user_id 也有 cas_employee_number
    （双通道过渡期）。
    """
    if not _table_has_column(connection, "business_users", "supabase_user_id"):
        connection.execute("ALTER TABLE business_users ADD COLUMN supabase_user_id TEXT")
    if not _table_has_column(connection, "business_users", "cas_employee_number"):
        connection.execute("ALTER TABLE business_users ADD COLUMN cas_employee_number TEXT")
    if not _table_has_column(connection, "business_users", "cas_username"):
        connection.execute("ALTER TABLE business_users ADD COLUMN cas_username TEXT")

    # WHERE 子句让 NULL 不进索引，避免多个 NULL 冲突
    connection.execute(
        "CREATE UNIQUE INDEX IF NOT EXISTS idx_business_users_supabase_user_id "
        "ON business_users(supabase_user_id) WHERE supabase_user_id IS NOT NULL"
    )
    connection.execute(
        "CREATE UNIQUE INDEX IF NOT EXISTS idx_business_users_cas_employee "
        "ON business_users(cas_employee_number) WHERE cas_employee_number IS NOT NULL"
    )
```

**关键点解释**：
- `_table_has_column` 是项目已有的辅助函数（第 52 行），检查列是否存在
- SQLite 的 `ALTER TABLE ADD COLUMN` 不支持 `IF NOT EXISTS`（3.35+ 才支持，安全起见用 `_table_has_column` 判断）
- 索引用 **部分索引**（`WHERE ... IS NOT NULL`）：允许多条记录该列都为 NULL（ `supabase_user_id` 没填时不冲突）
- 列名加前缀 `supabase_` 和 `cas_` 是为了区分不同身份源

### 步骤 2 · 接入 ensure_runtime_tables

**找到 `ensure_runtime_tables` 函数末尾**（具体行号因版本而异，搜 `def ensure_runtime_tables`）。

函数结构大概是：
```python
def ensure_runtime_tables() -> None:
    with connection_scope() as connection:
        connection.executescript(
            """
            CREATE TABLE IF NOT EXISTS ...
            CREATE TABLE IF NOT EXISTS business_users (...);
            ...
            """
        )
        # ... 各种 _table_has_column 判断和 ALTER TABLE
```

**在函数末尾（connection 还打开时），`connection.commit()` 之前**，加一行：
```python
        _ensure_business_users_columns(connection)
```

**如果这个函数最后没有 `connection.commit()`**（因为 `isolation_level=None` 后手动控制事务），要确认 `executescript` 后面的修改被提交。最稳的做法：
```python
def ensure_runtime_tables() -> None:
    with connection_scope() as connection:
        connection.executescript("""... 原有建表 SQL ...""")
        # ... 原有的 _table_has_column 升级 ...
        _ensure_business_users_columns(connection)     # 新增这一行
        connection.commit()                            # 确保提交（原来如果没这行就加上）
```

### 步骤 4 · `supabase_auth_service.py` 加缓存层

**打开 `backend/services/supabase_auth_service.py`**。

**文件顶部**（`from functools import lru_cache` 之后，`import requests` 之前）**加**：
```python
import time
from threading import Lock

# ------- 内存缓存 -------
# 结构：{access_token: (cached_at_timestamp, user_dict_or_None)}
# 容量控制：超过 _AUTH_CACHE_MAX 条清理一半（LRU-ish）
_AUTH_CACHE: dict[str, tuple[float, dict | None]] = {}
_AUTH_CACHE_LOCK = Lock()
_AUTH_CACHE_TTL = 300          # 5 分钟（Supabase access_token 本身 1 小时过期）
_AUTH_CACHE_MAX = 10000        # 超过这个阈值就清理
```

**为什么 TTL 是 5 分钟**：
- Supabase access_token 默认 1 小时过期
- 5 分钟内如果用户被后台 revoke（封号、改密），最多 5 分钟后感知到
- 权衡：更长的 TTL 命中率更高但撤销延迟；更短 TTL 反之
- 5 分钟是常见中位值

**容量清理**：当缓存 > 10000 条，按 `cached_at` 升序删掉最老的 5000 条。简单粗暴，够用。

### 步骤 5 · 新增 `_ensure_local_user_from_supabase`

**文件末尾加**：
```python
def _ensure_local_user_from_supabase(supabase_user: dict) -> dict:
    """首次见到这个 Supabase user 时，在 business_users 建一条本地记录；
    已见过的用户直接返回本地 user_id + email。

    返回值结构：
        {"id": 本地 user_id, "email": ..., "raw_user": 原始 Supabase payload}

    "id" 字段永远是 business_users.user_id（本地主键），不是 Supabase UUID。
    这是 CAS 迁移时"业务代码零改动"的关键。
    """
    from backend.database import connection_scope

    supabase_uid = supabase_user["id"]
    email = supabase_user.get("email")

    with connection_scope() as conn:
        # 路径 A：已经有绑定记录
        row = conn.execute(
            "SELECT user_id, email FROM business_users WHERE supabase_user_id = ?",
            (supabase_uid,),
        ).fetchone()
        if row:
            return {
                "id": row["user_id"],
                "email": row["email"] or email,
                "raw_user": supabase_user,
            }

        # 路径 B：同邮箱的 preview/seed 账号存在，把它绑定到 Supabase
        if email:
            row = conn.execute(
                "SELECT user_id FROM business_users WHERE lower(email) = lower(?)",
                (email,),
            ).fetchone()
            if row:
                conn.execute(
                    "UPDATE business_users SET supabase_user_id = ?, auth_source = 'supabase', "
                    "updated_at = ?, last_seen_at = ? WHERE user_id = ?",
                    (supabase_uid, _now_iso(), _now_iso(), row["user_id"]),
                )
                conn.commit()
                return {
                    "id": row["user_id"],
                    "email": email,
                    "raw_user": supabase_user,
                }

        # 路径 C：全新用户，插入一条
        import uuid
        now_iso = _now_iso()
        # local user_id 保留一点可读性：前缀 + Supabase UID 前 8 位 + 随机 8 位
        local_id = f"sb_{supabase_uid.replace('-', '')[:8]}_{uuid.uuid4().hex[:8]}"
        display_name = (email or local_id).split("@")[0]

        conn.execute(
            """
            INSERT INTO business_users(
                user_id, email, display_name, title, organization, bio, description,
                tier, status, role_home_path, auth_source, locale, is_seed,
                supabase_user_id,
                created_at, updated_at, last_seen_at
            ) VALUES (
                ?, ?, ?, NULL, NULL, NULL, NULL,
                'free_member', 'active', '/me', 'supabase', 'zh-CN', 0,
                ?,
                ?, ?, ?
            )
            """,
            (local_id, email, display_name, supabase_uid, now_iso, now_iso, now_iso),
        )
        conn.commit()
        return {
            "id": local_id,
            "email": email,
            "raw_user": supabase_user,
        }


def _now_iso() -> str:
    """文件内小工具。如果文件顶部已有等价函数，可直接复用。"""
    from datetime import datetime
    return datetime.now().replace(microsecond=0).isoformat()
```

**为什么 local_id 形如 `sb_xxx_yyy`**：
- 前缀 `sb_` 表示来源是 Supabase（未来 CAS 用 `cas_`、seed 账号用 `mock-` 继续保留）
- Supabase UID 前 8 位给人一点辨识度（排查问题时能快速定位）
- 末尾 8 位随机 hex 防撞（Supabase UID 完全不撞，但保险）

**三条路径的意图**：
- A：老用户回访，直接命中
- B：项目可能有预埋的种子账号（`mock-admin` 等），或者管理员先在 `business_users` 手工创了一条，再让用户注册。通过邮箱匹配绑定
- C：全新用户

**注意**：业务代码里对 `user_id` 的 SQL 查询都用 `business_users.user_id`，所以 B 和 C 生成的记录对所有已有查询完全透明。

### 步骤 6 · 改 `get_authenticated_user`

**找到现在的 `get_authenticated_user`**（第 43 行附近）。

**原代码**：
```python
def get_authenticated_user(
    authorization: str | None,
    *,
    debug_user_id: str | None = None,
    debug_user_email: str | None = None,
) -> dict | None:
    if not is_supabase_auth_enabled():
        return _build_debug_user(debug_user_id, debug_user_email)

    if response.status_code != 200:

    payload = response.json()
    user_id = payload.get("id")
```

**整段替换为**：
```python
def get_authenticated_user(
    authorization: str | None,
    *,
    debug_user_id: str | None = None,
    debug_user_email: str | None = None,
) -> dict | None:
    if not is_supabase_auth_enabled():
        return _build_debug_user(debug_user_id, debug_user_email)
    access_token = _extract_bearer_token(authorization)
    if not access_token:
        return _build_debug_user(debug_user_id, debug_user_email)

    # ---- 缓存命中检查 ----
    now = time.time()
    with _AUTH_CACHE_LOCK:
        entry = _AUTH_CACHE.get(access_token)
        if entry is not None and (now - entry[0]) < _AUTH_CACHE_TTL:
            return entry[1]  # 可能是 user dict 也可能是 None（上次验证失败也缓存）

    # ---- Miss，远程验证 ----
    try:
        response = requests.get(
            _user_endpoint(),
            headers={
                "apikey": SUPABASE_ANON_KEY,
                "Authorization": f"Bearer {access_token}",
            },
            timeout=SUPABASE_AUTH_TIMEOUT_SECONDS,
        )
    except requests.RequestException:
        return None      # 网络失败不缓存（下次重试）

    if response.status_code != 200:
        # 缓存失败结果，避免对无效 token 反复查询 Supabase
        _cache_set(access_token, None)
        return None

    payload = response.json()
    supabase_uid = payload.get("id")
    if not supabase_uid:
        _cache_set(access_token, None)
        return None

    # ---- 绑定到本地 business_users ----
    local_user = _ensure_local_user_from_supabase({
        "id": supabase_uid,
        "email": payload.get("email"),
        "raw_user": payload,
    })

    _cache_set(access_token, local_user)
    return local_user


def _cache_set(access_token: str, value: dict | None) -> None:
    """把 (token → user 或 None) 放进缓存，溢出时清理。"""
    now = time.time()
    with _AUTH_CACHE_LOCK:
        _AUTH_CACHE[access_token] = (now, value)
        if len(_AUTH_CACHE) > _AUTH_CACHE_MAX:
            # 按 cached_at 升序，删最老的一半
            sorted_keys = sorted(_AUTH_CACHE.keys(), key=lambda k: _AUTH_CACHE[k][0])
            for k in sorted_keys[: _AUTH_CACHE_MAX // 2]:
                _AUTH_CACHE.pop(k, None)


def clear_auth_cache() -> None:
    """管理接口：清空缓存（改完用户权限后用）。"""
    with _AUTH_CACHE_LOCK:
        _AUTH_CACHE.clear()
```

**关键设计**：
- 失败结果也缓存（但不缓存网络错误）：避免攻击者用一个无效 token 反复打垮 Supabase 限流
- `clear_auth_cache()` 导出给管理接口用（admin 改完某人的 tier 后调一下，强制下次重新走 Supabase 刷新）
- `_ensure_local_user_from_supabase` 每次 miss 都调用——但因为 DB 里已经有记录，是 SELECT-only，很快

## 常见错误与排查

| 症状 | 原因 | 修复 |
|---|---|---|
| `no such column: supabase_user_id` | 迁移没跑 | 显式 `python -c "from backend.database import ensure_runtime_tables; ensure_runtime_tables()"` |
| `UNIQUE constraint failed: business_users.supabase_user_id` | 部分索引没生效（旧 SQLite） | 确认 `sqlite3.sqlite_version >= "3.8.0"`（支持部分索引）|
| 缓存命中率 0% | `_AUTH_CACHE` 每次 import 都重新创建 | 检查是不是用 uvicorn `--reload` 跑的（reload 会重载模块）|
| 首次登录没建档 | `_ensure_local_user_from_supabase` 里的 commit 失败 | 加日志，看看是哪条 SQL 报错；多半是 `INSERT` 漏了某个 NOT NULL 列 |
| 业务接口返回 500 找不到 user | business_users 里 tier='admin' 但 code 读的是 supabase UUID | 看 `get_membership_profile` 是否按 `user_id` 而不是 Supabase UUID 查 |
| pytest 失败：mock user 走不通 | `_build_debug_user` 返回的 id 是 `mock-admin`（没经过建档）| 测试断言如果要求 `id.startswith("sb_")`，要在 mock 场景下放宽 |

## 关于多 worker 下的缓存

每个 gunicorn worker 是独立进程，`_AUTH_CACHE` 不共享。一个 token 在 worker A 缓存了，worker B 还要重新验一次。

**这是刻意的设计**：
- 共享缓存要引入 Redis，多一层复杂度
- 每 worker 独立缓存命中率约 75%（假设 4 worker、同一用户连续请求会轮询到各 worker）
- 命中率 75% 已经足够——miss 也只是多一次 Supabase 调用（100-300ms）

**何时升级到 Redis 共享缓存**：
- QPS 超过 50，且 miss 带来的 Supabase 费用增加
- 出现 Supabase rate limit 告警
- 希望用户封号后全局秒级生效（目前最多 5 分钟）
