# 20 上线前严格审核整改记录

> 本文承接 `19_上线前严格审核报告.md`。结论是：报告方向合理，P0/P1 里能在代码和容器配置内完成的项已经转成本轮整改；必须依赖真实云资源、域名、证书、CAS 白名单、异地备份和合规流程的项，改为上线前硬门槛，不允许在本机标记为“已完成”。

## 已代码化整改

- Redis：`docker-compose.prod.yml` 要求 `REDIS_PASSWORD`，`REDIS_URL` 使用带密码形式，Redis healthcheck 同步鉴权。
- 前端构建：`frontend/Dockerfile` 强制 `VITE_ENABLE_DEBUG_AUTH=0`，生产构建不再接受误传 debug auth。
- 挂载面：frontend 容器不再挂载整个 `./data`，只读挂载 `audio`、`uploads/editorial`、`uploads/media`。
- 媒体访问：公开接口返回鉴权流地址 `/api/media/{kind}/{slug}/stream`，Nginx 使用 `internal` location 承接 `X-Accel-Redirect`，付费音视频不再通过裸直链开放。
- Nginx：补充 API/auth/chat 限流、连接限制、HSTS、安全头、`index.html` no-cache、音视频和脚本/字幕 internal 规则。
- 后端安全：生产 CORS 必须显式白名单，debug auth header 在生产直接拒绝。
- 健康检查：新增 `/api/ready`，同时检查 SQLite 和 Redis 队列；容器 healthcheck 切到 readiness。
- SQLite：补齐 `cache_size`、`temp_store`、`mmap_size`、`wal_autocheckpoint`。
- 上传：增加音视频、图片、文本大小上限；音视频和图片做基础魔数校验；上传入口禁 `.html/.htm` 和 HTML 内容。
- 异步任务：AI 任务默认 3 次尝试，最终失败进入 dead-letter 队列。
- CAS：启动阶段清理过期/已撤销 session，保留期由 `CAS_SESSION_RETENTION_DAYS` 控制。
- 容器治理：生产 compose 增加资源限制、日志轮转、Gunicorn 参数环境变量。
- preflight：生产环境强校验 HTTPS 域名、CORS 白名单、Redis 密码、debug auth、CAS、Supabase 残留、备份加密/异地配置、`.env.production` 权限。

## 仍需真实云端完成的硬门槛

以下项不能在本地伪造完成，真实上线前必须全部通过：

- 域名和 HTTPS：`SITE_BASE_URL`、`ALLOWED_ORIGINS`、宿主机 Nginx `server_name`、证书路径必须全部换成真实域名，且 `curl -I https://域名` 能看到 HSTS。
- CAS：学校 CAS 必须完成服务地址白名单；`CAS_SERVICE_BASE_URL` 与回调 URL 必须和真实域名一致；管理员工号或用户名必须填入生产白名单。
- Secret：`.env.production` 不进 Git，权限必须 `600` 或 `640`；`REDIS_PASSWORD` 必须是生产随机长密码，禁止示例值。
- 备份：必须配置异地目标和加密接收方，完成一次备份、一次下载、一次恢复演练，并记录恢复耗时。
- 监控：宿主机、容器、Nginx、磁盘、Redis、SQLite WAL、worker dead-letter 必须有告警。
- 安全边界：云安全组只开放 80/443/SSH 管理入口；后台管理入口必须绑定 CAS 管理员身份。
- 压测：上线前用真实云规格跑读流量、写探针、媒体鉴权流、worker 任务回归；结果必须记录到新的服务器验收文档。
- 回滚：必须演练发布包回滚、数据库备份恢复和 Nginx 配置回退。
- PostgreSQL 边界：SQLite 只允许灰度和中低并发；若进入持续高写入、多人后台并发运营或公网大流量，必须切 PostgreSQL/托管数据库。

## 上云执行顺序

1. 本地完成 `python -m compileall backend deploy`、`npm run build`、`docker compose config`、Docker 重建和本地压测。
2. 生成发布包，上传到私有云固定目录。
3. 在云端复制 `.env.production.example` 为 `.env.production`，填真实域名、CAS、Redis 密码、备份和资源参数。
4. 执行 `bash deploy/preflight.sh`，任何失败都禁止继续。
5. 执行 `docker compose --env-file .env.production -f docker-compose.prod.yml build`。
6. 执行 `docker compose --env-file .env.production -f docker-compose.prod.yml up -d`。
7. 执行 `bash deploy/acceptance_check.sh`、读写压测、worker 回归、媒体鉴权流回归。
8. 配置宿主机 HTTPS Nginx，完成证书签发、跳转、HSTS 和访问日志检查。
9. 灰度开放，观察 24 小时；确认无 5xx、Redis 错误、SQLite 锁、worker dead-letter 堆积后再扩大范围。
