## 2. v3 三审新发 bug（必须在上云前修）

### ❌ B1 [P0 阻塞] 付费音视频播放 100% 失败：`<audio>/<video>` 不带 Authorization 头

**位置**：
- 后端路由：`backend/routers/media.py:193-217` `GET /api/media/{kind}/{slug}/stream`，只检查 `Authorization: Bearer <token>` header。
- 业务逻辑：`backend/services/media_service.py:336-342` `_public_asset_url` 把 `media_url` 重写为 `/api/media/{kind}/{slug}/stream`。
- 前端：`frontend/src/pages/MediaDetailPage.jsx:30-33` `buildPlayableUrl(item)` 返回上面那个 URL。
- 前端：`MediaDetailPage.jsx:266-280` `<audio src={playableUrl}>` 和 `:288-300` `<video src={playableUrl}>` 直接把 URL 塞进 `<audio>/<video>` 元素。

**bug 机理**：
1. 浏览器对 `<audio src>` / `<video src>` 的 GET 请求由**媒体元素自身发起**，不走 fetch，**不会自动携带 localStorage 里的 CAS token**。
2. 请求到达 `media_public_stream` → `get_authenticated_user(None)` → `user=None`
3. `_resolve_access` 对付费内容必然判定 False → **HTTPException 403**
4. 浏览器 `<audio>/<video>` 展示 "播放失败" 状态，付费会员用户看到的是错误

**复现步骤（在真实环境里）**：
- 登录一个付费会员账号
- 打开 `/audio/<某付费音频 slug>`
- 点击播放按钮
- **会看到 403 或空白播放器**

**修复方案（二选一）**：

方案 A（最小改动，推荐）——接受 query-param token：
```python
# backend/routers/media.py
@router.get("/{kind}/{slug}/stream")
def media_public_stream(
    kind: str,
    slug: str,
    token: str | None = Query(default=None),   # 新增
    authorization: str | None = Header(default=None),
    ...
):
    effective_auth = authorization or (f"Bearer {token}" if token else None)
    user = get_authenticated_user(effective_auth, ...)
    ...
```

前端 `buildPlayableUrl`:
```js
function buildPlayableUrl(item, token) {
  const url = item?.accessible ? item?.media_url : item?.preview_url || item?.media_url
  if (!url) return null
  const absolute = resolveApiUrl(url)
  if (token && url.startsWith('/api/media/') && url.endsWith('/stream')) {
    const separator = absolute.includes('?') ? '&' : '?'
    return `${absolute}${separator}token=${encodeURIComponent(token)}`
  }
  return absolute
}
```
**注意**：token 会出现在 Nginx access log，开关 log：`deploy/nginx/default.conf` 里给 stream 这条 location 加 `access_log off;` 或 `access_log ... combined_no_query;`，避免 token 泄漏。

方案 B（更彻底，工程量大）——改用 HLS 分片 + 每次签名。不建议本阶段做。

**阻塞级别**：P0。不修完就推云 = 付费会员投诉立即涌入。

### ⚠️ B2 [P1] Blue-Green 同时挂同一份 SQLite — 潜在 schema 竞态

**位置**：`deploy/blue_green_deploy.sh:7` `COMPOSE_SERVICES="redis backend-web frontend"`，启动候选 project 时挂载的 volume 是同一份 `./data`。
**机理**：
- candidate backend-web 启动时走 `_prepare_runtime` → `ensure_database_ready()` + `ensure_runtime_tables()`（执行 `CREATE TABLE IF NOT EXISTS ...`），同一时间老 backend-web 仍在写事务。
- SQLite WAL 下读写并发 OK，但**DDL（CREATE TABLE / CREATE INDEX）会对全库加 exclusive lock**。老 backend-web 的写请求如果撞上 DDL，会 busy_timeout 重试；如果老 backend-web 已经在一个 BEGIN IMMEDIATE 事务里，DDL 会阻塞到事务释放。
- 如果候选版本带了 schema 变更（新增字段、索引），**可能跨 project 产生锁争抢**，偶发 `database is locked`。

**修复**：
- 方案一（最稳）：在 `blue_green_deploy.sh` 里加一段，如果检测到本轮有 schema 变更（`backend/database.py` 或 `ensure_runtime_tables` 改动）就走"先停旧再起新"路径；否则走真·并行 blue-green。
- 方案二（简化）：文档明确写"涉及 DB schema 变更不要用 blue-green，用 `rollback_release.sh` 式的重启部署"。

**阻塞级别**：P1（不修在 schema 无变化的版本发布里不会出问题，但首次 DB 改动时容易翻车）。

### ⚠️ B3 [P1] `/metrics` 端点对公网暴露（仅 token 门）

**位置**：`backend/main.py:386-388` `@app.get("/metrics")` 注册在根路径。`deploy/nginx/default.conf` 的 `location /api/`（line 70）虽然限流，但 `/metrics`（无 `/api/` 前缀）被 `location /` 兜底捕获：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

所以 nginx 实际会把 `/metrics` 解释为前端 SPA 的 path，返回 `index.html`。**backend 的 `/metrics` 端点在 Nginx 外部不可达**——乍看很好，**但这意味着 `/metrics` 路由是死端**，Prometheus 抓不到。

查 `backend/main.py:391-393` 另有 `@app.get("/api/metrics")`，这条是可达的（匹配 `/api/` location）。所以实际能用，但路由冗余 + 没有 nginx 侧 IP 白名单。

**修复**：
1. 删掉 `@app.get("/metrics")` 路由；只保留 `/api/metrics`。
2. Nginx 加 `/api/metrics` 专用 location：
   ```nginx
   location = /api/metrics {
       allow 10.0.0.0/8;      # Prometheus 网段
       allow 127.0.0.1;
       deny all;
       proxy_pass http://fdsm_backend;
   }
   ```
即使 token 泄漏，外网也打不到 backend。

**阻塞级别**：P1（token 本身是常量时间比较，安全但面太宽；做好 IP 白名单才真正合规）。

## 3. 其他观察（不阻塞但值得注意）

### O1 `request_observability_middleware` 异常路径 response 未定义

`backend/main.py:220-253`：当下游 `call_next` 抛异常时，`response` 变量从未被赋值，进入 `finally` 块后第 238 行 `response.headers["X-Request-Id"]` 会触发 `NameError`，但第 237 行 `try/except Exception: pass` 吞掉了。

**后果**：异常请求的 `X-Request-Id` 响应头丢失；JSON log 仍能打出（不依赖 response 变量）；Sentry 自动上报不受影响。**不致命**，但调优：

```python
response = None
try:
    response = await call_next(request)
    status_code = int(response.status_code)
    return response
...
finally:
    ...
    if response is not None:
        try:
            response.headers["X-Request-Id"] = request_id
        except Exception:
            pass
```

### O2 startup sentinel 只在 container 生命周期内有效，`sync_local_audio_library` 可能不重新跑

`backend/main.py:83-95`：`sentinel_path.exists()` 早退会跳过 `sync_local_audio_library()`。sentinel 路径是 `/tmp/fdsmarticles-<hash>/startup.done`，**容器重启时 `/tmp` 被清空**，所以每次 container restart 都会重跑 → OK；但**多 gunicorn worker 同时启动时**，只有第一个拿到 fcntl 锁的 worker 真正执行；其余 worker 看到 sentinel 存在就跳过。这是预期行为（避免并发迁移），**正确**。

### O3 `request_observability_middleware` 里 `response.headers` 可能 break 流式响应

如果某个 API 返回 StreamingResponse，中间件修改响应头时机 OK（FastAPI middleware 是在头送出前）——没问题。

### O4 CSP 没加

nginx 加了 X-Frame-Options、X-Content-Type-Options、Referrer-Policy、Permissions-Policy、HSTS，**没加 Content-Security-Policy**。生产建议：

```nginx
add_header Content-Security-Policy "default-src 'self'; img-src 'self' data: blob:; media-src 'self' blob:; font-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'self'; connect-src 'self'; frame-ancestors 'self';" always;
```

Vite 默认构建会含 inline style（Tailwind），`'unsafe-inline'` 必须保留；若严格 CSP 就要改 script 加 nonce。**非阻塞**。

### O5 `X-Accel-Redirect` path 里的 `%` 编码

`_safe_accel_path` 返回 `quote(normalized_path, safe="/._-")`。如果文件名含中文或空格（`_sanitize_filename` 已经过滤非 word 字符，但中文保留——实际不包含中文文件名），Nginx alias 接到 URL encoded path 后会不会去 decode 查文件？**会**——Nginx alias 自动 URL-decode。所以 safe。

### O6 备份容器没 healthcheck

`docker-compose.prod.yml:256-288` backup 服务无 `healthcheck:`。如果 gpg/rclone 失败导致 `create_backup` 抛异常，`run_loop` 整个退出 → compose 会 `restart: unless-stopped`，但不会告警。建议加一个 heartbeat 文件式 healthcheck，同 housekeeping 容器。

### O7 Nginx `fdsmarticles-https.conf` 与 bootstrap 脚本逻辑冲突风险

`deploy/bootstrap_https.sh` 用 `sed` 替换 `knowledge.example.edu` 和端口。如果 `fdsmarticles-https.conf` 模板里出现 `server_name` 以外的 `knowledge.example.edu` 字样，会被全部替换——目前只有 2 处 `server_name` + 2 处证书路径，替换 OK。**但要求运维不要手工改模板 comment**。
