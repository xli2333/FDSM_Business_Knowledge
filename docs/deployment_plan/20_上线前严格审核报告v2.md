### P1-2 细化：Gunicorn 还差 3 件

```
gunicorn ... --workers ${GUNICORN_WORKERS:-3} --timeout ${GUNICORN_TIMEOUT:-180} \
    --graceful-timeout ${GUNICORN_GRACEFUL_TIMEOUT:-30} \
    --keep-alive ${GUNICORN_KEEP_ALIVE:-5} \
    --max-requests ${GUNICORN_MAX_REQUESTS:-1000} \
```

3. **`--access-logformat`**：默认 combined 格式会丢 `X-Forwarded-For`，`Host` 改用 `$forwarded_for` 格式：

### P1-10 细化：备份是假修复

`deploy/backup_sqlite.py` 整个 103 行文件没变过：
- 只做本机 `sqlite3.backup()` API → 落到 `/backups/*.db`
- 没有 `gzip`、没有 `gpg`、没有 `rclone`/`aws s3 cp`/`ossutil`。
- `.env.production.example:80` 的 `BACKUP_OFFSITE_TARGET` 和 `:81` 的 `BACKUP_ENCRYPTION_RECIPIENT` 在代码里**找不到引用**（用 `grep BACKUP_OFFSITE\|BACKUP_ENCRYPTION backend/ deploy/` 只在 preflight 和 env 示例里出现，业务逻辑零引用）。
- **结论**：等于 preflight 挡住了"用户没填这些变量"，但真填了也不会自动做异地+加密；这是"合规门面"不是"合规实现"。

**必须补**：在 `deploy/backup_sqlite.py` 里实现：
```python
import subprocess
def _encrypt_and_ship(local_path: Path) -> None:
    if os.getenv("BACKUP_ENCRYPTION_REQUIRED") == "1":
        recipient = os.environ["BACKUP_ENCRYPTION_RECIPIENT"]
        subprocess.check_call(["gpg", "--batch", "--yes", "--encrypt",
                               "--recipient", recipient, str(local_path)])
        local_path.unlink()
        local_path = local_path.with_suffix(local_path.suffix + ".gpg")
    if os.getenv("BACKUP_OFFSITE_REQUIRED") == "1":
        target = os.environ["BACKUP_OFFSITE_TARGET"]  # e.g. s3://fdsm-backup/prod/
        subprocess.check_call(["rclone", "copy", str(local_path), target])
```

## 4. 本轮新发问题（3 项）

### 🆕 N1 [P1] `cleanup_expired_sessions` 只在启动时跑一次，不是定时清理

**位置**：`backend/main.py:63-75`、`backend/services/cas_auth_service.py:281-296`

```python
with _exclusive_startup_lock(lock_path):
    if sentinel_path.exists():
        return
    ensure_database_ready()
    ensure_runtime_tables()
    try:
        from backend.services.cas_auth_service import cleanup_expired_sessions
        cleanup_expired_sessions()
    except Exception as exc:
        print(f"[startup] cas session cleanup skipped: {exc}", flush=True)
    sync_local_audio_library()
    sentinel_path.write_text("ok", encoding="utf-8")
```

**问题**：
- `sentinel_path.exists()` 早退使得**同一容器生命周期内只运行一次**；容器不重启就再也不清理。
- 商用场景下 backend-web 可能连续运行 30-60 天，`auth_sessions` 表会无界增长（每日登录产生的过期 session 从不删除）。
- 30 天 × 1000 DAU × 3 登录/天 = 9 万行积压后才触发容器重启手工清理。

**修复建议**：
把 `cleanup_expired_sessions` 接入 `backend-worker` 的 async_task_service，新增一个 cron 风格任务每小时触发；或在 `backup` 容器里并行跑一个 `python -m backend.scripts.housekeeping --every 3600`。

### 🆕 N2 [P1] 所有路由仍接受 `X-Debug-User-Id / X-Debug-User-Email` 头参数

**位置**：`backend/routers/media.py:57-58, 107-108, ...`（全部 router 同模式）

```python
x_debug_user_id: str | None = Header(default=None, alias="X-Debug-User-Id"),
x_debug_user_email: str | None = Header(default=None, alias="X-Debug-User-Email"),
```

**问题**：
- 这些头在 `IS_PRODUCTION=True` 时被 `backend/main.py:107-113` middleware 拦成 400 ✅
- 但**攻击面仍在**：middleware 如果因为配置错误/热重载失效被绕过，所有路由的 `x_debug_user_*` 会被传递到 `get_authenticated_user`。`supabase_auth_service._build_debug_user`（`:30-40`）在 `PREVIEW_AUTH_ENABLED=True` 时直接用 debug_user_id 构造假身份。
- `PREVIEW_AUTH_ENABLED = DEV_AUTH_ENABLED or ((not IS_PRODUCTION) and not SUPABASE_AUTH_ENABLED)`（`backend/config.py:129`）。理论上生产为 False，但**如果 `APP_ENV` 不是正好等于 `production` 字符串**（比如 `PRODUCTION`、`prod`、空值），middleware 不阻拦 + `IS_PRODUCTION=False` + Supabase 也没配 → PREVIEW_AUTH 打开 → debug 头可以伪造任意用户身份。

**纵深防御建议**：
1. 在生产部署里**彻底删掉** router 里所有 `x_debug_user_*` 参数；debug 注入只走 `dev_auth_service` 单独路径，生产编译产物直接不引入这些参数。
2. 或在每个 router 文件最顶处 `if IS_PRODUCTION: _DEBUG_AUTH_ALLOWED = False`，函数体内显式 assert。
3. `APP_ENV` 值做白名单校验（`backend/config.py` 启动时 `assert APP_ENV in {"production","staging","development","test"}`）。

### 🆕 N3 [P2] `cpus:`/`mem_limit:` 是 v2 compose "legacy" 字段，现代 compose 应该用 `deploy.resources`

**位置**：`docker-compose.prod.yml:60-61, 115-116, 144-145, 164-165, 189-190`

```yaml
cpus: ${BACKEND_WEB_CPUS:-2.0}
mem_limit: ${BACKEND_WEB_MEMORY:-2g}
```

**问题**：
- `cpus`/`mem_limit` 是 Compose v2 兼容老 v1 的字段（等价于 `docker run --cpus=... --memory=...`），在**非 swarm 模式**下可用，但不支持 `reservations`（最低保留）。
- 现代写法应该是 `deploy.resources.limits` + `deploy.resources.reservations`，同时 `docker compose` 和 `docker stack deploy` 通吃。
- 当前写法功能上没错，但少了 `reservations`——没有内存下限保证，在 OOM 选择杀进程时不占优。

**建议**：
```yaml
deploy:
  resources:
    limits: { cpus: '2.0', memory: 2G }
    reservations: { cpus: '0.5', memory: 512M }
```

## 7. 最终判定

**能否进入小流量灰度**：✅ **可以**。
