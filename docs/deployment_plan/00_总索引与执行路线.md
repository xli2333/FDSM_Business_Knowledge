# 00 · 部署总索引与执行路线

> **当前状态提示（2026-04-22）**：本文是早期 01-12 分阶段路线索引，保留用于理解改造脉络；当前代码已经完成后续 round175-round189 的落地。真正私有云部署请以 [16_私有云发布包_Nginx_HTTPS上线验收.md](./16_私有云发布包_Nginx_HTTPS上线验收.md)、[17_最终压测性能预算与交付清单.md](./17_最终压测性能预算与交付清单.md)、[18_部署文档对齐复查.md](./18_部署文档对齐复查.md)、[20_上线前严格审核整改记录.md](./20_上线前严格审核整改记录.md)、[21_上线前严格审核v2整改记录.md](./21_上线前严格审核v2整改记录.md)、[22_上线自动化与回滚.md](./22_上线自动化与回滚.md)、[23_依赖锁定与构建可复现.md](./23_依赖锁定与构建可复现.md)、[24_备份恢复演练.md](./24_备份恢复演练.md)、[25_前端Supabase依赖退场.md](./25_前端Supabase依赖退场.md)、[26_后端Supabase生产路径硬隔离.md](./26_后端Supabase生产路径硬隔离.md)、[27_三审上线前必修Bug整改.md](./27_三审上线前必修Bug整改.md) 以及 `docker-compose.prod.yml` 为准。本文中关于 `Dockerfile.frontend`、`docker-compose.override.yml`、容器内 HTTPS、`IMAGE_TAG` 镜像仓库发布、前端内联 Supabase 变量、生产默认 Supabase/dual 鉴权、根路径 `/metrics`、媒体播放只依赖 Authorization header 的内容属于历史方案，不作为当前执行命令。

> **本文档作用**：把整个"本地 → Docker → 云端 → 高并发"的迁移过程拆成 12 份独立可执行的原子级文档。每份文档顶部有前置依赖、预计耗时、完成标志。按顺序执行，中途可暂停。
> **最终目标**：本地 Docker 环境做到生产级完美，推到 Linux 私有云后只改鉴权配置或修小 bug 即可上线。
> **总预估工期**：熟手 3-4 天，边学边做 7-10 天。

## 执行原则

1. **必须按顺序**（01 → 02 → 03 → 04 → 05 → 06/07 并行 → 08 → 09 → 10 → 11 → 12）。02-05 之间任何一步错了都会让后面的改造基础不对。
2. **每份文档完成后 git commit 一次**。分支 `deployment/docker-prod`，commit message 格式：`deploy(step-XX): <做了什么>`。
3. **10 的 17 项不过不许上云**。这是铁律——本地不稳定推到云端只会更糟。
4. **遇到问题不要跳步**。每份文档末尾都有"常见错误"，先查；真不会就回退这个步骤的改动再重来，不要带着半截状态继续。
5. **鉴权/CAS 切换不在 12 份文档的主线里**。上线后再按 12 操作。
