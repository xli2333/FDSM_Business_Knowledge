# 23 依赖锁定与构建可复现

本文承接 round185，用于约束云端 Docker 重建时的 Python 依赖版本。

## 当前策略

- `requirements.txt`：只保留人工维护的直接依赖清单。
- `requirements.lock.txt`：生产 Docker 构建实际安装的完整锁文件。
- `Dockerfile.backend`：使用 `pip install -r requirements.lock.txt`，不再直接安装未 pin 的 `requirements.txt`。
- `deploy/check_requirements_lock.py`：校验 `requirements.txt` 中的直接依赖都存在于锁文件，并要求锁文件每行使用 `==` pin。
- 前端继续使用 `package-lock.json` + `npm ci`。

## 更新 Python 依赖的正确流程

1. 修改 `requirements.txt` 的直接依赖。
2. 在干净 Docker 构建环境中生成锁文件，不能用本机全局 Python 环境直接 `pip freeze`。
3. 重建后端镜像并确认 `pip check` 通过。
4. 用容器中的 `pip freeze` 更新 `requirements.lock.txt`。
5. 运行：

```bash
python deploy/check_requirements_lock.py
python -m compileall backend deploy
docker compose --env-file .env.docker build backend-web
docker compose --env-file .env.docker run --rm backend-web pip check
```

6. 再运行专项测试、Compose config 和发布包重建。

## CI 约束

`.github/workflows/cloud-release-qa.yml` 会使用 `requirements.lock.txt` 安装后端依赖，并运行锁文件校验。任何直接依赖缺失或锁文件未 pin 都会失败。

## preflight 约束

`deploy/preflight.sh` 会检查 `requirements.lock.txt` 是否存在。真实云端发布包缺少锁文件时，不能进入构建。
