## 逐项执行

### 项 1 · 健康检查

**命令**：
```powershell
curl http://localhost:8080/api/health -UseBasicParsing
```

**预期**：
```json
{
  "status": "ok",
  "service": "复旦管院商业智识库 API",
  "checks": {
    "database": {"ok": true, "article_count": 2100+},
    "faiss_index": {"ok": true, "path": "/data/faiss_index_business"},
    "uploads": {"ok": true}
  }
}
```

**如果失败**：
- `database.ok: false` → §02 WAL 改造有问题，或 DB 文件没挂进容器
- `faiss_index.ok: false` → volume 没挂对，进容器 `docker compose exec backend-web ls /data/`
- `uploads.ok: false` → 目录没建，`docker compose exec backend-web ls /data/uploads`

**通过标准**：HTTP 200 + 三个 checks 全 ok。

### 项 2 · 前端首页可打开

**命令**：
```powershell
Start-Process http://localhost:8080/
```

**预期**：浏览器打开，看到首页（文章列表、推荐等）。

**如果失败**：
- 页面白屏 → F12 看 Console，多半是 `VITE_SUPABASE_URL` 没传对（构建时要 `.env.production` 或 `.env.development` 里有）
- 502 错误 → backend-web 还没 healthy，等一会再试
- 样式错乱 → 清浏览器缓存

**通过标准**：首页内容可见、无 console 红色 error。

### 项 3 · 首页 API 返回数据

**命令**：
```powershell
curl "http://localhost:8080/api/home/feed?language=zh" -UseBasicParsing | ConvertFrom-Json | Select-Object -First 1
```

**预期**：返回一个包含文章数组的 JSON 对象。

**如果失败**：
- 404 → 反代路径错，检查 Nginx `location /api/` 配置
- 500 → 后端异常，`docker compose logs backend-web --tail=50`
- 空数组 → DB 里真的没文章，但 `/api/health` 里 article_count > 0 说明 DB 正常；多半是业务 SQL 条件问题

**通过标准**：返回 JSON，含若干篇文章。

### 项 5 · Dev 模式 debug 账号可登录

**前提**：当前 `.env.development`（或 `.env.production` 本地版）里 `APP_ENV=development` + `DEV_AUTH_ENABLED=1`。

**命令**：
```powershell
$resp = curl -UseBasicParsing http://localhost:8080/api/auth/status `
    -Headers @{"X-Debug-User-Id"="mock-admin"; "X-Debug-User-Email"="admin@example.com"} `
    | ConvertFrom-Json

Write-Host "authenticated = $($resp.authenticated)"
Write-Host "user.id = $($resp.user.id)"
Write-Host "tier = $($resp.membership.tier)"
```

**预期**：
```

user.id = mock-admin
tier = admin
```

**如果失败**：
- authenticated=False → §03 StripDebugHeadersMiddleware 误杀了（本地应该是 development 不剥离）
- 用户不存在 → seed user 没入库，跑 `docker compose exec backend-web python -c "from backend.services.user_profile_service import *; ..."` 看 SEED_USER_DEFINITIONS 有没有写进 DB

**通过标准**：debug 账号能认证成 admin。

### 项 13 · 资源占用合理

```powershell
docker stats --no-stream
```

**预期范围**（空闲状态）：

| 容器 | CPU | 内存 |
|---|---|---|
| fdsm-backend-web | < 5% | 400-700 MB |
| fdsm-backend-worker | < 3% | 200-400 MB |
| fdsm-frontend | < 1% | 10-20 MB |
| fdsm-redis | < 1% | 20-80 MB |

**总内存 < 1.5 GB**（空闲）；做过 FAISS 搜索后 backend-web 会涨到 900 MB-1.2 GB（FAISS 常驻）。

**如果 backend-web 内存 > 2 GB**：
- FAISS 多次加载（每 worker 一份 ≈ 4 × 200 MB = 800 MB，加上其他 = 1.2 GB）
- 不正常偏高 → 可能泄漏，看 `docker compose logs backend-web | Select-String memory`

**如果 CPU 一直 > 50%**：
- worker 可能在 busy loop（`pop_task` 没阻塞），检查 worker_loop.py

**通过标准**：内存 < 2GB，CPU 空闲 < 10%。

### 项 15 · AI 任务异步执行

```powershell
# 入队一个摘要任务
$body = @{article_id=1} | ConvertTo-Json
$resp = curl -UseBasicParsing -Method POST `
    -Uri "http://localhost:8080/api/editorial/ai/tasks/summarize" `
    -Headers @{"X-Debug-User-Id"="mock-admin"; "Content-Type"="application/json"} `
    -Body $body `
    | ConvertFrom-Json

$taskId = $resp.task_id
Write-Host "Task ID: $taskId, initial status: $($resp.status)"

# 轮询
1..30 | ForEach-Object {
    $task = curl -UseBasicParsing "http://localhost:8080/api/editorial/ai/tasks/$taskId" `
        -Headers @{"X-Debug-User-Id"="mock-admin"} | ConvertFrom-Json
    Write-Host "[$(Get-Date -Format HH:mm:ss)] status: $($task.status)"
    if ($task.status -in @("done","failed")) { break }
    Start-Sleep -Seconds 2
}
```

**预期**：
- 立即返回 `task_id` + `status: pending`（< 100 ms）
- 2-30 秒内看到 status 变化：`pending → running → done`
- worker 容器日志里有 `task xxx: done in X.Xs`

**如果 60 秒还没 done**：
- Gemini API 不通（国内本地跑会慢甚至失败）
- worker 日志有 exception：`docker compose logs backend-worker | Select-String -Pattern "task.*fail"`
- 网络差：改用更小的文章再试

**如果 status 一直 pending**：
- worker 没消费：`docker compose logs backend-worker --tail=50`
- Redis URL 错：backend-web 和 backend-worker 连的不是同一个 Redis

**通过标准**：task 成功进入 done 状态。

## 常见集成问题

### 问题 1：跨步骤互相干扰

**症状**：单独跑某项通过，跑完整 17 项时第 X 项挂了。

**原因**：某项留了副作用（比如测试数据没清）。

**修**：每项测试**前后都不应该改状态**。如果必须改，在测试末尾清理。

### 问题 2：多 worker 竞争锁

**症状**：AI 任务偶发 `database is locked`。

**原因**：worker_loop 里的 flush_view_counts 和 AI handler 同时写 SQLite。

**修**：worker loop 主循环里 AI 任务和 flush 本来就是串行的（一次只做一件事），除非你改了调度。回去看 §07 worker_loop 主循环逻辑。

### 问题 3：Nginx 加完限流 API 全 503

**症状**：限流 zone 配太紧，正常请求都被限。

**修**：调整 `rate=` 和 `burst=`。或者临时注释限流跑其他测试。

### 问题 4：backend-web healthcheck 一直失败

**症状**：容器 status 卡在 starting。

**逐项排查**：
1. `docker compose exec backend-web curl http://localhost:8000/api/health` → 看容器内能否访问
2. `docker compose logs backend-web --tail=100` → 看有没有 traceback
3. `docker compose exec backend-web ls /data/` → volume 挂对没
4. `docker compose exec backend-web python -c "from backend.database import get_connection; c=get_connection(); print(c.execute('SELECT 1').fetchone())"` → DB 能连吗
