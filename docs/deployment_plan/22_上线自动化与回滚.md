## 蓝绿发布约定

默认端口：

- blue：`${BLUE_PORT:-${APP_PORT:-8080}}`
- green：`${GREEN_PORT:-8081}`

默认活动颜色记录在 `/srv/fdsm/app/.active_color`。首次采用蓝绿脚本时，如果当前 Nginx 指向 `APP_PORT=8080`，可把 blue 视为当前线上版本：

```bash
cd /srv/fdsm/app
echo blue > .active_color
```

启动候选栈但不切流：

```bash
cd /srv/fdsm/app
bash deploy/blue_green_deploy.sh
```

脚本会启动候选颜色的 `redis/backend-web/frontend`，执行：

- `/healthz`
- `/api/health?details=1`
- `/api/home/feed?language=zh`
- 生产 debug header 400 探针

涉及 SQLite schema 变更时不要默认并行启动候选栈。脚本默认开启 `SCHEMA_CHANGE_GUARD=1`：

```bash
SCHEMA_CHANGE_REQUIRED=1 bash deploy/blue_green_deploy.sh
```

预期会拒绝发布，并提示改走维护窗口非并行部署。若使用 Git 发布并希望自动检测本轮 diff，可传入：

```bash
SCHEMA_CHANGE_BASE_REF=<上一版本tag或commit> bash deploy/blue_green_deploy.sh
```

只有人工确认本次 schema DDL 不会影响线上栈时，才允许：

```bash
ALLOW_SCHEMA_BLUE_GREEN=1 SCHEMA_CHANGE_REQUIRED=1 bash deploy/blue_green_deploy.sh
```

候选栈健康后切 Nginx：

```bash
cd /srv/fdsm/app
sudo SWITCH_NGINX=1 bash deploy/blue_green_deploy.sh
```

切流后如需停掉旧颜色：

```bash
sudo SWITCH_NGINX=1 STOP_OLD_AFTER_SWITCH=1 bash deploy/blue_green_deploy.sh
```

## 回滚

如果切流后出现异常，回滚到另一颜色：

```bash
cd /srv/fdsm/app
sudo bash deploy/rollback_release.sh
```

只启动回滚栈但不切 Nginx：

```bash
SWITCH_NGINX=0 bash deploy/rollback_release.sh
```

指定回滚颜色：

```bash
sudo ROLLBACK_COLOR=blue bash deploy/rollback_release.sh
```

回滚成功后，如需停掉故障颜色：

```bash
sudo STOP_FAILED_AFTER_ROLLBACK=1 bash deploy/rollback_release.sh
```

## 注意事项

- 候选栈会复用同一份 `data/`，只建议做短时间候选验收，不要长期并行承载写流量。
- 包含 `backend/database.py`、`ensure_runtime_tables`、DDL、schema/migration 路径变更的版本，应走维护窗口停旧起新，不走默认 blue-green 并行候选。
- 首次发布仍需先完成 `deploy/preflight.sh`、HTTPS bootstrap、真实备份恢复演练。
- 大流量或强写入场景下，蓝绿发布应升级到独立数据库/对象存储/外部 Redis 后再做长期双栈运行。
