## 背景与目标

### 当前问题

#### 问题 1：API base url 硬编码指向本地

```javascript
// frontend/src/api/index.js:4
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || 'http://127.0.0.1:8000/api'
```

生产时前端和后端**同域部署**（都走 Nginx），应该用 `/api` 的相对路径。环境变量漏设时默认值应该是 `/api` 而不是 `http://127.0.0.1:8000/api`。

#### 问题 2：debug header 在生产构建里裸奔

```javascript
// frontend/src/api/index.js:35
const debugAuthHeaders = authToken ? {} : getDebugAuthHeaders()
```

`getDebugAuthHeaders()` 从 localStorage 读 debug 账号并注入 `X-Debug-User-Id`。即使后端的 `StripDebugHeadersMiddleware` 生产环境会剥离，**前端产物里**还是有这段代码——攻击者分析 JS 就能发现这个机制。而且本地 dev 账号数据会混进用户的 localStorage，有数据泄漏风险。

#### 问题 3：Supabase token 注入机制不清晰

现在 `request(path, { authToken })` 要求调用方显式传 `authToken`。每个 `.jsx` 组件都要 `getSession()` 取 token 再传进来——繁琐且容易漏。需要一个"默认情况下自动从 Supabase 取最新 token"的机制。

### 改完之后

- 生产构建：API 走 `/api`（同域，不需要 CORS），Supabase token 自动注入，无任何 debug 代码痕迹
- 开发构建：API 走 `http://localhost:8000/api`（后端容器暴露端口），debug 账号可用
- 构建时通过 Vite `import.meta.env.PROD` 和 `.env.development` / `.env.production` 文件切换

### 步骤 1 · 改 `frontend/src/api/index.js` 默认 API_BASE_URL

**打开 `frontend/src/api/index.js`**。

**找到第 4 行**：
```javascript
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || 'http://127.0.0.1:8000/api'
const API_ORIGIN = API_BASE_URL.replace(/\/api\/?$/, '')
```

**替换为**：
```javascript
// 生产时走同域相对路径 /api（Nginx 反代）；
// 开发时如果没设 env 就指向本地后端容器的 localhost:8000
const API_BASE_URL = (
  import.meta.env.VITE_API_BASE_URL ||
  (import.meta.env.DEV ? 'http://127.0.0.1:8000/api' : '/api')
).replace(/\/$/, '')

// 用于拼接静态资源（/audio-files、/editorial-uploads 等）
// 生产同域时 API_ORIGIN 为空字符串，前端代码里 `${API_ORIGIN}/audio-files/xxx` 会自然变成 `/audio-files/xxx`
const API_ORIGIN = API_BASE_URL.startsWith('http')
  ? API_BASE_URL.replace(/\/api\/?$/, '')
  : ''
```

**为什么这样写**：
- `import.meta.env.DEV` 是 Vite 内置的 boolean，开发时 true、`npm run build` 出来的产物里 false
- 开发时默认 `http://127.0.0.1:8000/api` 方便本地直连后端容器
- 生产时默认 `/api`（同域走 Nginx）
- 允许 env 显式覆盖（staging 可能需要）

### 步骤 2 · 改 `frontend/src/api/index.js` 自动注入 Supabase token

**找到 `request` 函数**（第 31 行）：
```javascript
async function request(path, options = {}) {
  const isFormData = options.body instanceof FormData
  const visitorId = typeof window !== 'undefined' ? getOrCreateVisitorId() : null
  const authToken = options.authToken
  const debugAuthHeaders = authToken ? {} : getDebugAuthHeaders()
  const response = await fetch(`${API_BASE_URL}${path}`, {
    headers: {
      ...(isFormData ? {} : { 'Content-Type': 'application/json' }),
      ...(visitorId ? { 'X-Visitor-Id': visitorId } : {}),
      ...(authToken ? { Authorization: `Bearer ${authToken}` } : {}),
      ...debugAuthHeaders,
      ...(options.headers || {}),
    },
    ...options,
  })
  // ...
}
```

**整段替换为**（新增异步拿 token 的步骤）：
```javascript
// 从 Supabase 客户端拿最新 access_token（不阻塞，拿不到就不带）
async function getSupabaseAccessToken() {
  if (typeof window === 'undefined') return null
  try {
    // 延迟 import 避免启动时循环依赖
    const { supabase } = await import('../auth/supabaseClient.js')
    const { data } = await supabase.auth.getSession()
    return data?.session?.access_token ?? null
  } catch {
    return null
  }
}

async function request(path, options = {}) {
  const isFormData = options.body instanceof FormData
  const visitorId = typeof window !== 'undefined' ? getOrCreateVisitorId() : null

  // 三条取 token 的优先级：
  //   1. 调用方显式传 authToken（比如刚登录后，session 还没存到 supabase）
  //   2. 从 supabase 异步获取
  //   3. 开发模式下的 debug header（getDebugAuthHeaders 在生产构建里返回 {}）
  const explicitToken = options.authToken
  const supabaseToken = explicitToken ? null : await getSupabaseAccessToken()
  const token = explicitToken || supabaseToken
  const debugAuthHeaders = token ? {} : getDebugAuthHeaders()

  const response = await fetch(`${API_BASE_URL}${path}`, {
    headers: {
      ...(isFormData ? {} : { 'Content-Type': 'application/json' }),
      ...(visitorId ? { 'X-Visitor-Id': visitorId } : {}),
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      ...debugAuthHeaders,
      ...(options.headers || {}),
    },
    ...options,
  })

  // 其余 response 处理保持不变，接续原来的 if (!response.ok) ...
  // ...
}
```

**这段改动的核心**：
1. 新增 `getSupabaseAccessToken()` 异步取 token
2. `request` 函数变成完全异步（原本就是 async，没破坏性）
3. 优先级：`options.authToken` > Supabase session > debug（仅 dev）

if (!url || !key) {

### 步骤 3 · 改 `frontend/src/auth/debugAuth.js` 生产守卫

**打开 `frontend/src/auth/debugAuth.js`**。

**整个文件改为**：
```javascript
const DEBUG_AUTH_STORAGE_KEY = 'fdsm-debug-auth'

// 通过 Vite 的 import.meta.env.PROD 在构建时做编译优化
// 生产构建里所有 if (import.meta.env.PROD) 的分支会被 tree-shake 掉
const IS_PROD = typeof import.meta !== 'undefined' && import.meta.env && import.meta.env.PROD

export function loadDebugAuth() {
  if (IS_PROD) return null
  if (typeof window === 'undefined') return null
  try {
    const raw = window.localStorage.getItem(DEBUG_AUTH_STORAGE_KEY)
    if (!raw) return null
    const payload = JSON.parse(raw)
    if (!payload?.user_id) return null
    return {
      user_id: payload.user_id,
      email: payload.email || '',
      display_name: payload.display_name || '',
      tier: payload.tier || 'free_member',
    }
  } catch {
    return null
  }
}

export function saveDebugAuth(account) {
  if (IS_PROD) return
  if (typeof window === 'undefined') return
  window.localStorage.setItem(
    DEBUG_AUTH_STORAGE_KEY,
    JSON.stringify({
      user_id: account.user_id,
      email: account.email || '',
      display_name: account.display_name || '',
      tier: account.tier || 'free_member',
    }),
  )
}

export function clearDebugAuth() {
  if (typeof window === 'undefined') return
  window.localStorage.removeItem(DEBUG_AUTH_STORAGE_KEY)
}

export function getDebugAuthHeaders() {
  if (IS_PROD) return {}
  const payload = loadDebugAuth()
  if (!payload?.user_id) return {}
  return {
    'X-Debug-User-Id': payload.user_id,
    ...(payload.email ? { 'X-Debug-User-Email': payload.email } : {}),
  }
}
```

**关键**：
- 顶层 `const IS_PROD = ...` 在构建时求值为常量 `true`
- Vite 的 esbuild 会把 `if (true) return null; ...` 的死代码删掉
- 生产 bundle 里 `loadDebugAuth`、`saveDebugAuth`、`getDebugAuthHeaders` 函数体里除了 `return null` / `return {}` 什么都没有
- `clearDebugAuth` 保留功能（登出时清本地残留，哪怕是开发模式残留的 storage）

## 关于"动态 import supabaseClient" 的考虑

步骤 2 的代码里用了 `await import('../auth/supabaseClient.js')`。这样写的原因：

1. **避免启动时循环依赖**：`api/index.js` 被很多组件 import，如果直接 `import { supabase } from '../auth/...'`，而 `supabaseClient.js` 又需要环境变量就绪，容易出现 "supabase is undefined" 的启动错误
2. **轻微延迟**：动态 import 第一次会异步加载模块（~1ms）。后续缓存命中几乎无开销
3. **便于 mock**：单测里想替换 supabase 实现更简单

**如果项目整体用 Context 管理**（`AuthProvider` 传 supabase 实例给子组件），更好的做法是加一个 hook：
```javascript
// 某处：
export const useApi = () => {
  const { supabase } = useAuth()
  return useMemo(() => createApi(supabase), [supabase])
}
```

但这需要重构大量调用方。当前项目 API 都是通过顶层函数（`fetchHomeFeed` 等）调用的，动态 import 是最小侵入改法。
