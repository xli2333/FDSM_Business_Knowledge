# 六大板块栏目对接低风险筛选

> 结论边界：本轮只看“栏目如何承接六大板块”。音频、视频、媒体后台、播放器、播客详情页、视频详情页都不纳入改动范围。

## 1. 重新审阅后的总体判断

项目里已经有一套完整的栏目体系，可以承接“六大板块”：

- 后端栏目接口：`backend/routers/columns.py`
- 栏目查询与栏目文章列表：`backend/services/catalog_service.py`
- 栏目数据表：`columns`、`article_columns`
- 前端栏目页：`frontend/src/pages/ColumnPage.jsx`
- 首页栏目运营位：`home_content_slots` 里的 `column_navigation`
- 搜索页栏目筛选：`frontend/src/pages/SearchPage.jsx`
- 发布后台栏目选择：`frontend/src/pages/EditorialWorkbenchPage.jsx`
- 关注体系里的栏目关注：`backend/services/follow_service.py`

所以六大板块不需要新建六个大模块，也不需要改音视频系统。最稳妥的做法是：把“六大板块”定义成新版对外栏目，继续复用现有栏目页、栏目文章列表、栏目筛选、首页栏目导航和发布后台栏目归属。

当前数据库里已有 4 个栏目：

| 现有栏目 | slug | 当前定位 |
| --- | --- | --- |
| 深度洞察 | `insights` | 深度分析、专访、案例与长文 |
| 行业观察 | `industry` | 产业、公司、市场与商业模式 |
| 学术前沿 | `research` | 研究洞见、论文解读与管理新知 |
| 院长说 | `deans-view` | 院长、教授与核心师资观点 |

规格里的目标是 6 个板块：

| 目标板块 | 建议栏目定位 | 是否适合本轮低风险承接 |
| --- | --- | --- |
| 复旦观点 | 教授、院长、学者观点型栏目 | 适合 |
| 案例决策 | 商业案例、管理选择、决策复盘型栏目 | 适合 |
| 产业圆桌 | 产业议题、嘉宾观点、圆桌纪要型栏目 | 适合，但只做文章栏目，不动音频视频 |
| 热点拆解 | 热点事件、数据拆解、编辑部判断型栏目 | 适合 |
| 管理视野 | 管理思想、研究转译、杂志内容型栏目 | 适合，但不做播客 |
| 复理学堂 | 项目、课程、校友、学习路径型栏目 | 适合，但不做课程视频 |

## 2. 推荐的低风险方案

### 方案 A：六大板块作为新版栏目上线

这是最符合“栏目对接板块”的方案。

做法：

- 在 `columns` 表新增 6 个对外栏目。
- 不删除旧栏目，不改旧栏目 URL，避免破坏历史数据和外部链接。
- 首页 `column_navigation` 只配置这 6 个新版栏目。
- 栏目详情页继续使用现有 `/column/:slug`。
- 搜索页继续使用现有栏目筛选能力。
- 发布后台继续使用现有主栏目选择能力。

建议 slug：

| 板块 | 建议 slug | 说明 |
| --- | --- | --- |
| 复旦观点 | `fudan-viewpoints` | 对应教授观点、反共识判断、研究底座 |
| 案例决策 | `case-decisions` | 对应案例、选择题、决策复盘 |
| 产业圆桌 | `industry-roundtable` | 对应产业议题和圆桌纪要，不接音频视频 |
| 热点拆解 | `hot-briefing` | 对应热点、数据图表、编辑部判断 |
| 管理视野 | `management-review` | 对应管理思想、杂志文章、研究转译 |
| 复理学堂 | `fudan-classroom` | 对应课程、项目、校友、学习路径 |

风险判断：低。它主要是栏目数据和栏目展示层的调整，不需要改整体架构。

注意：如果新增 6 个栏目但保留旧 4 个栏目，搜索筛选和发布后台可能会同时看到 10 个栏目。若正式测试希望前台只出现六大板块，可以做一个很小的前端筛选：公共页面只展示六大板块栏目，旧栏目保留给历史 URL 和后台过渡使用。

### 方案 B：复用旧栏目做名称映射

这是代码和数据改动最少的方案，但产品表达不够准确。

可映射为：

| 目标板块 | 可临时映射到现有栏目 | 问题 |
| --- | --- | --- |
| 复旦观点 | 院长说 | 基本匹配，但“复旦观点”范围更大 |
| 热点拆解 | 行业观察 | 可以承接热点和产业事件 |
| 管理视野 | 学术前沿 | 能承接研究转译，但名字不匹配 |
| 案例决策 | 深度洞察 | 可承接案例，但缺少明确案例心智 |
| 产业圆桌 | 行业观察 | 会和热点拆解混在一起 |
| 复理学堂 | 学术前沿或深度洞察 | 不匹配，容易让用户找不到 |

风险判断：极低，但不建议作为正式测试版本，因为六大板块的产品结构会显得不清楚。

## 3. 可以优先做的简单模块内修改

以下都是在现有栏目体系内部完成，不涉及模块化大改。

### 3.1 新增六大板块栏目数据

可做内容：

- 新增 6 条 `columns` 数据。
- 设置名称、slug、简介、图标名、主题色、排序。
- 首页栏目运营位配置这 6 个栏目。

涉及范围：

- 数据库 `columns`
- 如需让新环境自动带出栏目，可加一个很小的 seed/upsert 逻辑
- 不需要新增页面，不需要新增接口

推荐等级：第一优先级。

### 3.2 栏目页增加六大板块专属文案

现有栏目页已经能展示栏目标题、简介、关注按钮和文章列表。可以只在栏目页头部补一层“板块定位”，例如：

- 复旦观点：教授判断、反共识观点、可迁移框架
- 案例决策：真实商业情境、关键选项、决策复盘
- 产业圆桌：产业议题、嘉宾观点、共识与分歧
- 热点拆解：数据事实、三层视角、编辑部判断
- 管理视野：管理思想、研究转译、实践启发
- 复理学堂：课程项目、学习路径、校友经验

涉及范围：

- `frontend/src/pages/ColumnPage.jsx`
- 可用静态配置，不需要新接口

推荐等级：第一优先级。

### 3.3 首页栏目导航改成六大板块入口

项目已有首页栏目运营位 `column_navigation`，最多支持 6 个实体，正好对应六大板块。

可做内容：

- 首页只露出六大板块入口。
- 每个入口链接到现有 `/column/:slug`。
- 不做新的首页模块，不改推荐流。

涉及范围：

- 优先通过现有运营位数据配置完成
- 如果需要默认配置，可调整 `backend/services/content_operations_service.py` 的默认栏目位

推荐等级：第一优先级。

### 3.4 搜索页栏目筛选改名为“板块”

搜索页已有栏目筛选。可以做很小的文案调整：

- “栏目”改成“板块”
- 六大板块优先显示
- 旧栏目如果保留，不在前台筛选里展示，或放到后台过渡

涉及范围：

- `frontend/src/pages/SearchPage.jsx`
- 不需要改搜索接口

推荐等级：第二优先级。

### 3.5 发布后台增加板块写作提示

发布后台已经有主栏目选择、AI 推荐栏目和手动确认栏目。可以在选择六大板块时显示轻量提示，不改发布流程。

示例：

| 选择板块 | 后台提示 |
| --- | --- |
| 复旦观点 | 需要突出教授观点、核心判断、可迁移框架 |
| 案例决策 | 建议包含情境、关键选择、复盘结论 |
| 产业圆桌 | 建议包含议题背景、不同观点、最终共识或分歧 |
| 热点拆解 | 建议包含数据事实、三层视角、明确判断 |
| 管理视野 | 建议突出管理启发和实践适用场景 |
| 复理学堂 | 建议突出课程、项目、学习路径或校友经验 |

涉及范围：

- `frontend/src/pages/EditorialWorkbenchPage.jsx`
- 不改审核流、发布流、AI 生成链路

推荐等级：第二优先级。

### 3.6 文章详情页展示“所属板块”

文章已经有栏目归属。可以把文章详情页中的栏目呈现调整成更明确的“所属板块”。

可做内容：

- 在文章标题区或元信息区展示所属板块。
- 点击板块进入 `/column/:slug`。
- 不改文章数据结构。

涉及范围：

- `frontend/src/pages/ArticlePage.jsx`
- 可能也包括 `frontend/src/components/ArticleCard.jsx`

推荐等级：第二优先级。

### 3.7 新稿件自动归入六大板块

当前后端有栏目归类规则，主要服务于现有 4 个栏目。可以小范围把规则改成六大板块版本。

可做内容：

- 复旦观点：教授、院长、学者、观点、访谈、研究团队等信号
- 案例决策：案例、复盘、决策、公司、战略、经营等信号
- 产业圆桌：产业、圆桌、论坛、嘉宾、对话、行业等信号
- 热点拆解：热点、政策、市场、数据、图表、事件等信号
- 管理视野：管理、组织、领导力、研究、论文、杂志等信号
- 复理学堂：课程、项目、课堂、校友、招生、学习等信号

涉及范围：

- `backend/services/clustering_rules.py`

推荐等级：第三优先级。

原因：它会影响新稿件的默认栏目归类。虽然改动不大，但需要用样本文章测一下，避免自动归类偏差。

## 4. 本轮不建议做的需求

这些需求要么涉及新模块，要么会牵动数据结构、互动系统或媒体系统，不适合当前“只做简单模块内修改”的边界。

| 需求 | 不建议原因 |
| --- | --- |
| 案例决策三选一互动 | 需要投票、状态、统计、重复提交控制，属于互动模块 |
| 投票分布和财务推演 | 需要新增业务数据结构和计算逻辑 |
| 教授点评结构化卡片 | 如果只是正文内容可以做；如果做独立点评模块则暂不建议 |
| 产业圆桌播客、音频、视频切片 | 用户已明确音频视频不改动 |
| 直播问答 | 需要直播、评论、审核或互动队列 |
| 管理视野 30 分钟播客 | 用户已明确音频视频不改动 |
| 复理学堂课程视频 | 用户已明确音频视频不改动 |
| 教授档案系统 | 需要人物实体、关系、文章关联，属于新模块 |
| 评论/讨论圈 | 需要评论、审核、通知、权限，属于互动模块 |
| 投稿系统 | 需要投稿状态、审核后台、权限流，属于新模块 |
| 全量重洗历史文章栏目 | 数据影响面大，正式测试前应单独规划和回滚方案 |

## 5. 推荐执行顺序

### 第一批：最稳，适合正式测试前做

1. 新增六大板块栏目数据。
2. 首页栏目导航只配置六大板块。
3. 栏目页增加六大板块定位文案。
4. 文章详情和文章卡片把栏目表达调整为“板块”。

这批改完后，用户已经能从首页、搜索、文章详情和栏目页看到清晰的六大板块结构。

### 第二批：仍然低风险，但需要更多检查

1. 搜索筛选把“栏目”改成“板块”，并优先显示六大板块。
2. 发布后台栏目选择增加六大板块写作提示。
3. 新稿件自动归类规则改成六大板块。

这批主要改善运营和编辑效率，不是前台结构成立的前置条件。

### 第三批：暂缓

1. 历史文章大规模迁移到六大板块。
2. 案例互动、投票、推演。
3. 教授档案、投稿、评论圈。
4. 所有音频、视频、媒体后台相关需求。

这些可以等六大板块栏目结构稳定后，再按独立功能拆项目。

## 6. 建议采用的最终口径

正式测试前建议采用：

> 六大板块先作为“栏目体系”上线，不新增大模块，不触碰音视频系统。前台展示、搜索筛选、发布后台和首页导航全部围绕六个栏目完成。旧栏目保留为历史兼容，不在主要入口强调。

这个方案对现有项目影响最小，也最容易在正式测试阶段回滚和调整。
