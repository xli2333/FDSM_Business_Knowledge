# 12 --- 数据管道

本章完整描述复旦商业知识库从原始网页到可检索、可翻译、可对话的全流程数据管道。整条管道分为四个阶段，每个阶段对应一个或多个脚本，前一阶段的输出即后一阶段的输入。

---

## 12.1 管道总览

```
Stage 1               Stage 2                Stage 3                Stage 4
网页爬虫               数据库构建              FAISS 索引构建          AI 批处理
(crawler)             (build_business_db)    (create_vector_db)     (article_ai_batch)

 微信公众号 ──┐                                                      ┌─ 摘要 (summary)
 复旦管院   ──┤        content.txt ─────► SQLite DB ─────► FAISS     ├─ 排版 (formatted_md)
 第三方源   ──┘        + 图片                articles      索引       └─ 翻译 (translation)
                       │                    tags                       │
                       │                    columns                    ▼
                       ▼                    topics              article_ai_outputs
               Fudan_Business_             featured_articles    article_translations
               Knowledge_Data/             chat_sessions
               YYYY-MM/日期_标题/
```

**执行顺序**

| 顺序 | 脚本 | 入口命令 | 说明 |
|------|------|----------|------|
| 1 | `business_knowledge_crawler.py` | `python business_knowledge_crawler.py` | 爬取网页，保存到本地文件系统 |
| 2 | `build_knowledge_base.py` | `python build_knowledge_base.py` | 读取本地文件，构建 SQLite 数据库 |

> 阶段 1 和阶段 4 均涉及外部网络请求（分别是网页爬取和 Gemini API 调用），运行时间较长。阶段 2 和阶段 3 纯本地计算，通常在数分钟内完成。

## 12.2 网页爬虫 --- `business_knowledge_crawler.py`

### 12.2.1 职责

从预定义的链接文件中逐条读取文章 URL，发送 HTTP 请求，解析 HTML 页面，提取正文文本和图片，保存到本地文件系统的标准目录结构中。

### 12.2.2 配置区域

| 配置项 | 值/说明 |
|--------|---------|
| `SAVE_ROOT` | 项目根目录下的源目录名（如 `锦缎`），按数据来源命名 |
| `LINKS_FILE` | 同级目录下的 `{源名}_links.txt`，每行一条链接记录 |
| `HEADERS` | 模拟 Chrome 121 的 User-Agent，避免被反爬策略拦截 |
| `MAX_WORKERS` | 并发线程数，默认 `5` |

### 12.2.3 链接文件格式

```
2025-12-19 | 文章标题文本 | https://mp.weixin.qq.com/s/xxxxxxxx
2025-12-18 | 另一篇标题   | https://www.fdsm.fudan.edu.cn/xxxxx
```

每行由 ` | ` 分隔为三段：日期、标题、URL。脚本会忽略空行和不符合三段格式的行。URL 必须以 `http` 开头。

### 12.2.4 输出目录结构

```
Fudan_Business_Knowledge_Data/
  2025-12/
    2025-12-19_文章标题最多50字/
      content.txt        <- 正文文本
      image_1.jpg        <- 页面内图片
      image_2.png
    2025-12-18_另一标题/
      content.txt
      image_1.jpg
  2025-11/
    ...
```

**目录层级规则：**
- 第一层：`YYYY-MM` 月份目录，取自日期字符串前 7 位
- 第二层：`YYYY-MM-DD_安全标题` 文章目录，标题经 `clean_filename()` 清洗后截断前 50 字符
- 文件：`content.txt` 为正文，`image_{序号}.{ext}` 为图片

**文件名清洗规则 (`clean_filename`)：**

```python
re.sub(r'[\\/*?:\"<>|\n\t]', "", text).strip()
```

移除所有 Windows 文件系统不支持的字符。

### 12.2.5 `content.txt` 文件格式

```
标题: 文章标题
日期: 2025-12-19
链接: https://mp.weixin.qq.com/s/xxxxxxxx
来源模式: wechat
----------------------------------------

正文内容...
```

文件头包含四个元数据字段（标题、日期、链接、来源模式），以 40 个短横线 `----------------------------------------` 分隔后接正文。正文由 `BeautifulSoup.get_text(separator="\n", strip=True)` 提取，保留自然换行。

### 12.2.6 页面解析策略

爬虫采用三级回退策略识别页面正文区域：

| 优先级 | 识别方式 | 来源模式 (`source_mode`) | 说明 |
|--------|----------|--------------------------|------|
| 1 | `div#js_content` 或 `div.rich_media_content` | `wechat` | 微信公众号标准正文容器 |
| 2 | `div.detail-con` | `school_native` | 复旦管院官网正文容器 |
| 3 | 全局搜索 `mmbiz.qpic.cn` 图片 | `wechat_fallback_global` | 纯图片文章的兜底策略 |

**详细流程：**

1. 首先尝试查找 `div#js_content` 或 `div.rich_media_content`（微信公众号正文区域）
2. 如果未找到，尝试查找 `div.detail-con`（学校官网正文区域）
3. 如果仍未找到，进入全局回退模式：
   - 搜索页面中所有包含 `mmbiz.qpic.cn` 域名的图片（同时检查 `src` 和 `data-src` 属性）
   - 如果找到图片，保存图片并写入 `content.txt`（标记为 "Fallback Global" 模式）
   - 如果连图片都没有，打印警告并跳过该页面

**微信图片处理注意事项：**
- 微信图片的真实 URL 存储在 `data-src` 属性而非 `src`（微信模式下优先取 `data-src`）
- 图片后缀通过 URL 参数判断：`wx_fmt=png` -> `.png`，`wx_fmt=gif` -> `.gif`，`wx_fmt=jpeg` 或 `wx_fmt=jpg` -> `.jpg`
- 协议相对路径（`//example.com/img.png`）自动补全 `https:` 前缀
- 非微信模式下优先取 `src` 属性

### 12.2.7 断点续爬

```python
if os.path.exists(content_file):
    safe_print(f"   [跳过] (已存在) {safe_title}...")
    return
```

如果目标目录的 `content.txt` 已存在，直接跳过。这意味着：
- 脚本可以安全地重复执行
- 增量爬取只需在链接文件中追加新行
- 需要重爬时，手动删除对应目录即可

### 12.2.8 并发控制与线程安全

```python
MAX_WORKERS = 5

with ThreadPoolExecutor(max_workers=MAX_WORKERS) as executor:
    futures = [executor.submit(process_item, line) for line in lines]
    for future in futures:
        try: future.result()
        except: pass
```

使用 `concurrent.futures.ThreadPoolExecutor`，默认 5 个并发线程。所有 `print` 输出通过 `threading.Lock` (`print_lock`) 保护的 `safe_print()` 函数进行，防止终端输出交错。

每个线程独立处理一行链接记录，异常被静默捕获（`except: pass`），不会导致整个批次中断。

### 12.2.9 错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 链接格式不正确（少于 3 段） | 直接返回，跳过该行 |
| URL 不以 `http` 开头 | 直接返回，跳过该行 |
| HTTP 请求失败 | 捕获异常，打印 `[错误]` 日志 |
| 图片下载失败 | 捕获异常，打印 `Warning` 日志，不影响文本保存 |
| 无法识别页面结构 | 打印 `[警告]`，跳过该页面 |

## 12.3 数据库构建 --- `build_business_db.py`

### 12.3.1 职责

遍历 `Fudan_Business_Knowledge_Data/` 下所有 `content.txt` 文件，解析元数据与正文，进行标签分配、栏目分配、特征计算，最终将全部数据写入 SQLite 数据库。

入口脚本 `build_knowledge_base.py` 是一个薄包装层，直接调用 `backend.scripts.build_business_db.main()`。

### 12.3.2 完整处理流程

### 12.3.3 数据模型 --- `RawDocument`

`RawDocument` 是内部数据传输对象（dataclass），包含一篇文章的所有计算结果：

| 字段 | 类型 | 说明 |
|------|------|------|
| `article_id` | `int` | 自增主键，按文件遍历顺序分配 |
| `doc_id` | `str` | 基于 `relative_path` 的 SHA1 哈希前 20 位，跨重建稳定 |
| `slug` | `str` | URL 友好标识，格式 `{日期}-{标题}`，最长 96 字符 |
| `relative_path` | `str` | 相对于 `BUSINESS_DATA_DIR` 的路径 |
| `title` | `str` | 文章标题，优先取 content.txt 头部，其次取 AI 输出，最后取目录名 |
| `publish_date` | `str` | 发布日期 `YYYY-MM-DD`，缺失时默认 `2000-01-01` |
| `link` | `str` | 原始来源 URL |
| `source_mode` | `str` | 来源模式 (`wechat` / `school_native` / `business`) |
| `content` | `str` | 正文文本 |
| `excerpt` | `str` | 正文前 180 字符摘要，由 `excerpt_from_content()` 生成 |
| `word_count` | `int` | 去空格后的字符数 |
| `article_type` | `str` | 文章类型（专访/对谈/评论观点等） |
| `main_topic` | `str \| None` | 主话题（来自 AI 分析） |
| `series_or_column` | `str \| None` | 所属系列/专栏（来自 AI 分析） |
| `primary_org_name` | `str \| None` | 主要关联机构 |
| `cover_image_path` | `str \| None` | 封面图片相对路径（取目录内按字母排序首个图片） |
| `tag_names` | `list[tuple[str, str, float]]` | 标签列表 `(名称, 分类, 置信度)` |
| `column_slugs` | `list[str]` | 分配到的栏目 slug 列表 |
| `people_text` | `str` | 人物名称，`\|` 分隔，最多 4 人 |
| `org_text` | `str` | 机构名称，`\|` 分隔，最多 4 个 |
| `tag_text` | `str` | 标签名称，`\|` 分隔 |
| `search_text` | `str` | 全文搜索索引文本（拼接标题+主话题+标签+人物+机构+摘要） |
| `view_count` | `int` | 初始浏览量（算法生成） |
| `is_featured` | `int` | 是否精选（0/1） |

### 12.3.4 元数据解析

**`split_header_and_body(text)`**

按照 `----------------------------------------`（40 个短横线）分隔符将 `content.txt` 分为头部和正文。如果文件中没有分隔符，则取前 6 行作为头部，其余为正文。

**`parse_header(header_lines)`**

从头部行中提取四个关键字段：

| 前缀 | 字段名 |
|------|--------|
| `标题:` | `title` |
| `日期:` | `publish_date` |
| `链接:` | `link` |
| `来源模式:` | `source_mode` |

**`slugify(value)`**

将标题转换为 URL 友好的 slug：
1. 转小写
2. 替换 `/`、`|`、`丨`、`·`、空格为 `-`
3. 用正则移除非 `\w`、非中文、非 `-` 的字符
4. 合并连续 `-`，去掉首尾 `-`
5. 如果结果为空，使用 SHA1 哈希前 12 位

### 12.3.5 AI 预分析结果合并

`load_ai_output()` 从 `GEMINI_AGGREGATE_PATH` 加载 JSON 文件，按 `relative_path` 建立索引。每篇文章的字段优先级为：

| 字段 | 优先级：content.txt 头部 > AI 输出 > 默认值 |
|------|----------------------------------------------|
| `title` | 头部 `标题:` > `ai_row.title` > 目录名 |
| `publish_date` | 头部 `日期:` > `ai_row.publish_date` > `"2000-01-01"` |
| `link` | 头部 `链接:` > `ai_row.source_url` > `""` |
| `source_mode` | 头部 `来源模式:` > `ai_row.source_mode` > `"business"` |

AI 预分析提供以下独有增强字段（`model_output` 子对象）：

| AI 输出字段 | 用途 |
|-------------|------|
| `article_type` | 文章类型分类 |
| `main_topic` | 主话题 |
| `series_or_column` | 所属系列 |
| `topic_keywords` | 主题关键词列表（最多取 5 个） |
| `primary_org_name` | 主要关联机构 |
| `people` | 文中提及人物列表（含 `person_name`、`org_name`、`fdsm_relation`） |

### 12.3.6 文章类型推断

当 AI 预分析结果的 `article_type` 不可用时，`infer_article_type()` 通过标题和正文前 200 字的关键词规则推断：

| 检测条件 | 推断类型 |
|----------|----------|
| `专访` 出现在标题或正文前200字 | 专访 |
| `对谈` 出现在标题 | 对谈 |
| `访谈` 出现在标题 | 访谈 |
| `案例` 出现在标题 | 案例报道 |
| `研究` 或 `论文` 出现在标题 | 研究/论文解读 |
| `预告` 或 `报名` 出现在标题 | 活动/预告 |
| `书` + (`阅读` / `书评` / `翻书`) 出现在标题 | 书摘/书评 |
| 以上均不匹配 | 评论/观点 |

### 12.3.7 标签分配

标签通过 `taxonomy_service.build_tag_entries()` 生成，每个标签包含三元组 `(name, category, confidence)`。标签分为五个类别：

| 分类 (`category`) | 颜色 | 说明 | 示例 |
|--------------------|------|------|------|
| `industry` | `#ea6b00` (橙色) | 行业标签 | 金融科技、新能源 |
| `topic` | `#0d0783` (深蓝) | 主题标签 | AI/人工智能、ESG/可持续 |
| `type` | `#4f46e5` (靛蓝) | 类型标签 | 专访、研究 |
| `entity` | `#64748b` (灰色) | 实体标签 | 机构名称、人物名称 |
| `series` | `#7c3aed` (紫色) | 系列标签 | 翻书党 |

**关键词筛选规则（全局阈值过滤）：**

| 规则 | 条件 |
|------|------|
| 关键词准入 | 出现次数 >= 6 次 |
| 系列准入 | 出现次数 >= 8 次 |
| 关键词长度上限 | 18 字符 |
| 系列名称长度上限 | 20 字符 |
| 停用词过滤 | `KEYWORD_STOPWORDS` 集合 |
| 噪声系列过滤 | `SERIES_NOISE` 集合 |

这种全局阈值设计确保只有在语料库中具有统计显著性的标签才会被保留，避免长尾噪声标签污染标签云。

### 12.3.8 栏目分配

`clustering_rules.derive_column_slugs()` 根据以下特征将文章分配到一个或多个栏目：

| 输入特征 | 说明 |
|----------|------|
| `word_count` | 字数 |
| `article_type` | 文章类型 |
| `series_or_column` | 系列/专栏名称 |
| `tag_entries` | 标签列表 |
| `fdsm_hits` | 复旦管院关联关键词命中情况 |

**复旦管院关联关键词 (`FDSM_RELATION_KEYWORDS`)：**

`教师`、`教授`、`院长`、`教职`、`老师`、`学术院长`、`特聘教授`

`normalize_fdsm_relation()` 检查 AI 输出中的 `fdsm_relation` 字段是否包含上述关键词，有则返回匹配的关键词，供栏目分配使用。

栏目定义来自 `backend.config.COLUMN_DEFINITIONS`，每个栏目包含以下字段：`name`、`slug`、`description`、`icon`、`sort_order`、`accent_color`。

### 12.3.9 浏览量计算

`stable_view_count()` 基于文档特征生成一个**确定性**的初始浏览量，不依赖真实用户行为：

```python
age_days = max((date.today() - date.fromisoformat(publish_date)).days, 1)
recency_factor = max(0.2, 3.2 - age_days / 1200)   # 时效因子：越新越高
length_factor = min(2.4, word_count / 1800)           # 长度因子：越长越高，上限 2.4
noise = int(sha1(doc_id[:6]).hexdigest(), 16) % 1800  # 确定性随机噪声 0-1799
view_count = int(520 + recency_factor * 950 + length_factor * 420 + hot_tag_count * 180 + noise)
```

| 因子 | 范围 | 说明 |
|------|------|------|
| 基础值 | 520 | 所有文章的最低浏览量基线 |
| 时效因子 | 0.2 -- 3.2 | 发布日距今越近，值越大 |
| 长度因子 | 0 -- 2.4 | 字数越多，值越大 |
| 热门标签计数 | 0 -- N | `topic` 和 `industry` 类标签的数量 |
| 确定性噪声 | 0 -- 1799 | 由 `doc_id` 的 SHA1 决定，保证跨重建稳定 |

该算法保证同一篇文章在多次数据库重建中获得相同的浏览量值（由 `doc_id` 决定的确定性计算）。

### 12.3.10 精选文章

`feature_score()` 综合以下因素为每篇文章评分：

| 因子 | 权重/加分 |
|------|-----------|
| `view_count / 600` | 基础分 |
| `word_count / 2000` (上限 3.5) | 长文加分 |
| 文章类型为专访/访谈/对谈/案例报道 | +1.2 |
| 包含 "AI/人工智能" 标签 | +0.8 |
| 包含 "ESG/可持续" 标签 | +0.6 |
| 系列属于 `SERIES_NOISE` | -1.4 |

排名前 24 的文章标记为 `is_featured = 1`，并写入 `featured_articles` 表：

| 排名 | `position` 值 |
|------|---------------|
| 第 1 名 | `hero` |
| 第 2--7 名 | `editor-1` 至 `editor-6` |

### 12.3.11 数据库表结构

## 12.4 FAISS 索引构建 --- `create_vector_db_faiss.py`

### 12.4.1 职责

从 SQLite 数据库读取文章数据，进行文本分块和向量化，构建 FAISS 向量索引，用于语义搜索和 RAG 对话。

### 12.4.2 配置

| 配置项 | 值 |
|--------|-----|
| `SQLITE_DB_PATH` | `项目根/fudan_knowledge_base.db` |
| `FAISS_DB_DIR` | `项目根/faiss_index_business/` |
| `BATCH_SIZE` | `50`（每批嵌入的文档块数） |
| 嵌入模型 | `GEMINI_EMBEDDING_MODEL`（来自 `backend.config`） |
| 距离策略 | `DistanceStrategy.COSINE`（余弦相似度） |

### 12.4.3 数据加载 --- `load_documents()`

从 `articles` 表按 `id ASC` 排序读取全部记录，提取以下字段：

```sql
SELECT id, title, publish_date, link, content, article_type, main_topic, tag_text
FROM articles
ORDER BY id ASC
```

将每篇文章格式化为结构化文本：

```
Title: 文章标题
Date: 2025-12-19
Type: 专访
Main Topic: 人工智能
Tags: AI/人工智能 | 金融科技

正文内容全文...
```

每条记录包装为 LangChain `Document` 对象，`metadata` 包含：

| 元数据字段 | 来源 | 说明 |
|------------|------|------|
| `article_id` | `articles.id` | 用于检索结果回溯 |
| `title` | `articles.title` | 展示用 |
| `publish_date` | `articles.publish_date` | 展示用 |
| `link` | `articles.link` | 原始来源链接 |
| `source` | 固定值 `"business"` | 来源标识 |

### 12.4.4 文本分块 --- `split_documents()`

使用 LangChain 的 `RecursiveCharacterTextSplitter`：

| 参数 | 值 | 说明 |
|------|----|------|
| `chunk_size` | `800` | 每个文本块最大 800 字符 |
| `chunk_overlap` | `120` | 相邻块重叠 120 字符，保证上下文连续性 |
| `length_function` | `len` | 按 Python 字符数计算长度（中文字符等价于1） |

分块后每个 chunk 继承原文档的 `metadata`，便于检索结果回溯到原始文章。`RecursiveCharacterTextSplitter` 会优先在段落、句子、单词边界处分割，尽量保持语义完整性。

### 12.4.5 向量索引构建 --- `create_index()`

```python
embeddings = GoogleGenerativeAIEmbeddings(
    model=GEMINI_EMBEDDING_MODEL,
    google_api_key=google_api_key,
    task_type="retrieval_document",
)
```

**API Key 解析优先级：**

1. `GOOGLE_API_KEY` 环境变量（优先）
2. `GEMINI_API_KEYS` 环境变量（逗号分隔列表，取第一个）
3. 以上均无则抛出 `ValueError`

**构建过程：**

1. 初始化 `GoogleGenerativeAIEmbeddings`，`task_type="retrieval_document"` 表示为文档索引生成嵌入向量
2. 按 `BATCH_SIZE=50` 分批处理全部文本块
3. 第一批使用 `FAISS.from_documents()` 创建向量存储，指定 `COSINE` 距离策略
4. 后续批次使用 `vectorstore.add_documents()` 追加到已有索引
5. 每批最多重试 3 次（覆盖 API 限流等临时错误），重试间隔 2 秒
6. 第 3 次重试仍失败则抛出异常
7. 全部处理完成后调用 `vectorstore.save_local(FAISS_DB_DIR)` 持久化

使用 `tqdm` 进度条展示构建进度，单位为 `batch`。

### 12.4.6 输出文件

```
faiss_index_business/
  index.faiss          <- FAISS 二进制索引文件（向量数据）
  index.pkl            <- Python pickle 文件（Document 对象和元数据序列化）
```

### 12.4.7 主入口

```python
if __name__ == "__main__":
    create_index(split_documents(load_documents()))
```

三个函数管道式调用：`load_documents()` -> `split_documents()` -> `create_index()`。

## 12.5 AI 批处理 --- `article_ai_batch.py`

### 12.5.1 职责

对数据库中的所有文章进行三类 AI 处理：**中文摘要生成**、**Markdown 排版格式化**、**英文全文翻译**。使用多工作进程并行调用 Gemini API，支持断点续跑和状态追踪。

### 12.5.2 目录结构

```
Fudan_Business_Knowledge_Data/article_ai_batch/
  manifest.json           <- 待处理文章清单（从数据库生成）
  run_meta.json           <- 运行元信息（开始时间、配置等）
  logs/                   <- 工作进程日志目录
  state/                  <- 工作进程状态文件目录
    orchestrator.json     <- 编排器状态
    worker-0.json         <- 各分片工作进程状态
    worker-1.json
    ...
```

### 12.5.3 命令行接口

脚本通过 `argparse` 提供四个子命令：

#### `manifest` --- 构建/重建文章清单

```bash
python -m backend.scripts.article_ai_batch manifest [--limit N] [--rebuild]
```

| 参数 | 说明 |
|------|------|
| `--limit N` | 限制清单中的文章数量（0 = 全部） |
| `--rebuild` | 强制重建清单，忽略已有 `manifest.json` |

从 SQLite 数据库查询文章列表，生成 `manifest.json`。

#### `run-worker` --- 运行单个工作分片

```bash
python -m backend.scripts.article_ai_batch run-worker \
  --shard-index 0 --total-shards 10 [--limit N] [--rebuild-manifest] [--worker-name NAME]
```

| 参数 | 说明 |
|------|------|
| `--shard-index` | 当前分片索引（必需） |
| `--total-shards` | 总分片数（必需） |
| `--limit N` | 处理文章数量上限 |
| `--rebuild-manifest` | 先重建清单再处理 |
| `--worker-name` | 自定义工作进程名 |

#### `run-all` --- 运行完整多工作进程批处理

```bash
python -m backend.scripts.article_ai_batch run-all \
  [--workers 20] [--limit N] [--poll-seconds 15] [--rebuild-manifest]
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--workers` | `20` | 并发工作进程数 |
| `--poll-seconds` | `15` | 编排器轮询子进程状态的间隔秒数 |
| `--limit` | `0`（全部） | 处理文章数量上限 |
| `--rebuild-manifest` | `false` | 先重建清单 |

编排器通过 `subprocess` 启动多个子进程，每个子进程运行 `run-worker`，分片处理清单中的文章。

#### `status` --- 查看批处理状态

```bash
python -m backend.scripts.article_ai_batch status
```

读取 `state/` 目录下的状态文件，汇总展示整体进度和各工作进程状态。

### 12.5.4 模型配置

| 用途 | 模型变量 | 默认值 |
|------|----------|--------|
| 摘要生成 | `FLASH_MODEL` | `gemini-2.5-flash` |
| 排版格式化 | `FLASH_MODEL` | `gemini-2.5-flash` |
| 英文翻译 | `TRANSLATION_MODEL_CHAIN` | `[FLASH_MODEL, CHAT_MODEL]` 依次尝试 |

翻译模型链：先使用 `GEMINI_FLASH_MODEL`，如果失败再回退到 `GEMINI_CHAT_MODEL`。如果两者相同则链中只有一个模型。

### 12.5.5 Prompt 模板

#### 摘要 Prompt (`SUMMARY_PROMPT`)

角色定位为"复旦商业知识编辑部的摘要助手"，要求：
1. 保留原文逻辑，不编造事实
2. 优先用 5--8 个短标题或要点组织内容
3. 语言凝练、专业，避免空话
4. 仅输出 Markdown

输入变量：`{title}`、`{content}`

#### 排版 Prompt (`FORMAT_PROMPT`)

角色定位为"复旦商业知识编辑部的排版助手"，要求：
1. 保留全部关键事实、论点、案例与引语
2. 补出标题层级、导语、小标题、列表与引用
3. 一级标题必须是文章标题
4. 采访/对话/演讲/案例文章保留原有说话人和结构
5. 不输出 HTML，只输出 Markdown
6. 参考"复旦商业知识 / AI_writer"公众号长文排版风格

输入变量：`{title}`、`{excerpt}`、`{main_topic}`、`{article_type}`、`{organization}`、`{tags}`、`{content}`

#### 翻译 Prompt (`TRANSLATION_PROMPT`)

英文翻译，要求：
1. 独立于摘要和排版，直接从中文原文翻译
2. 翻译全文，保留所有有意义的内容
3. 保留段落、列表、引用结构
4. 返回严格 JSON 格式

输出 JSON 结构：

```json
{
  "title": "English title",
  "excerpt": "English deck or short intro",
  "summary": "English Markdown summary",
  "content": "Full translated Markdown body"
}
```

### 12.5.6 翻译字段别名映射

`TRANSLATION_FIELD_ALIASES` 定义了翻译 JSON 输出中各字段的可接受别名，增强对 AI 输出格式变化的容错性：

| 标准字段 | 接受的别名 |
|----------|-----------|
| `title` | `title`, `english_title`, `translated_title` |
| `excerpt` | `excerpt`, `deck`, `intro`, `lead`, `subtitle` |
| `summary` | `summary`, `abstract`, `overview`, `brief` |
| `content` | `content`, `body`, `translation`, `translated_content`, `article` |

### 12.5.7 错误处理与重试

| 参数 | 值 | 说明 |
|------|----|------|
| `MAX_API_ATTEMPTS` | `8` | 单个文章单个任务的最大重试次数 |
| `CONNECT_TIMEOUT_SECONDS` | `30` | HTTP 连接超时 |
| `READ_TIMEOUT_SECONDS` | `300` | HTTP 读取超时（5 分钟，适应长文翻译） |

**重试策略：**
- 错误消息以 `NONRETRYABLE:` 前缀开头的为不可重试错误（如输入格式问题），直接跳过
- `is_non_retryable_batch_error()` 检测该前缀
- 可重试错误（API 限流、网络超时等）在最多 8 次尝试内重试

**状态持久化：**
- 每个工作进程的状态通过 JSON 文件持久化到 `state/` 目录
- `write_json_atomic()` 使用临时文件（`.tmp` 后缀）+ `Path.replace()` 保证写入原子性
- 支持断点续跑：重启后读取状态文件，跳过已完成的文章

### 12.5.8 API Key 管理

```python
def load_api_keys() -> list[str]:
    keys = [item.strip() for item in GEMINI_API_KEYS if item.strip()]
    if PRIMARY_GEMINI_KEY and PRIMARY_GEMINI_KEY not in keys:
        keys.insert(0, PRIMARY_GEMINI_KEY)
    if not keys:
        raise BatchError("No Gemini API keys are configured.")
    return keys
```

- 从 `GEMINI_API_KEYS`（列表）加载所有可用 Key
- `PRIMARY_GEMINI_KEY` 如果存在且不在列表中，插入队首优先使用
- 多个 Key 支持轮换，分散 API 配额压力
- 无可用 Key 时抛出 `BatchError`

### 12.5.9 输出存储

处理结果写入两个数据库表：

| 存储位置 | 键 | 内容 |
|----------|-----|------|
| `article_ai_outputs` | `article_id` | 中文摘要、排版后 Markdown、英文翻译全文、source_hash、模型名 |
| `article_translations` | `(article_id, target_lang, source_hash)` | 翻译缓存（title/excerpt/summary/content/model） |

`source_hash` 用于检测源文章是否发生变化，确保翻译结果与源文章一致。由 `build_article_source_hash()` 计算。

翻译结果同时通过 `upsert_article_translation()` 写入 `article_translations` 表，作为翻译服务的缓存层。

## 12.6 其他批处理脚本概述

### 12.6.1 `backend/services/taxonomy_service.py`

标签分类服务，提供 `build_tag_entries()` 函数。核心逻辑：
- 从标题、正文、AI 分析结果中提取关键词
- 通过 `normalize_keyword()` 标准化关键词（合并同义词、统一格式）
- 过滤停用词 (`KEYWORD_STOPWORDS`) 和噪声系列 (`SERIES_NOISE`)
- 根据关键词的全局频次判断是否纳入
- 输出 `(name, category, confidence)` 三元组列表

### 12.6.2 `backend/services/clustering_rules.py`

栏目分配规则服务，提供 `derive_column_slugs()` 函数。根据文章特征（字数、类型、系列、标签、复旦关联）将文章映射到一个或多个栏目 slug。

### 12.6.3 `backend/services/topic_engine.py`

专题引擎，提供两个核心函数：
- `auto_generate_topics(limit)` --- 通过 API 端点触发，自动生成专题
- `rebuild_topics(connection, limit_auto)` --- 在数据库构建过程中调用，基于 `TOPIC_AUTO_CLUSTERS` 配置自动聚合文章生成专题页面

### 12.6.4 `backend/services/tag_engine.py`

标签引擎，提供 `generate_tags_for_articles(limit, regenerate)` 函数。支持批量为文章生成标签，可通过 API 端点 `POST /api/tags/batch-generate` 触发。

## 12.7 管道维护指南

### 增量更新场景

| 场景 | 操作步骤 |
|------|----------|
| 新增文章来源 | 在链接文件中追加新行 -> 运行爬虫 -> 重建数据库 -> 重建 FAISS -> (可选) AI 批处理 |
| 修正已有文章 | 删除对应文章目录 -> 重新爬取 -> 重建数据库 -> 重建 FAISS |
| 仅更新 AI 处理 | 运行 `article_ai_batch run-all`（不影响数据库和 FAISS） |
| 完整重建 | 按顺序执行阶段 1 -> 2 -> 3 -> 4 |
| 新增数据源 | 创建新的链接文件和 `SAVE_ROOT` -> 修改 `BUSINESS_DATA_DIR` 配置 -> 全量重建 |

### 环境变量依赖

| 变量 | 用途 | 必需 |
|------|------|------|
| `GOOGLE_API_KEY` | Gemini 嵌入向量生成 | 是（FAISS 构建阶段） |
| `GEMINI_API_KEYS` | 批处理多 Key 轮换 | 推荐（AI 批处理阶段） |
| `PRIMARY_GEMINI_KEY` | 优先使用的 Gemini Key | 否 |

### 典型耗时参考

| 阶段 | 500 篇文章估算 |
|------|----------------|
| 爬虫（5 并发） | 10--30 分钟（取决于网络和目标站点响应速度） |
| 数据库构建 | 1--3 分钟 |
| FAISS 索引（50/批） | 5--15 分钟（取决于 Gemini Embedding API 速率限制） |
| AI 批处理（20 worker） | 30--90 分钟（取决于文章长度和 API 配额） |

### 常见问题排查

| 问题 | 排查方向 |
|------|----------|
| FAISS 索引构建失败 | 检查 `GOOGLE_API_KEY` 是否有效，API 配额是否充足 |
| AI 批处理卡住 | 查看 `state/` 目录下各 worker JSON 的状态，检查日志 |
| 数据库重建后文章数不对 | 确认 `BUSINESS_DATA_DIR` 路径正确，检查是否有 `content.txt` 缺失 |
| 翻译结果为空 | 检查 `TRANSLATION_FIELD_ALIASES` 是否覆盖了 AI 输出的字段名 |
| 标签过少 | 调整全局阈值（关键词 >= 6 次、系列 >= 8 次），或检查停用词列表 |
