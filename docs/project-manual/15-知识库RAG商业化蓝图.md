# 15 知识库 RAG 商业化蓝图

> 版本：v1.0  
> 日期：2026-04-16  
> 适用范围：To C 主产品，不包含企业租户版本  
> 状态：方案冻结，可进入实施拆解

---

## 15.1 文档目标

本文档定义复旦知识库后续升级为可直接商业化落地的 RAG 主方案，目标不是做一套演示系统，而是建立一套可以长期扩容、可持续运营、可被付费用户稳定使用的知识基础设施。

本文档重点回答两个问题：

1. 后续新增文章如何自动进入主知识库，并以稳定、可控、可观测的方式接入 RAG。
2. 用户私人知识库在收藏文章越来越多之后，如何继续保持检索准确、响应稳定、成本可控和产品可运营。

本文档采用的核心判断如下：

- 平台只建设一套规范化主语料层，不建设多套重复知识库。
- 用户私人知识库不是独立复制一份向量库，而是建立在主语料之上的作用域视图。
- 检索能力采用生产级混合检索路线：全文检索 + 向量检索 + 元数据过滤 + rerank + 引用回答。
- 内容入库必须异步化、可重试、可回溯，不能继续依赖“导入完就算已接入”的弱流程。
- 私人知识库规模化的关键不在于复制数据，而在于关系建模、作用域过滤、主题画像、路由和缓存。

---

## 15.2 面向 To C 的产品边界

本方案只面向 To C 主产品，因此做如下边界约束：

- 不做企业级租户隔离、多组织 ACL、部门权限树。
- 不做企业文档共享空间、管理员分组授权、组织级审计工作台。
- 用户的私人知识库默认由两类内容组成：
  - 平台公开文章的收藏与主题归档
  - 后续可能支持的用户私人上传文档
- 权限模型只保留 To C 必需的最小集合：
  - 公开内容
  - 付费内容
  - 用户私有内容
  - 用户私人主题范围

这意味着本方案可以把复杂度集中在检索质量、内容接入、成本控制和用户体验，而不把资源浪费在当前不需要的企业端功能上。

## 15.3 当前系统现状与主要瓶颈

### 15.3.1 已有能力

当前系统已经具备以下基础：

- 平台公开文章库已经存在统一 `articles` 表。
- 搜索服务已经具备三路混合检索雏形：
  - 词法打分
  - BM25
  - 本地 FAISS 向量检索
- 私人知识库已经具备主题和文章映射关系：
  - `user_knowledge_themes`
  - `user_knowledge_theme_articles`
- 主题聊天已经支持把 `selected_article_ids` 作为范围过滤条件传入检索链路。

### 15.3.2 当前主要瓶颈

当前实现可以支持原型与早期上线，但距离商业化生产方案仍有明显差距：

1. 检索粒度仍然偏文章级。
当前系统主要以文章为召回对象，而不是以 chunk 为召回对象。文章级召回在长文、综合报道、专题综述和跨主题材料里精度会明显不足。

2. 索引形态偏本地单体。
当前检索依赖进程内缓存、本地 BM25、本地 FAISS 文件，适合单机原型，不适合持续扩容、滚动更新和多实例部署。

3. 新文章接入链路不完整。
当前没有一条明确的、异步的、可回溯的“发布后自动进入 RAG”的入库流水线。

4. 私人知识库仍然是“关系层产品”，还不是“规模化检索层产品”。
当前私人知识库在几十篇文章内可以使用，但当收藏规模扩大到几百甚至几千篇时，直接按文章 ID 过滤会越来越重，也不利于做主题路由、个性化摘要和成本控制。

5. 缺少系统级观测与质量评估。
现在还缺少面向 RAG 的正式指标体系，例如召回率、引用覆盖率、无答案率、索引新鲜度、单位查询成本和 rerank 命中收益。

## 15.4 目标架构总览

### 15.4.1 一句话架构

平台采用“一套规范主语料 + 一套生产级检索引擎 + 多种用户作用域视图”的主方案。

### 15.4.2 架构图

```text
内容来源
  ├─ 编辑后台发布文章
  ├─ 爬虫/批量导入
  └─ 后续用户私人上传

        ↓

异步入库流水线
  ├─ 内容标准化
  ├─ 元数据抽取
  ├─ 版本判定
  ├─ 结构化切块
  ├─ Embedding
  ├─ 全文索引写入
  ├─ 向量索引写入
  └─ 状态回填为 searchable

        ↓

规范主语料层
  ├─ documents / articles
  ├─ document_versions
  ├─ document_chunks
  ├─ tags / topics / columns
  └─ profile 派生层

        ↓

检索服务层
  ├─ scope 解析
  ├─ query 理解
  ├─ hybrid retrieval
  ├─ metadata filtering
  ├─ rerank
  ├─ context packing
  └─ grounded answer

        ↓

产品层
  ├─ 全站搜索
  ├─ AI 助理
  ├─ 我的资料库
  ├─ 私人主题知识库
  └─ 个性化阅读与追踪
```

## 15.5 核心架构决策

### 15.5.1 决策一：只保留一套规范主语料，不复制多套知识库

所有公开文章只存一份权威正文、元数据和索引身份。  
用户收藏、私人主题、资料库、AI 问答，全部建立在同一套主语料之上。

这样做的好处：

- 避免重复 embedding 和重复索引成本。
- 避免一篇文章在多个库中版本不一致。
- 收藏关系和知识结构可以长期演进，而不用迁移重复数据。
- 主题、资料库、推荐、搜索都能共享同一底座。

### 15.5.2 决策二：私人知识库是作用域，不是另一套向量库

用户的私人知识库本质是三类作用域：

- 我的全部收藏
- 某个私人主题
- 当前手动选中的文章子集

因此，私人知识库不额外构建一套独立索引，而是在查询时把作用域解释为过滤条件和候选集合。

### 15.5.3 决策三：检索粒度升级为 chunk 级

主检索单位从文章升级为 chunk。  
引用展示仍然可以回到文章，但召回和排序必须基于 chunk。

这样做是因为：

- 一篇长文往往包含多个论点和多个主题。
- chunk 级召回可以减少“整篇相关文章被拉进来，但命中段落很少”的噪声。
- 后续时间线、对比、观点追踪、差异摘要都依赖更细粒度的证据块。

### 15.5.4 决策四：采用生产级混合检索，而不是单一向量检索

正式方案固定为：

- 全文检索
- 向量检索
- 元数据过滤
- RRF 或同级融合
- rerank
- grounded answer

原因很直接：

- 中文商业内容里，标题、机构、人物、专业名词依然高度依赖词法匹配。
- 纯向量检索在命名实体、时间点、具体政策名和上市公司简称场景下容易失真。
- 混合检索是当前业界最成熟、最稳定的生产路线。

### 15.5.5 决策五：所有新增内容都必须走异步入库流水线

文章发布成功不等于已进入 RAG。  
只有当解析、切块、嵌入、索引和状态回填全部完成后，内容才真正进入可检索状态。

## 15.6 产品对象模型

### 15.6.1 平台主知识库

平台主知识库是唯一的知识底座，包含：

- 平台公开文章
- 平台付费文章
- 已审核并允许进入主库的扩展内容

### 15.6.2 我的资料库

我的资料库是用户收藏过、标记过、归档过的全部内容集合。  
它不复制正文，只存关系、画像、缓存和派生信息。

### 15.6.3 私人主题知识库

私人主题知识库是用户在资料库之上创建的长期工作集，适合如下场景：

- 长期追踪一个主题
- 只围绕一组精选文章做 AI 分析
- 建立个人判断、时间线和结论归档

### 15.6.4 当前选中文章集

这是最窄的即时作用域，用于用户在主题页中勾选几篇材料后，与 AI 做定向分析。

## 15.7 目标数据架构

### 15.7.1 核心控制表

建议的核心表如下：

| 表名 | 用途 | 说明 |
|------|------|------|
| `articles` | 平台文章主表 | 继续作为公开文章权威主表 |
| `article_versions` | 文章版本表 | 记录正文变化与哈希，防止重复嵌入 |
| `article_chunks` | chunk 主表 | 每篇文章切块后的结构化片段 |
| `ingestion_jobs` | 入库任务表 | 记录每次入库状态、耗时、错误 |
| `user_saved_articles` | 用户收藏关系 | 我的资料库的基础关系层 |
| `user_knowledge_themes` | 用户主题表 | 延续现有表 |
| `user_knowledge_theme_articles` | 主题文章映射 | 延续现有表 |
| `user_library_profiles` | 资料库画像 | 用户资料库的聚合摘要与向量中心 |
| `user_theme_profiles` | 主题画像 | 单主题的摘要、标签、活跃度、中心向量 |
| `retrieval_events` | 检索日志 | 记录检索输入、候选、点击、命中、耗时 |
| `answer_events` | 回答日志 | 记录引用覆盖、模型成本、反馈结果 |

建议字段如下：

| 字段 | 说明 |
|------|------|
| `id` | chunk 主键 |
| `article_id` | 所属文章 |
| `version_id` | 所属版本 |

| `section_title` | 所在章节标题 |

| `is_active` | 是否可检索 |
| `content_hash` | 内容哈希 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 15.7.3 检索索引文档建议字段

Elastic 中的检索文档建议直接按 chunk 存储，并携带以下元数据：

- `chunk_id`
- `article_id`
- `article_title`
- `section_title`
- `chunk_text`
- `publish_date`
- `language`
- `tags`
- `columns`
- `topics`
- `orgs`
- `people`
- `access_scope`
- `owner_user_id`
- `is_paid`
- `is_public`
- `theme_hint_ids`

这使得搜索层可以直接做：

- 全文检索
- 向量检索
- 时间过滤
- 资料库过滤
- 主题过滤
- 仅公开或仅付费过滤

### 15.7.4 画像表的职责

画像表不是正文存储，而是派生层，用于做规模化与体验增强。

`user_library_profiles` 保存：

- 收藏总量
- 最近活跃时间
- 高频标签
- 高频机构
- 高频人物
- 全库摘要
- library centroid embedding
- 自动聚类结果摘要

`user_theme_profiles` 保存：

- 主题摘要
- 主题关键词
- 主题高频标签
- 主题最近更新
- 主题内文章数
- 主题内 chunk 数
- theme centroid embedding
- 推荐追问模板

## 15.8 新文章自动接入主知识库的正式链路

### 15.8.1 触发源

以下事件会触发异步入库：

- 编辑后台发布新文章
- 编辑后台更新已发布文章正文
- 批量导入文章
- 爬虫导入新文章
- 后续可能的用户私人上传

### 15.8.2 入库任务状态机

`ingestion_jobs.status` 建议至少包含：

- `pending`
- `parsing`
- `normalizing`
- `versioning`
- `chunking`
- `embedding`
- `indexing`
- `profiling`
- `ready`
- `failed`
- `dead_letter`

### 15.8.3 标准化阶段

标准化阶段需要完成：

1. 清洗正文中的冗余 HTML、样式残留和上传噪声。
2. 统一标题、发布日期、来源、栏目、标签、机构、人名等元数据。
3. 识别语言。
4. 生成正文哈希。
5. 判断这次是否属于内容变更。

原则：

- 如果正文哈希未变化，不重复做 embedding。
- 如果只有摘要、封面、展示字段变了，不重新切块。
- 如果正文结构变化，才触发完整重建。

### 15.8.4 结构化切块策略

切块必须采用“结构优先 + 大小控制 + 小重叠”的策略。

建议规则：

- 先按 H1/H2/H3、小标题、段落边界切分。
- 单 chunk 优先控制在 300 至 700 中文字等效长度。
- 太短的段落向上或向下合并。
- 重要引用、结论段、列表段尽量保留结构完整。
- 相邻 chunk 保留少量重叠，避免跨段断裂。

禁止做法：

- 整篇文章一个 embedding
- 完全固定字数生切
- 不保留章节语义

### 15.8.5 embedding 与索引写入

每个 chunk 进入两个索引面：

1. 全文索引
- 写入 chunk 文本与完整元数据

2. 向量索引
- 写入 chunk embedding

注意事项：

- embedding 模型必须固定版本，并记录在任务元数据中。
- 新模型上线时通过离线重建，不在线混用不同代 embedding。
- 索引写入要支持幂等重试。

### 15.8.6 可检索状态回填

文章只有在满足以下条件后才进入 `searchable`：

- 主版本已落库
- chunk 已生成
- 全文索引写入成功
- 向量索引写入成功
- 画像派生任务已至少完成基础摘要

### 15.8.7 失败与补偿

失败处理规则：

- 单篇失败可重试，不阻塞全局。
- 多次失败进入 `dead_letter` 队列。
- 后台必须提供“查看失败原因”和“重试入库”入口。

## 15.9 检索与回答主链路

### 15.9.1 统一检索入口

后续所有产品查询，都应统一走一个 Retrieval Service：

- 全站搜索
- AI 助理
- 我的资料库问答
- 私人主题问答
- 当前选中文章问答

统一入口的好处：

- 质量策略一致
- 日志可统一观测
- 成本可统一治理
- 产品迭代只改一条主链路

### 15.9.2 第一步：作用域解析

任何查询先解析 scope，再开始检索。

支持的 scope：

- `global_public`
- `member_public`
- `my_library`
- `my_theme`
- `selected_articles`

解析结果至少包含：

- user_id
- scope_type
- allowed_article_ids 或 allowed_theme_ids
- access_scope filter
- 是否需要主题画像预路由

### 15.9.3 第二步：查询理解

查询理解负责以下工作：

- 去噪与标准化
- 识别语言
- 识别时间条件
- 识别人物、机构、主题词
- 生成少量同义扩展词

规则：

- 扩展词必须保守，不能扩大语义边界。
- 不允许 LLM 把明确问题改写成模糊问题。

### 15.9.4 第三步：候选路由

当作用域较大时，不能直接全量检索。

资料库与主题库建议增加候选路由层：

- 如果是当前选中文章，直接跳过路由。
- 如果是单主题，优先用主题画像缩小语义范围。
- 如果是我的资料库，先按资料库画像与自动聚类筛出候选主题簇，再做正式检索。

### 15.9.5 第四步：混合检索

正式召回固定为：

1. lexical / BM25 召回
2. dense vector 召回
3. metadata filter
4. RRF 融合

返回对象是 chunk，不是 article。

### 15.9.6 第五步：rerank

rerank 分为两层：

1. 基础 rerank
- 所有付费 AI 查询都启用
- 用轻量模型或搜索引擎原生 rerank

2. 高成本 rerank
- 仅在复杂主题问答或长答案任务中启用
- 用更高质量模型做 top N 重排

原则：

- 不对所有免费检索都启用高成本 rerank。
- rerank 的预算必须纳入单次问答成本模型。

### 15.9.7 第六步：上下文打包

上下文打包不是简单拼接文本，而应遵循：

- 控制总 token
- 优先保留高相关 chunk
- 去重相似 chunk
- 保留标题和出处
- 保留时间信息

### 15.9.8 第七步：grounded answer

回答服务必须只基于已检索证据作答，并输出可追溯引用。

要求：

- 每个关键判断都能回指到来源 chunk 或文章。
- 当证据不足时明确说证据不足。
- 不能把模型常识写成知识库结论。

### 15.9.9 引用策略

正式产品里，引用要同时满足：

- 对用户可读
- 对系统可追踪

建议：

- 前台展示文章级来源
- 后台日志保留 chunk 级来源

## 15.10 私人知识库规模化方案

### 15.10.1 设计原则

私人知识库规模化的关键原则如下：

- 收藏变多时，不复制正文。
- 收藏变多时，不重复 embedding。
- 收藏变多时，不每次都直接拿大量文章 ID 硬过滤到底。
- 收藏变多时，靠主题画像、资料库画像和自动聚类缩小候选范围。

### 15.10.2 用户规模分层策略

建议按用户资料库规模分层处理：

| 资料库规模 | 处理策略 |
|------|------|
| 0 - 200 篇 | 直接按 `article_id` 过滤检索 |
| 200 - 2000 篇 | 启用资料库画像 + 主题画像路由 |
| 2000 篇以上 | 增加自动聚类层，先选簇再检索 |

### 15.10.3 自动聚类

当用户收藏规模变大时，系统应异步生成自动聚类：

- 按主题语义聚类
- 按时间线聚类
- 按机构/人物聚类

自动聚类的用途：

- 提高大库问答的首轮召回精度
- 为用户提供“自动主题建议”
- 降低大范围检索成本

### 15.10.4 主题画像

主题画像要承担三项职责：

1. 检索前路由
2. 主题卡片与主题页文案
3. AI 问答的默认背景上下文

### 15.10.5 资料库画像

资料库画像用于：

- 生成“我的资料库”摘要
- 辅助推荐追踪方向
- 识别用户长期关注主题
- 缩小全库问答时的候选范围

### 15.10.6 冷热分层

建议对用户资料库增加冷热分层：

- 热数据：最近收藏、最近访问、最近问过的问题相关内容
- 冷数据：长期未访问内容

用途：

- 提高缓存命中
- 降低 rerank 成本
- 为 UI 提供“最近工作集”

## 15.11 成本控制与性能策略

### 15.11.1 embedding 成本控制

- 只对正文版本变化的内容重新 embedding
- 批量入库时使用异步队列
- 避免主题和收藏关系引发重复 embedding

### 15.11.2 检索成本控制

- 优先让搜索引擎完成多数候选裁剪
- 只对 top N 候选做 rerank
- 不对每次普通搜索都调用高成本 LLM

### 15.11.3 回答成本控制

- 主题页快捷问题可优先复用回答缓存
- 相同 scope + 相同 query + 相同文档版本的请求可命中 answer cache
- 长答案任务设置更严格的配额与频率控制

### 15.11.4 缓存设计

建议至少建立三层缓存：

1. retrieval cache
- scope + normalized query + filter hash

2. profile cache
- user library profile
- theme profile

3. answer cache
- scope hash + query hash + source version hash

## 15.12 推荐技术栈

### 15.12.1 主推荐方案

面向 To C 商业化落地，推荐如下技术栈：

- 主数据库：PostgreSQL
- 搜索与向量：Elastic Cloud / Elasticsearch
- 队列与任务：Redis + Celery
- 对象存储：S3 / OSS / Blob
- API 服务：FastAPI
- 前端：继续沿用 React

### 15.12.2 为什么推荐 Elastic 主方案

原因如下：

- 全文检索和向量检索天然在同一平台内协同
- 元数据过滤、时间过滤、标签过滤、排序、观测更成熟
- 后续从全站搜索、AI 助理到私人知识库都能走统一检索平面
- 比继续维护本地 BM25 + FAISS + 进程缓存更适合商业化

### 15.12.3 为什么不建议继续以当前本地索引形态扩展

不建议继续把以下形态当作长期方案：

- SQLite 作为最终生产数据库
- 本地 FAISS 文件作为长期在线向量库
- 进程内全量文章缓存作为主要搜索底座

这些形态对单机验证足够，但在以下场景会明显失效：

- 多实例部署
- 高频增量更新
- 数据量增长
- 检索分析与观测
- 回滚与重建

## 15.13 面向 To C 的分期落地路线

### 15.13.1 第一阶段：主语料正规化

目标：

- 建立 `article_versions`
- 建立 `article_chunks`
- 建立 `ingestion_jobs`
- 新文章发布后自动入队

- 内容何时可检索有明确定义

### 15.13.2 第二阶段：检索底座升级

目标：

- 将检索从本地索引迁移到生产级混合检索服务
- 检索对象从文章升级为 chunk
- 增加 metadata filter 和正式 rerank

交付结果：

- 全站搜索与 AI 检索质量提升
- 私人主题检索开始具备规模化基础

### 15.13.3 第三阶段：私人知识库规模化

目标：

- 上线 `user_saved_articles`
- 上线 `user_library_profiles`
- 上线 `user_theme_profiles`
- 上线自动聚类和候选路由

交付结果：

- 用户收藏越来越多时，资料库仍然可控可用
- 私人知识库从“文章列表”升级为“长期知识工作空间”

### 15.13.4 第四阶段：个性化运营增强

目标：

- 个性化主题建议
- 主题追踪提醒
- 资料库定期简报

交付结果：

- 用户愿意长期维护私人知识库
- 付费价值从“搜索”升级为“持续判断辅助”

## 15.15 面向产品与运营的用户体验要求

### 15.15.1 新文章接入体验

用户不应感知“系统在建索引”，但系统必须有真实状态。

建议：

- 编辑后台看到“已发布 / 索引中 / 可检索”
- 主题页或搜索页只对 `searchable` 内容开放问答

### 15.15.2 私人知识库体验

私人知识库必须从“收藏夹”升级为“工作台”。

核心体验包括：

- 我的资料库总览
- 自动推荐主题
- 主题内精确问答
- 一组选中文章的比较与总结
- 最近工作集
- 主题摘要自动更新

### 15.15.3 用户长期留存体验

资料库规模变大后，不能让用户只看到越来越长的列表。  
系统必须逐步提供：

- 自动聚类
- 自动追踪
- 自动时间线
- 自动判断变化摘要

## 15.17 实施时必须避免的错误路线

必须明确避免以下路线：

1. 给每个用户复制一套独立向量库。
2. 继续以文章级 embedding 作为长期主方案。
3. 把发布成功误认为已接入 RAG。
4. 用一次性脚本重建索引替代正式增量入库。
5. 用大模型生成回答替代检索质量建设。
6. 当收藏规模增长后，继续每次直接拿大量 article_id 全量过滤到底。

## 15.19 下一步实施顺序

正式实施建议严格按以下顺序推进：

1. 先建数据模型与异步入库任务流。
2. 再建 chunk 与新索引。
3. 再切检索服务。
4. 再把私人主题与我的资料库接到新检索层。
5. 最后做主题画像、资料库画像和自动聚类。

原因：

- 如果先做前端产品，不做底座，后面会全部返工。
- 如果先做私人知识库规模化，不先解决 chunk 和正式入库，质量不会稳定。
- 如果先做个性化运营，不先解决资料库画像，推荐只会变成噪声。

## 15.21 最终结论

面向 To C 主产品，最正确的长期路线不是再做一套“私人向量库”，而是：

- 建立一套规范主语料
- 建立一条异步、可观测的新文章入库链路
- 建立一套生产级 chunk 级混合检索底座
- 用作用域、主题画像和资料库画像来承接私人知识库规模化

这条路线能够同时满足：

- 商业化上线
- 后续规模增长
- 私人知识库长期可用
- 搜索与 AI 问答质量稳定

它也是当前项目从“可运行原型”迈向“可持续产品基础设施”的必要升级路径。
