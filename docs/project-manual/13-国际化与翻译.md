## 13.1 前端 i18n --- `LanguageProvider`

### 13.1.1 架构概述

前端国际化基于 React Context 实现，核心文件位于 `frontend/src/i18n/` 目录：

| 文件 | 职责 |
|------|------|
| `LanguageContext.js` | 创建 React Context 对象 |
| `LanguageProvider.jsx` | Context Provider 组件，封装语言状态管理和检测逻辑 |
| `messages.js` | 中英文消息键值对树 |

### 13.1.2 支持的语言

```javascript
const SUPPORTED_LANGUAGES = new Set(['zh', 'en'])
```

系统支持两种语言：`zh`（中文，默认）和 `en`（英文）。任何不在此集合中的语言标识都会被规范化为 `zh`。

### 13.1.3 语言检测优先级 --- `detectInitialLanguage()`

页面加载时，`LanguageProvider` 按以下顺序检测用户语言偏好：

| 优先级 | 检测源 | 说明 | 示例 |
|--------|--------|------|------|
| 1 (最高) | URL 查询参数 `?lang=` | 适用于分享链接和外部跳转 | `?lang=en` |
| 2 | `localStorage` 键 `fdsm-language` | 用户上次选择的语言偏好 | `localStorage.getItem('fdsm-language')` |
| 3 (最低) | 浏览器 `navigator.language` | 浏览器默认语言 | `en-US` -> `en`；`zh-CN` -> `zh` |
| 回退 | 固定值 `'zh'` | SSR 环境或所有检测均未命中 | --- |

**详细逻辑：**

```javascript
function detectInitialLanguage() {
  if (typeof window === 'undefined') return 'zh'       // SSR 回退

  // 1. URL 参数
  const searchParams = new URLSearchParams(window.location.search)
  const queryLang = normalizeLanguage(searchParams.get('lang'))
  if (SUPPORTED_LANGUAGES.has(searchParams.get('lang') || '')) return queryLang

  // 2. localStorage
  const storedLanguage = normalizeLanguage(window.localStorage.getItem(STORAGE_KEY))
  if (window.localStorage.getItem(STORAGE_KEY)) return storedLanguage

  // 3. 浏览器语言
  const browserLanguage = String(window.navigator.language || '').toLowerCase()
  return browserLanguage.startsWith('en') ? 'en' : 'zh'
}
```

注意第 1 步的检测逻辑：只有当 `lang` 参数的原始值本身是合法语言代码时才使用（而非只看规范化后的结果），这避免了 `?lang=xyz` 被错误规范化为 `zh` 并覆盖 localStorage 的偏好。

### 13.1.4 语言状态副作用

当 `language` 状态变化时，`useEffect` 执行以下三个副作用：

```javascript
useEffect(() => {
  if (typeof window === 'undefined') return

  // 1. 持久化到 localStorage
  window.localStorage.setItem(STORAGE_KEY, language)

  // 2. 更新 HTML lang 属性
  document.documentElement.lang = language === 'en' ? 'en' : 'zh-CN'

  // 3. 同步 URL 参数
  const url = new URL(window.location.href)
  if (language === 'en') {
    url.searchParams.set('lang', 'en')          // 英文模式添加 ?lang=en
  } else {
    url.searchParams.delete('lang')              // 中文模式移除 lang 参数
  }
  window.history.replaceState({}, '', `${url.pathname}${url.search}${url.hash}`)
}, [language])
```

| 副作用 | 说明 |
|--------|------|
| `localStorage.setItem` | 持久化偏好，下次访问自动恢复 |
| `document.documentElement.lang` | 设置 HTML 根元素 `lang` 属性（`en` 或 `zh-CN`），影响浏览器字体渲染和可访问性 |
| URL 参数同步 | 英文模式下 URL 带 `?lang=en`，中文模式移除参数。使用 `replaceState` 不产生浏览历史 |

### 13.1.5 Context 提供的值

`LanguageProvider` 向子组件提供以下 Context 值：

| 属性/方法 | 类型 | 说明 |
|-----------|------|------|
| `language` | `string` | 当前语言代码（`'zh'` 或 `'en'`） |
| `isEnglish` | `boolean` | 是否为英文模式的快捷判断 |
| `setLanguage(lang)` | `function` | 设置指定语言（会经过 `normalizeLanguage` 校验） |
| `toggleLanguage()` | `function` | 在中英文之间切换（`en` -> `zh`，`zh` -> `en`） |
| `t(key)` | `function` | 翻译函数，根据当前语言返回对应文本 |

### 13.1.6 翻译函数 `t(key)` --- `resolveMessage()`

`t(key)` 接受点分隔的键路径（如 `'navbar.home'`），按以下逻辑解析：

```
1. 在当前语言的 messages 中查找 key
   └─ 找到 string 类型的值 → 返回
   └─ 未找到或不是 string →

2. 在中文 messages (messages.zh) 中查找 key（回退到中文）
   └─ 找到 string 类型的值 → 返回
   └─ 未找到 →

3. 返回 key 本身作为兜底
```

这确保了即使英文翻译缺失，也能显示中文文案而非空白。如果中文也不存在，显示 key 路径（如 `'navbar.unknownKey'`），便于开发者发现缺失的翻译。

## 13.3 后端翻译服务 --- `translation_service.py`

### 13.3.1 三层翻译策略

后端翻译服务 `get_article_translation()` 采用三层回退策略解析英文翻译，确保在各种场景下都能返回可用结果：

```
请求 GET /api/article/{id}/translation?lang=en
    │
    ▼
 第一层：article_ai_outputs（预计算翻译）
    │
    │  检查条件：translation_ready == True
    │            AND source_hash_matches_current == True
    │
    ├─ 命中 → 返回预计算翻译（cached: True, model: "precomputed"）
    │
    ▼
 第二层：article_translations（翻译缓存表）
    │
    │  查询：WHERE article_id = ? AND target_lang = ? AND source_hash = ?
    │
    ├─ 命中 → 返回缓存翻译（cached: True, model: 缓存记录的模型名）
    │
    ▼
 第三层-a：英文原文直通
    │
    │  检查条件：_is_english_source_article() == True
    │           （源文章前4000字符不包含CJK字符）
    │
    ├─ 命中 → 原文直通，model = "source-pass-through"
    │         同时写入 article_translations 缓存
    │
    ▼
 第三层-b：Gemini Flash 实时翻译
    │
    │  前提：is_ai_enabled() == True
    │
    ├─ 调用 translate_article_to_english()
    │  翻译完成后写入 article_translations 缓存
    │  返回（cached: False, model: 实际使用的模型名）
    │
    ▼
 兜底：抛出 RuntimeError
       "Gemini Flash translation is not configured."
       或 "English translation is temporarily unavailable..."
```

### 13.3.2 第一层 --- 预计算翻译 (`article_ai_outputs`)

由 `article_ai_batch.py` 批处理脚本预先生成并存储。通过 `get_article_ai_output_detail(article_id)` 查询。

**命中条件（两者同时满足）：**
- `translation_ready == True` --- 翻译已完成
- `source_hash_matches_current == True` --- 源文章未发生变化（source_hash 一致）

**返回字段来源：**

| 翻译字段 | AI 输出字段 | 说明 |
|----------|-------------|------|
| `title` | `translation_title_en` | 英文标题 |
| `excerpt` | `translation_excerpt_en` | 英文摘要句 |
| `summary` | `translation_summary_en` | 英文摘要（经 `normalize_summary_display_markdown` 处理） |
| `content` | `formatted_markdown_en` 或 `translation_content_en` | 英文正文（优先取排版后版本） |
| `summary_html` | `summary_html_en` | 摘要 HTML 预览 |
| `html_web` / `html_wechat` | `html_wechat_en` | Web/微信 HTML 版本 |

如果文章处于锁定状态（`access.locked == True`），内容会经过 `preview_markdown()` 截断处理。

### 13.3.3 第二层 --- 翻译缓存 (`article_translations`)

数据库查询：

```sql
SELECT title, excerpt, summary, content, model, created_at, updated_at
FROM article_translations
WHERE article_id = ? AND target_lang = ? AND source_hash = ?
```

**缓存键组成：**
- `article_id` --- 文章 ID
- `target_lang` --- 目标语言（当前仅 `"en"`）
- `source_hash` --- 源文章内容哈希

`source_hash` 由 `build_article_source_hash(article)` 计算。当源文章内容发生变化时，`source_hash` 会改变，旧缓存自动失效，触发重新翻译。

### 13.3.4 第三层-a --- 英文原文直通

`_is_english_source_article(article)` 检测源文章是否已经是英文：

```python
sample = " ".join(
    str(part or "").strip()
    for part in (article.get("title"), article.get("main_topic"),
                 article.get("excerpt"), article.get("content"))
    if str(part or "").strip()
)
return not contains_cjk(sample[:4000])
```

取文章标题、主话题、摘要、正文拼接后的前 4000 字符，调用 `contains_cjk()` 检测是否包含 CJK 字符。如果不包含，视为英文原文。

**直通翻译的处理 (`_build_passthrough_translation`)：**

| 字段 | 来源 |
|------|------|
| `title` | 原文标题（直接使用） |
| `excerpt` | `main_topic` 或 `excerpt` |
| `content` | 原文内容经 `normalize_article_display_markdown()` 处理 |
| `summary` | 从 excerpt 或 `preview_markdown()` 生成 |
| `model` | `"source-pass-through"` |

结果写入 `article_translations` 缓存，后续请求命中第二层。

### 13.3.5 第三层-b --- Gemini Flash 实时翻译

当以上层均未命中时，调用 `ai_service.translate_article_to_english()` 进行实时翻译：

```python
translated = translate_article_to_english(
    article["title"],
    article.get("main_topic") or article.get("excerpt") or "",
    article.get("content") or "",
)
```

翻译完成后通过 `upsert_article_translation()` 写入 `article_translations` 缓存，下次请求直接命中第二层。

**前提条件：** `is_ai_enabled()` 返回 `True`（Gemini API 已配置）。否则抛出 `RuntimeError`。

### 13.3.6 翻译响应结构

所有层返回统一的响应结构：

```python
{
    "article_id": int,              # 文章 ID
    "language": "en",               # 目标语言
    "source_hash": str,             # 源文章哈希
    "title": str,                   # 英文标题
    "excerpt": str,                 # 英文摘要句
    "summary": str,                 # 英文 Markdown 摘要
    "summary_html": str | None,     # 摘要 HTML 预览
    "content": str,                 # 英文正文
    "html_web": str | None,         # Web HTML 版本
    "html_wechat": str | None,      # 微信 HTML 版本
    "model": str,                   # 翻译模型名
    "cached": bool,                 # 是否来自缓存
    "content_scope": str,           # "full" 或 "preview"
    "access_locked": bool,          # 文章是否被锁定
    "updated_at": str,              # 翻译更新时间
}
```

### 13.3.7 HTML 渲染解析

**`_resolve_translation_html()`**

检查已有 HTML 是否为有效的复旦微信预览 HTML（通过 `is_fudan_wechat_preview_html()` 检测）。如果是，直接复用；否则返回 `None`。

**`_resolve_summary_html()`**

1. 检查存储的 HTML 是否为有效摘要预览 HTML（通过 `is_summary_preview_html()` 检测）
2. 如果有效，直接使用
3. 否则调用 `render_summary_preview_html(summary, language)` 生成新的摘要预览 HTML

## 13.4 翻译内容格式

### 13.4.1 四个翻译字段

翻译结果包含四个核心内容字段：

| 字段 | 说明 | 典型长度 |
|------|------|----------|
| `title` | 英文标题 | 10--100 字符 |
| `excerpt` | 英文短摘要/导语 | 50--200 字符 |
| `summary` | 英文 Markdown 摘要 | 200--800 字符 |
| `content` | 英文全文（Markdown） | 1000--10000+ 字符 |

### 13.4.2 Markdown 后处理

翻译内容在返回前经过两个后处理函数：

| 函数 | 适用字段 | 功能 |
|------|----------|------|
| `normalize_article_display_markdown(content, language)` | `content` | 规范化文章展示 Markdown |
| `normalize_summary_display_markdown(summary, language)` | `summary` | 规范化摘要展示 Markdown |

两个函数均接受 `language` 参数以进行语言相关的格式调整。

### 13.4.3 Source Hash 机制

`source_hash` 是翻译缓存失效的关键机制：

```
文章源数据 → build_article_source_hash() → SHA 哈希
    │
    ▼
翻译缓存查询时：
    WHERE source_hash = 当前哈希
    └─ 匹配 → 缓存有效
    └─ 不匹配 → 缓存失效，重新翻译
```

当文章内容被编辑后，`source_hash` 变化，旧翻译自动作废，下次请求时触发重新翻译。

## 13.5 锁定文章预览翻译

### 13.5.1 什么是锁定文章

当用户的会员等级不足以阅读完整文章时，文章处于"锁定"状态（`access.locked == True`）。此时用户只能看到文章的预览部分。

### 13.5.2 预览翻译处理流程

```
锁定文章翻译请求
    │
    ▼
 正常走三层翻译策略获取完整翻译
    │
    ▼
 content = preview_markdown(full_content)    ← 截断正文
    │
    ▼
 返回响应：
   content_scope: "preview"
   access_locked: true
```

**关键行为：**

| 字段 | 锁定文章 | 完整文章 |
|------|----------|----------|
| `content` | `preview_markdown(full_content)` 截断后的预览 | 完整正文 |
| `content_scope` | `"preview"` | `"full"` |
| `access_locked` | `true` | `false` |
| `summary` | 完整摘要（不截断） | 完整摘要 |
| `summary_html` | 完整摘要 HTML | 完整摘要 HTML |

即使文章被锁定，摘要（`summary`）始终完整返回，仅正文（`content`）被截断。

### 13.5.3 预计算翻译的锁定处理

第一层（预计算翻译）在锁定场景下的特殊处理：

```python
full_content = str(
    precomputed.get("formatted_markdown_en") or
    precomputed.get("translation_content_en") or ""
)
content = preview_markdown(full_content) if article.get("access", {}).get("locked") else full_content
```

优先使用排版后的英文 Markdown（`formatted_markdown_en`），其次使用翻译原文（`translation_content_en`）。

### 13.5.4 直通翻译的锁定处理

英文原文直通也尊重锁定状态：

```python
access_locked = bool(article.get("access", {}).get("locked"))
html = _resolve_translation_html(
    ...,
    stored_html=None if access_locked else article.get("html_wechat"),
)
```

锁定文章不返回微信 HTML 版本，避免通过 HTML 泄露完整内容。

### 13.5.5 前端翻译状态提示

前端根据 `content_scope` 和 `access_locked` 展示不同的翻译状态提示：

| 条件 | 显示文案（中文） | 显示文案（英文） |
|------|------------------|------------------|
| 锁定 + 预览翻译 | 当前英文版仅覆盖你现在可见的预览内容。 | This English version currently covers only the preview content visible to you. |
| 完整翻译 | 全文英文版 | Full translation |
| 预览翻译 | 预览英文版 | Preview translation |

## 13.6 列表翻译 --- `?language=en` 参数

### 13.6.1 列表接口的语言参数

多个列表接口支持 `language` 查询参数以返回英文版内容：

| 接口 | 参数 | 默认值 |
|------|------|--------|
| `GET /api/home/feed` | `language=zh\|en` | `zh` |
| `GET /api/articles/latest` | `language=zh\|en` | `zh` |
| `GET /api/time_machine` | `language=zh\|en` | `zh` |
| `GET /api/billing/plans` | `language=zh\|en` | `zh` |
| `GET /api/billing/me` | `language=zh\|en` | `zh` |
| `POST /api/search` | `request.language` | `zh` |
| `POST /api/chat` | `request.language` | 自动检测 |

### 13.6.2 列表翻译与详情翻译的区别

| 维度 | 列表翻译 | 详情翻译 |
|------|----------|----------|
| 触发方式 | `?language=en` 查询参数 | `GET /api/article/{id}/translation?lang=en` |
| 翻译范围 | 卡片标题、摘要等轻量字段 | 完整正文、摘要、HTML |
| 实时翻译 | 否（仅返回已有翻译或中文原文） | 是（可触发 Gemini 实时翻译） |
| 缓存机制 | 由 `catalog_service` 在列表查询时处理 | 三层翻译策略 |

### 13.6.3 对话系统的语言处理

对话接口 (`POST /api/chat`) 的语言处理由 `_normalize_language()` 决定：

```python
def _normalize_language(requested_language, text=""):
    normalized = "en" if str(requested_language or "").strip().lower() == "en" else "zh"
    if contains_cjk(text):
        return "zh"
    return normalized
```

优先级：
1. 如果用户消息包含 CJK 字符 -> 强制中文
2. 否则使用 `request.language` 指定的语言
3. 默认中文

这确保了即使界面语言设为英文，当用户用中文提问时，AI 也以中文回复。

## 13.7 翻译系统性能考量

### 13.7.1 缓存层次

| 层次 | 位置 | 写入时机 | 失效条件 |
|------|------|----------|----------|
| L1 | `article_ai_outputs` | 批处理脚本写入 | `source_hash` 变化 |
| L2 | `article_translations` | 实时翻译后写入 | `source_hash` 变化 |
| L3 | 无缓存 | --- | 每次触发 Gemini API |

### 13.7.2 翻译成本优化

1. **批处理预计算**：通过 `article_ai_batch.py` 在离线状态下批量翻译所有文章，避免运行时 API 调用
2. **Source Hash 精确失效**：只有源文章内容变化时才重新翻译，避免无谓的重复翻译
3. **英文直通**：自动检测英文原文，跳过翻译
4. **模型链回退**：Flash 模型失败时回退到 Chat 模型，提高翻译成功率

### 13.7.3 翻译质量保障

1. Prompt 明确要求"独立于摘要和排版，直接从中文原文翻译"，避免二次加工导致的信息损失
2. `TRANSLATION_FIELD_ALIASES` 容错机制应对 AI 输出格式变化
3. Markdown 后处理（`normalize_article_display_markdown` / `normalize_summary_display_markdown`）确保输出格式一致
4. 预览截断只作用于 `content` 字段，`summary` 始终完整
