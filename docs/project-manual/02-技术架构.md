## 1. 整体架构概览

本系统采用 **前后端完全分离** 的架构设计。前端为 React 单页应用（SPA），部署于 Vercel CDN；后端为 Python FastAPI 服务，部署于 Render 云平台。两者通过 HTTP REST API 通信，认证由 Supabase Auth 统一管理。

### 1.2 数据流简述

1. **用户请求**：浏览器向 Vercel 发起静态资源请求，获取 React SPA 资源包。
2. **API 调用**：SPA 内部通过 `fetch` / API 模块向 FastAPI 后端发起 REST 请求，请求头携带 Supabase JWT。
3. **认证校验**：后端 `supabase_auth_service` 使用 Supabase Admin API 校验 JWT，提取用户身份。
4. **业务处理**：Router 层委托 Service 层执行业务逻辑（文章检索、AI 对话、编辑工坊等）。
5. **混合检索**：RAG 引擎同时调用 FAISS 向量检索和 BM25 关键词检索，融合排序后交由 Gemini 大模型生成回答。
6. **响应返回**：JSON 格式响应通过 HTTPS 返回前端，SPA 渲染结果。

## 2. 架构关键原则

### 2.1 前后端解耦

前端和后端是两个完全独立的代码模块，各自拥有独立的包管理（`package.json` / `requirements.txt`）、独立的构建流程（Vite build / Uvicorn run）、独立的部署目标（Vercel / Render）。它们仅通过 HTTP REST API 契约进行通信，任何一端的版本升级不会强制要求另一端同步变更。

### 2.2 无状态认证

系统采用 **Supabase Auth** 提供的 JWT Bearer Token 认证模式。后端不维护任何 session 或 cookie 状态，每一个请求都是自包含的——JWT 中携带了用户 ID、邮箱和过期时间。后端仅负责 **校验** token 的有效性（通过 Supabase Admin API），不负责签发 token。这使得后端天然无状态，适合水平扩展。

### 2.3 混合检索（Hybrid Retrieval）

知识问答（Chat）功能采用 **FAISS 向量检索 + BM25 关键词检索** 的混合检索策略：

| 检索方式 | 技术实现 | 优势 |
|---------|---------|------|
| **语义检索** | FAISS（faiss-cpu）+ Gemini Embedding | 捕获语义相似性，对同义改写鲁棒 |
| **关键词检索** | rank_bm25（BM25 算法） | 精确匹配专有名词、人名、机构名 |

两路检索结果经过 **倒数排名融合（RRF）** 或分数归一化后合并排序，最终上下文送入 Gemini 大模型进行 RAG 生成。

### 2.4 渐进增强

系统设计遵循渐进增强理念：

- **认证可选**：`DEV_AUTH_ENABLED` 开关允许开发环境跳过认证，方便本地调试。
- **支付可选**：`PAYMENTS_ENABLED` 开关控制是否启用真实支付流程，`PAYMENT_PROVIDER` 支持 `mock` 模式。
- **AI 可降级**：当 Gemini API 配额耗尽时，系统可回退到纯检索模式，不影响基础文章浏览功能。
- **多 API Key 轮转**：`GEMINI_API_KEYS` 支持逗号分隔的多 Key 配置，实现 Key 轮转和限流规避。

### 2.5 双重数据存储

| 存储层 | 技术 | 存储内容 | 特点 |
|-------|------|---------|------|
| **关系型** | SQLite | 文章元数据、标签、栏目、专题、用户互动、会员、账单、聊天记录 | ACID 事务、SQL 查询、单文件便携 |
| **向量索引** | FAISS | 文章文本的 Gemini Embedding 向量 | 高维近邻搜索、毫秒级响应 |

SQLite 作为唯一的结构化数据库，通过 Render Persistent Disk 持久化。FAISS 索引文件同样存储在磁盘上，服务启动时加载进内存。

### 3.1 运行时与框架

| 组件 | 说明 |

| **FAISS (faiss-cpu)** | Facebook 开源的向量相似度搜索库，CPU 版本，存储 Gemini Embedding 向量索引 |

### 3.3 AI 与检索

| 组件 | 说明 |
|------|------|

### 3.4 Gemini 模型配置

系统配置了 **4 个不同用途的 Gemini 模型**：

| 配置变量 | 默认值 | 用途 |
|---------|--------|------|
| `GEMINI_CHAT_MODEL` | `gemini-3-flash` | RAG 对话生成，知识问答主模型；当前运行时映射到 `gemini-3-flash-preview` |
| `GEMINI_FLASH_MODEL` | `gemini-2.5-flash` | 批量文章处理、摘要生成等高吞吐场景 |
| `GEMINI_EMBEDDING_MODEL` | `models/gemini-embedding-001` | 文本向量化，用于 FAISS 索引构建和查询 |
| `GEMINI_EDITORIAL_FORMAT_MODEL` | `gemini-3-flash-preview`（硬编码） | 编辑工坊排版格式化，使用最新预览模型 |

## 5. 部署方案

### 5.1 后端部署 — Render

| 配置项 | 值 |
|-------|-----|
| **平台** | Render Web Service |
| **运行时** | Python 3.11 |
| **启动命令** | `uvicorn backend.main:app --host 0.0.0.0 --port $PORT` |
| **健康检查端点** | `GET /` 返回 `{"status": "ok", "service": "复旦管院商业智识库 API", "scope": "business-only"}` |
| **持久化存储** | Render Persistent Disk，挂载数据目录（SQLite DB、FAISS 索引、音频文件、上传文件） |
| **环境变量** | 通过 Render Dashboard 配置（见第 7 节） |

**启动流程：**

```
1. ensure_database_ready()      — 检查 SQLite 数据库就绪，包含所有必要表
2. ensure_runtime_tables()      — 创建运行时表（如缺失）
3. MEDIA_UPLOADS_DIR.mkdir()    — 确保上传目录存在
4. sync_local_audio_library()   — 同步本地音频库元数据
5. FastAPI app 启动              — 注册中间件、挂载静态文件、注册路由
6. Uvicorn 开始监听              — 默认端口 8000（Render 通过 $PORT 覆盖）
```

### 5.4 CORS 配置

后端通过 FastAPI `CORSMiddleware` 配置跨域策略：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=ALLOWED_ORIGINS or ["*"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

`ALLOWED_ORIGINS` 通过环境变量指定，多个域名用逗号分隔。如未配置则默认允许所有来源。

## 7. 环境变量说明

### 7.1 后端环境变量

所有后端环境变量通过 `python-dotenv` 从项目根目录 `.env` 文件加载，或通过部署平台（Render Dashboard）注入。

#### AI 模型配置

| 变量名 | 类型 | 默认值 | 必须 | 说明 |
|--------|------|--------|------|------|
| `GOOGLE_API_KEY` | string | `""` | 推荐 | Google AI 主 API Key，优先使用 |
| `GEMINI_API_KEYS` | string | `""` | 推荐 | 逗号分隔的多 Gemini API Key 列表，用于轮转 |
| `GEMINI_CHAT_MODEL` | string | `gemini-3-flash` | 否 | RAG 对话使用的模型；当前运行时映射到 `gemini-3-flash-preview` |
| `GEMINI_FLASH_MODEL` | string | `gemini-2.5-flash` | 否 | 批量处理使用的快速模型 |
| `GEMINI_EMBEDDING_MODEL` | string | `models/gemini-embedding-001` | 否 | 文本向量化模型 |

> **注意：** `GEMINI_CHAT_MODEL` 的配置值默认是 `gemini-3-flash`，为兼容当前 Gemini `v1beta` 接口，运行时会映射到 `gemini-3-flash-preview`。`GEMINI_EDITORIAL_FORMAT_MODEL` 仍硬编码为 `gemini-3-flash-preview`，不可通过环境变量覆盖。

**Key 选择逻辑：** `PRIMARY_GEMINI_KEY = GOOGLE_API_KEY or GEMINI_API_KEYS[0]`。如果设置了 `GOOGLE_API_KEY`，它将作为主 Key；否则使用 `GEMINI_API_KEYS` 列表中的第一个。

#### 认证与权限

| 变量名 | 类型 | 默认值 | 必须 | 说明 |
|--------|------|--------|------|------|
| `SUPABASE_URL` | string | `""` | 生产必须 | Supabase 项目 URL（如 `https://xxx.supabase.co`） |
| `SUPABASE_ANON_KEY` | string | `""` | 生产必须 | Supabase 匿名公钥（anon key） |
| `SUPABASE_AUTH_TIMEOUT_SECONDS` | float | `8` | 否 | Supabase Auth API 调用超时时间（秒） |
| `ADMIN_EMAILS` | string | `""` | 否 | 管理员邮箱白名单，逗号分隔（如 `admin@fudan.edu.cn,ops@example.com`） |
| `DEV_AUTH_ENABLED` | string | `"0"` | 否 | 设为 `"1"` 启用开发模式认证旁路，跳过 JWT 校验 |

#### 支付与计费

| 变量名 | 类型 | 默认值 | 必须 | 说明 |
|--------|------|--------|------|------|
| `PAYMENTS_ENABLED` | string | `"0"` | 否 | 设为 `"1"` 启用真实支付流程 |
| `PAYMENT_PROVIDER` | string | `"mock"` | 否 | 支付提供商（`mock` 为模拟模式） |

#### 网络与跨域

| 变量名 | 类型 | 默认值 | 必须 | 说明 |
|--------|------|--------|------|------|
| `ALLOWED_ORIGINS` | string | `"*"` | 推荐 | 允许跨域访问的前端域名，逗号分隔（如 `https://app.example.com,http://localhost:5173`） |
| `SITE_BASE_URL` | string | `http://127.0.0.1:4173` | 否 | 网站基础 URL，用于生成绝对链接 |

#### 数据目录

| 变量名 | 类型 | 默认值 | 必须 | 说明 |
|--------|------|--------|------|------|
| `FDSM_DATA_DIR` | string | 项目根目录 | 否 | 数据文件根目录（SQLite DB、FAISS 索引、上传文件等的父目录） |

### 7.2 前端环境变量

Vite 项目中，环境变量必须以 `VITE_` 为前缀才会被注入到客户端代码中。

| 变量名 | 类型 | 说明 |
|--------|------|------|
| `VITE_API_BASE_URL` | string | 后端 API 基础地址（如 `https://api.example.com` 或 `http://localhost:8000`） |
| `VITE_SUPABASE_URL` | string | Supabase 项目 URL（与后端 `SUPABASE_URL` 相同） |
| `VITE_SUPABASE_ANON_KEY` | string | Supabase 匿名公钥（与后端 `SUPABASE_ANON_KEY` 相同） |

> **安全提示：** `.env` 文件已被 `.gitignore` 排除，不会提交至版本控制。前端 `VITE_` 变量会被编译进客户端 JS 包中，因此只能放入公开安全的值（如 anon key），绝不能放入 service_role key 等私密凭证。
