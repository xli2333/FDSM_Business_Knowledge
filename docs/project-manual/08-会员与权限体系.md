# 08 — 会员与权限体系

本章全面阐述 FDSMArticles 平台的用户身份认证、会员等级管理、内容访问控制、计费订阅以及审计日志等核心机制。系统围绕 **四级会员体系** 和 **三级内容访问控制** 构建，支持 Supabase 云认证与本地 DEV_AUTH 双模式，并配备完整的计费方案管理与角色变更审计链路。

---

## 8.1 四级会员等级（Membership Tiers）

系统定义了四个互斥的会员等级，由 `membership_service.py` 中的 `TIER_LABELS` 和 `BENEFIT_MAP` 两个字典统一管理：

| 等级键值 | 中文标签 | 说明 |
|---|---|---|
| `guest` | 游客 | 未登录状态，仅可浏览公开内容 |
| `free_member` | 免费会员 | 注册用户的默认等级 |
| `paid_member` | 付费会员 | 已订阅付费方案的用户 |
| `admin` | 管理员 | 拥有完整后台权限 |

### 8.1.1 各等级权益范围

**guest（游客）**

- 可浏览公开文章与公开音视频样刊
- 可查看商业化方案和知识库结构

**free_member（免费会员）**

- 可保留点赞、收藏与阅读历史
- 可访问会员专属音频和视频栏目中的基础内容
- 可逐步沉淀个人知识资产与推荐偏好

**paid_member（付费会员）**

- 可访问付费音频、付费视频和深度会员内容
- 适合扩展为专题专报、课程、闭门简报与陪跑服务
- 可承接更高价值的商业知识服务与内容订阅

**admin（管理员）**

- 拥有会员配置权限
- 可查看完整会员列表并调整等级
- 可继续扩展到专题运营、内容发布与商业后台管理

### 8.1.2 等级合法值校验

`VALID_TIERS = {"free_member", "paid_member", "admin"}` 定义了可写入数据库的合法等级集合（不含 `guest`，因为 guest 仅为序列化默认值，不存储在 `user_memberships` 表中）。调用 `_normalize_tier()` 时，若传入值不在此集合内，将直接抛出 HTTP 400 错误。

### 8.1.3 会员状态（Membership Status）

| 状态键值 | 中文标签 | 说明 |
|---|---|---|
| `anonymous` | 未登录 | 仅用于 guest 序列化，不存储 |
| `active` | 有效 | 会员资格正常 |
| `trial` | 试用中 | 试用期内 |
| `paused` | 已暂停 | 暂停会员权益 |
| `expired` | 已过期 | 会员已到期 |

合法可写入状态集合：`VALID_STATUSES = {"active", "trial", "paused", "expired"}`。

### 8.1.4 权限能力矩阵

`_serialize_profile()` 函数根据等级自动计算能力标记：

| 能力字段 | guest | free_member | paid_member | admin |
|---|---|---|---|---|
| `is_authenticated` | `False` | `True` | `True` | `True` |
| `is_admin` | `False` | `False` | `False` | `True` |
| `can_access_member` | `False` | `True` | `True` | `True` |
| `can_access_paid` | `False` | `False` | `True` | `True` |

计算逻辑：

```python
can_access_member = tier in {"free_member", "paid_member", "admin"}
can_access_paid   = tier in {"paid_member", "admin"}
is_admin          = tier == "admin"
```

### 8.1.5 权限守卫函数

系统提供两个权限守卫函数，供路由层直接调用：

- **`require_admin_profile(user)`** -- 若 `user` 为 `None` 则抛出 401，若不是 admin 则抛出 403。
- **`require_paid_profile(user)`** -- 若 `user` 为 `None` 则抛出 401，若不是 paid_member 或 admin 则抛出 403。

## 8.2 内容访问控制 — build_content_access()

每篇文章均有一个 `access_level` 字段，取值为三级之一。`build_content_access()` 函数根据文章的访问级别和当前用户的会员状态，生成完整的访问控制描述对象。

### 8.2.1 三级访问级别

| access_level | 中文标签 | 含义 |
|---|---|---|
| `public` | 公开 | 所有访客均可阅读全文 |
| `member` | 会员 | 需要 free_member 及以上等级 |
| `paid` | 付费 | 需要 paid_member 及以上等级 |

### 8.2.2 build_content_access() 返回结构

```python
def build_content_access(access_level: str, membership: dict | None) -> dict:
```

该函数首先调用 `normalize_access_level()` 对 access_level 进行标准化（空值默认为 `"public"`），然后通过 `membership_can_access()` 判断当前用户是否有权访问。

返回字典结构：

| 字段 | 类型 | 说明 |
|---|---|---|
| `access_level` | `str` | 标准化后的访问级别 |
| `access_label` | `str` | 中文标签（"公开"/"会员"/"付费"） |
| `locked` | `bool` | `True` 表示用户无权查看完整内容 |
| `required_membership` | `str \| None` | 解锁所需的最低会员等级 |
| `required_membership_label` | `str \| None` | 所需等级的中文标签 |
| `message` | `str` | 面向用户的提示文案 |

### 8.2.3 权限判定逻辑 — membership_can_access()

```python
def membership_can_access(membership: dict | None, access_level: str) -> bool:
```

判定规则：

1. `public` -- 直接返回 `True`，任何人都可访问。
2. `member` -- 检查 `membership["can_access_member"]`，即 free_member / paid_member / admin 均可。
3. `paid` -- 检查 `membership["can_access_paid"]`，仅 paid_member / admin 可。

若 `membership` 为 `None`（未登录游客），则默认使用 `_serialize_profile(None)` 生成 guest 模板，此时 `can_access_member` 和 `can_access_paid` 均为 `False`。

### 8.2.4 各级别提示文案

| 级别 | locked 时的 message |
|---|---|
| `public` | "公开内容，所有访客均可阅读全文。" |
| `member` | "登录成为免费会员后可阅读全文与完整 AI 摘要。" |
| `paid` | "升级为付费会员后可阅读全文、完整 AI 摘要与高价值会员内容。" |

## 8.3 锁定内容预览 — preview_markdown()

当 `locked == True` 时，前端不应展示完整正文。系统通过 `preview_markdown()` 函数（位于 `article_ai_output_service.py`）生成预览片段。

### 8.3.1 函数签名

```python
def preview_markdown(
    content: str,
    paragraph_limit: int = 4,
    char_limit: int = 900,
) -> str
```

### 8.3.2 截断逻辑

1. 将内容按行分割，过滤掉空行，得到段落列表。
2. 按顺序逐段收集，受两个限制条件约束：
   - **段落数上限** -- 默认为 4 段。
   - **字符数上限** -- 默认为 900 个字符。
3. 当某段落长度超过剩余字符配额时，截断该段并追加省略号 `"..."`。
4. 收集完毕后用双换行 `"\n\n"` 拼接输出。

### 8.3.3 使用场景

`catalog_service.py` 中调用此函数对锁定内容进行预览裁切，确保付费墙前只暴露开头有限内容，既给予用户对文章质量的初步感知，又保护了完整内容的商业价值。

## 8.4 Supabase 认证

系统通过 `supabase_auth_service.py` 实现与 Supabase Auth 的集成，作为生产环境的主要认证方式。

### 8.4.1 环境变量

| 变量 | 默认值 | 说明 |
|---|---|---|
| `SUPABASE_URL` | `""` | Supabase 项目 URL |
| `SUPABASE_ANON_KEY` | `""` | Supabase anon/public key |
| `SUPABASE_AUTH_TIMEOUT_SECONDS` | `8` | HTTP 请求超时秒数 |

### 8.4.2 认证是否启用

```python
def is_supabase_auth_enabled() -> bool:
    return bool(SUPABASE_URL and SUPABASE_ANON_KEY)
```

当 `SUPABASE_URL` 和 `SUPABASE_ANON_KEY` 均非空时，Supabase 认证启用。

### 8.4.3 Bearer Token 验证流程

`get_authenticated_user()` 是认证入口函数，完整流程如下：

```
请求进入
  |
  v
is_supabase_auth_enabled()?
  |-- No --> 尝试 DEV_AUTH 模式 --> 返回 debug user 或 None
  |-- Yes
  v
从 Authorization 头提取 Bearer token
  |
  |-- token 为空 --> 尝试 DEV_AUTH 模式
  |-- token 存在
  v
GET {SUPABASE_URL}/auth/v1/user
  Headers:
    apikey: {SUPABASE_ANON_KEY}
    Authorization: Bearer {access_token}
  Timeout: SUPABASE_AUTH_TIMEOUT_SECONDS
  |
  |-- 网络异常 (RequestException) --> 返回 None
  |-- HTTP status != 200 --> 返回 None
  |-- 200 OK
  v
解析 JSON，提取 id 和 email
  |-- id 为空 --> 返回 None
  |-- id 存在
  v
返回 {"id": ..., "email": ..., "raw_user": payload}
```

### 8.4.4 Bearer Token 提取

`_extract_bearer_token()` 函数从 `Authorization` 头提取 token：

- 头部值为空 → 返回 `None`
- 不以 `"bearer "` 开头（不区分大小写） → 返回 `None`
- 提取 `"bearer "` 之后的内容，去除首尾空白

### 8.4.5 auth/status 端点

路由 `GET /api/auth/status` 调用 `get_auth_status_payload()`，返回完整认证状态，包含：

| 字段 | 说明 |
|---|---|
| `enabled` | 认证是否已配置（Supabase 或 DEV_AUTH 任一启用） |
| `authenticated` | 当前请求是否已认证 |
| `user` | `{id, email}` 或 `null` |
| `auth_mode` | `"password"`（DEV_AUTH 模式）或 `"email_otp"`（Supabase 模式） |
| `membership` | 完整会员状态对象 |
| `business_profile` | 业务用户档案对象 |
| `role_home_path` | 该角色的默认首页路径 |

### 8.4.6 角色首页路径

`ROLE_HOME_PATHS` 定义了各等级登录后的默认跳转路径：

| 等级 | 路径 |
|---|---|
| `guest` | `/` |
| `free_member` | `/me` |
| `paid_member` | `/membership` |
| `admin` | `/admin` |

## 8.5 DEV_AUTH 模式

当 `DEV_AUTH_ENABLED = True`（环境变量 `DEV_AUTH_ENABLED=1`）时，系统启用本地调试认证模式，无需连接 Supabase。

### 8.5.1 启用条件

DEV_AUTH 在以下两种场景中生效：

1. **Supabase 未配置** -- `SUPABASE_URL` 或 `SUPABASE_ANON_KEY` 为空。
2. **Supabase 已配置但请求未携带 Bearer token** -- 此时回退到 DEV_AUTH。

### 8.5.2 调试请求头

| Header | 说明 |
|---|---|
| `X-Debug-User-Id` | 模拟用户 ID，如 `mock-admin` |
| `X-Debug-User-Email` | 模拟用户邮箱 |

`_build_debug_user()` 函数的处理逻辑：

1. 若 `DEV_AUTH_ENABLED` 为 `False`，直接返回 `None`。
2. 若 `debug_user_id` 为空或仅含空白字符，返回 `None`。
3. 构造并返回 `{"id": user_id, "email": email, "raw_user": {"debug": True}}`。

### 8.5.3 密码登录端点

路由 `POST /api/auth/login` 仅在 DEV_AUTH 模式下有效，接受 `email` + `password` 登录：

```python
@router.post("/login", response_model=AuthLoginResponse)
def auth_login(payload: AuthPasswordLoginRequest):
    account = authenticate_preview_account(payload.email, payload.password)
```

`authenticate_preview_account()` 函数检查：

1. `DEV_AUTH_ENABLED` 是否为 `True`。
2. 邮箱是否在 `PREVIEW_ACCOUNT_PASSWORDS` 字典中。
3. 密码是否严格匹配（区分大小写）。
4. 邮箱对应的种子用户定义是否存在。

全部通过后返回种子用户定义字典，否则返回 `None`，路由层抛出 401 错误。

## 8.6 管理员自动提升

系统通过 `ADMIN_EMAILS` 环境变量实现管理员邮箱白名单自动提升机制。

### 8.6.1 配置方式

环境变量 `ADMIN_EMAILS` 接受逗号分隔的邮箱列表：

```bash
ADMIN_EMAILS="alice@example.com,bob@example.com"
```

加载后转为小写去重的集合：

```python
ADMIN_EMAILS = {
    item.strip().lower()
    for item in os.getenv("ADMIN_EMAILS", "").split(",")
    if item.strip()
}
```

### 8.6.2 自动提升时机

邮箱匹配检查通过 `_is_admin_email()` 函数实现（大小写不敏感），在以下两个环节触发自动提升：

1. **首次创建会员记录** -- 在 `_ensure_membership_row()` 中，若新用户的邮箱匹配 ADMIN_EMAILS，直接以 `admin` 等级创建记录，而非默认的 `free_member`。

2. **已有用户登录** -- 在 `_ensure_membership_row()` 中，若已有用户的邮箱匹配 ADMIN_EMAILS 但当前等级不是 `admin`，自动将 `tier` 更新为 `admin`。

3. **管理员手动更新会员** -- 在 `update_membership()` 中，若被操作用户的邮箱匹配 ADMIN_EMAILS，即使管理员设定了其他等级，最终也会被强制覆盖为 `admin`。

### 8.6.3 判定函数

```python
def _is_admin_email(email: str | None) -> bool:
    return bool(email and email.strip().lower() in ADMIN_EMAILS)
```

## 8.7 会员记录自动创建 — _ensure_membership_row()

每次用户通过认证后，系统调用 `get_membership_profile()` → `_ensure_membership_row()` 来保证 `user_memberships` 表中存在该用户的记录。

### 8.7.1 完整流程

```
用户通过认证，得到 user = {id, email}
  |
  v
get_membership_profile(user)
  |
  v
_ensure_membership_row(connection, user_id, email)
  |
  v
SELECT user_id FROM user_memberships WHERE user_id = ?
  |
  |-- 不存在（新用户）
  |     |
  |     v
  |   判定初始等级：
  |     邮箱匹配 ADMIN_EMAILS → tier = "admin"
  |     否则 → tier = "free_member"
  |     |
  |     v
  |   INSERT INTO user_memberships
  |   (status = 'active', started_at = now)
  |
  |-- 已存在（老用户）
  |     |
  |     v
  |   检查是否需要提升为 admin 或更新 email
  |     |
  |     v
  |   若有变化 → UPDATE user_memberships SET email, tier
  |
  v
同步到 business_users 表：
  ensure_business_user_profile()
  auth_source = "seed"（mock-* 用户）或 "supabase"
```

### 8.7.2 与业务用户档案联动

创建或更新会员记录后，函数会调用 `ensure_business_user_profile()` 将等级和状态同步到 `business_users` 表。判定 `auth_source` 的规则：

- `user_id` 以 `"mock-"` 开头 → `auth_source = "seed"`
- 其他情况 → `auth_source = "supabase"`

## 8.9 计费系统

计费系统由 `billing_service.py` 实现，管理订阅方案、结账意向（checkout intent）、订单和订阅记录。

### 8.9.1 环境变量

| 变量 | 默认值 | 说明 |
|---|---|---|
| `PAYMENTS_ENABLED` | `"0"` | 设为 `"1"` 启用支付功能 |
| `PAYMENT_PROVIDER` | `"mock"` | 支付提供商标识 |
| `SITE_BASE_URL` | `"http://127.0.0.1:4173"` | 站点基础 URL，用于生成回调地址 |

### 8.9.2 三个计费方案

系统预置 3 个计费方案，在 `billing_plans` 表中管理，并在 `PLAN_LOCALIZATIONS` 中维护中英文描述：

| plan_code | 中文名 | 等级 | 计费周期 | 英文名 |
|---|---|---|---|---|
| `free_member_monthly` | 免费会员 | `free_member` | 月付 | Free Member |
| `paid_member_monthly` | 付费会员（月付） | `paid_member` | 月付 | Paid Member Monthly |
| `paid_member_yearly` | 付费会员（年付） | `paid_member` | 年付 | Paid Member Yearly |

#### 方案特性详情

**free_member_monthly**
- Keep reading history, likes, and bookmarks
- Unlock foundational member audio and video content
- Serve as the bridge into paid upgrades

**paid_member_monthly**
- Unlock paid articles, premium audio, and premium video
- Fit topic briefings, courses, and closed-door notes
- Stay compatible with renewals, trials, and promotional pricing

**paid_member_yearly**
- Includes a full year of paid-content access
- Fits annual briefings, course bundles, and offline programs
- Prepares the path for institution and seat-based editions

### 8.9.3 计费周期标签

`BILLING_PERIOD_LABELS` 提供中英文计费周期标签：

| 键值 | 中文 | 英文 |
|---|---|---|
| `month` | 月付 | month |
| `quarter` | 季付 | quarter |
| `year` | 年付 | year |
| `oneoff` | 一次性 | one-off |

### 8.9.4 方案列表 API

```python
def list_billing_plans(language: str = "zh") -> dict
```

从 `billing_plans` 表查询所有 `is_public = 1` 的方案，按 `sort_order` 和 `price_cents` 排序。返回结构：

```json
{
  "payments_enabled": true,
  "payment_provider": "mock",
  "items": [ /* 序列化后的方案列表 */ ]
}
```

每个方案的序列化字段包括：`plan_code`、`name`、`tier`、`tier_label`、`price_cents`、`currency`、`billing_period`、`billing_period_label`、`headline`、`description`、`features`（JSON 数组）、`is_public`、`is_enabled`、`checkout_available`、`sort_order`。

其中 `checkout_available = PAYMENTS_ENABLED and is_enabled` -- 仅当全局支付启用且该方案已启用时，才允许结账。

### 8.9.5 Checkout Intent 流程

`create_checkout_intent()` 函数实现结账意向的创建流程：

```
用户请求结账
  |
  v
校验用户是否已登录（否则 401）
校验 plan_code 是否存在且 is_public（否则 404）
  |
  v
判定 payments_ready = PAYMENTS_ENABLED and plan_row["is_enabled"]
  |
  |-- payments_ready = False
  |     status = "disabled"
  |     redirect_url = None
  |     note = "Payment infrastructure is configured, but live payment
  |              capture is not enabled in this environment."
  |
  |-- payments_ready = True
  |     status = "pending"
  |     redirect_url = success_url 或
  |       "{SITE_BASE_URL}/membership?checkout=mock&plan={plan_code}"
  |     note = "A checkout intent has been created. Replace the mock
  |              redirect with your live payment gateway when ready."
  |
  v
INSERT INTO billing_checkout_intents
  (user_id, email, plan_code, status, payment_provider,
   redirect_url, note, created_at, updated_at)
  |
  v
INSERT INTO billing_orders
  (user_id, email, plan_code, amount_cents, currency,
   status, payment_provider, checkout_intent_id, ...)
  |
  v
返回 {
  "intent_id": ...,
  "order_id": ...,
  "plan_code": ...,
  "status": "pending" | "disabled",
  "payment_provider": "mock",
  "payments_enabled": true | false,
  "checkout_url": redirect_url | null,
  "message": note
}
```

### 8.9.6 Mock 支付说明

当前系统使用 `PAYMENT_PROVIDER = "mock"` 作为默认支付提供商。这意味着：

1. 不会真正触发支付网关。
2. `checkout_url` 在 `payments_ready = True` 时指向站内 mock 回调地址。
3. 订单状态始终为 `"pending"` 或 `"disabled"`，需要接入真实支付网关后替换逻辑。
4. 系统已预留 `provider_order_id` 字段用于存储第三方支付平台的订单号。

## 8.10 审计日志

系统通过 `admin_role_audit_logs` 表记录所有会员等级变更操作，确保权限变更的可追溯性。

### 8.10.1 记录函数

```python
def record_admin_role_audit(
    connection,
    *,
    target_user_id: str,
    actor_user_id: str | None,
    actor_email: str | None,
    previous_tier: str | None,
    next_tier: str,
    previous_status: str | None,
    next_status: str,
    note: str | None = None,
) -> None
```

### 8.10.2 记录时机

审计日志在 `update_membership()` 函数中、会员信息写入数据库之后立即记录。无论是新建还是更新会员记录，均会产生审计条目。

### 8.10.3 记录字段

| 字段 | 说明 |
|---|---|
| `target_user_id` | 被操作的用户 ID |
| `actor_user_id` | 执行操作的管理员 ID（可为 None） |
| `actor_email` | 执行操作的管理员邮箱（可为 None） |
| `previous_tier` | 变更前等级（新建时为 None） |
| `next_tier` | 变更后等级 |
| `previous_status` | 变更前状态（新建时为 None） |
| `next_status` | 变更后状态 |
| `note` | 操作备注 |
| `created_at` | 记录创建时间（ISO 格式） |

### 8.10.4 管理后台概览

`get_admin_overview()` 函数提供管理后台的统计概览，包括：

- 总用户数（`business_users` 表）
- 付费会员数
- 管理员数
- 审计日志总数
- 各等级会员分布统计

### 8.10.5 数据完整性

- 审计日志仅追加，不可修改或删除。
- 日志记录与会员变更在同一个事务内（`connection.commit()` 在两者之后），确保原子性。
- `actor_user_id` 和 `actor_email` 来自 `actor_user` 参数，通常为当前登录的管理员信息。
