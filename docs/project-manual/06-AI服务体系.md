## 6.1 模型配置总览

系统同时使用四个 Google Gemini 模型，各司其职：

| 用途 | 模型标识符 | 配置来源 | 说明 |
|------|-----------|---------|------|
| **对话 / 重排** | `gemini-3-flash` | 环境变量 `GEMINI_CHAT_MODEL`，缺省值 `gemini-3-flash` | 默认的主力 LLM，用于所有 LangChain 路径调用（查询扩展、文章摘要、知识库问答、搜索重排、编辑元数据建议）；当前运行时映射到 `gemini-3-flash-preview` |
| **翻译** | `gemini-2.5-flash` | 环境变量 `GEMINI_FLASH_MODEL`，缺省值 `gemini-2.5-flash` | 专用于 `translate_article_to_english`，通过 LangChain 路径调用 |
| **编辑排版** | `gemini-3-flash-preview` | 硬编码常量 `GEMINI_EDITORIAL_FORMAT_MODEL` | 专用于 `auto_format_editorial_markdown`，通过 REST 直连路径调用 |
| **向量嵌入** | `models/gemini-embedding-001` | 环境变量 `GEMINI_EMBEDDING_MODEL`，缺省值 `models/gemini-embedding-001` | 用于 FAISS 向量检索的文本嵌入生成（在 `create_vector_db_faiss.py` 及 `rag_engine` 中使用） |

模型实例通过 `_build_llm(model_name)` 工厂函数构建，参数固定为：

```python
ChatGoogleGenerativeAI(
    model=model_name,
    google_api_key=keys[0],           # 取去重后的第一个 key
    temperature=0.2,
    convert_system_message_to_human=True,
)
```

两个 LangChain 模型实例（`get_llm` 和 `get_flash_llm`）各自被 `@lru_cache(maxsize=1)` 缓存，进程生命周期内只创建一次。

## 6.2 API Key 轮转机制

### 6.2.1 密钥来源与去重

```python
@lru_cache(maxsize=1)
def get_gemini_api_keys() -> tuple[str, ...]:
    keys: list[str] = []
    for key in (PRIMARY_GEMINI_KEY, *GEMINI_API_KEYS):
        cleaned = str(key or "").strip()
        if cleaned and cleaned not in keys:
            keys.append(cleaned)
    return tuple(keys)
```

- `PRIMARY_GEMINI_KEY`：优先取 `GOOGLE_API_KEY` 环境变量，若为空则取 `GEMINI_API_KEYS` 列表的第一个元素。
- `GEMINI_API_KEYS`：从环境变量 `GEMINI_API_KEYS` 按逗号分割，最多可配 6 个 key。
- **去重逻辑**：遍历时用 `list` 保序，遇到已出现的 key 跳过，最终返回不可变 `tuple`，并被 `@lru_cache` 缓存。
- 返回结果为空元组时，`is_ai_enabled()` 返回 `False`，全部 AI 功能降级。

### 6.2.2 轮转计数器

```python
_gemini_request_counter = itertools.count()
```

这是一个进程级的无限自增计数器（`itertools.count`），每次调用 `_request_gemini_text` 时执行 `next(_gemini_request_counter)` 取得 `start_offset`。

### 6.2.3 轮转逻辑

在 `_request_gemini_text` 的重试循环中：

```python
max_attempts = max(3, len(api_keys) * 2)

for attempt in range(max_attempts):
    api_key = api_keys[(start_offset + attempt) % len(api_keys)]
```

- **最大重试次数**：`max(3, key数量 * 2)`。若有 6 个 key，则最多 12 次尝试。
- **Key 选择**：`(start_offset + attempt) % len(api_keys)`。`start_offset` 逐次递增，保证不同请求从不同 key 开始，均匀分摊配额。

### 6.2.4 指数退避

```python
time.sleep(min(4.0, 0.6 * (attempt + 1)))
```

| attempt | 等待时间 |
|---------|---------|
| 0 | 0.6s |
| 1 | 1.2s |
| 2 | 1.8s |
| 3 | 2.4s |
| 4 | 3.0s |
| 5 | 3.6s |
| 6+ | 4.0s（上限） |

退避仅在以下两种情况触发：
1. HTTP 状态码属于可重试集合 `{429, 500, 502, 503, 504}`。
2. 发生 `requests.RequestException`（网络超时、连接中断等）。

对于其他 HTTP 错误（如 400、401、403），直接抛出 `RuntimeError`，不进入重试。

## 6.3 两条调用路径

### 6.3.1 LangChain 调用路径 --- `_invoke_prompt`

```
PromptTemplate.from_template(template)
        |
        v
ChatGoogleGenerativeAI (gemini-3-flash / gemini-2.5-flash；其中 gemini-3-flash 当前运行时映射到 gemini-3-flash-preview)
        |
        v
StrOutputParser()
        |
        v
chain.invoke(payload).strip()
```

**核心签名：**

```python
def _invoke_prompt(
    template: str,
    payload: dict[str, str],
    *,
    llm: ChatGoogleGenerativeAI | None = None,
) -> str
```

- `template`：包含 `{variable}` 占位符的 Prompt 字符串。
- `payload`：键值对，由 LangChain 的 `PromptTemplate` 填充。
- `llm`：可选，默认使用 `get_llm()`（配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）。翻译功能传入 `get_flash_llm()`。
- 内部通过 `|` 管道运算符将三个组件串成 `chain`，调用 `chain.invoke(payload)` 完成端到端推理。

**使用此路径的功能：** `expand_query`、`summarize_article`、`answer_with_sources`、`rerank_search_results`、`suggest_editorial_metadata`、`translate_article_to_english`。

### 6.3.2 REST 直连路径 --- `_request_gemini_text`

```
POST https://generativelanguage.googleapis.com/v1beta/models/{model}:generateContent?key={api_key}
```

**核心签名：**

```python
def _request_gemini_text(
    *,
    prompt: str,
    model_name: str,
    response_mime_type: str = "text/plain",
) -> str
```

**请求体结构：**

```json
{
  "contents": [{"parts": [{"text": "<prompt>"}]}],
  "generationConfig": {
    "temperature": 0.25,
    "responseMimeType": "text/plain"
  }
}
```

**超时配置：** `(10, 90)` --- 连接超时 10 秒，读取超时 90 秒。使用 `requests.Session` 保持连接复用。

**响应解析：** `_extract_gemini_text` 从 `response.json()["candidates"][0]["content"]["parts"]` 中提取并拼接所有 `text` 字段。

**使用此路径的功能：** `auto_format_editorial_markdown`。选择 REST 直连而非 LangChain 的原因是该功能需要指定 `responseMimeType` 并使用尚未被 LangChain 适配器完全支持的 `gemini-3-flash-preview` 模型。

## 6.4 JSON 解析工具链

`_parse_json_payload(raw)` 负责将 LLM 返回的非结构化文本解析为 Python 对象：

1. **代码围栏剥离**：`_strip_code_fence` 移除 `` ```json ... ``` `` 包裹。
2. **直接解析**：先尝试 `json.loads(text)`。
3. **数组兜底**：查找最外层 `[...]` 并解析。
4. **对象兜底**：查找最外层 `{...}` 并解析。
5. 若全部失败，抛出 `json.JSONDecodeError`。

该工具链被 `expand_query`、`rerank_search_results`、`suggest_editorial_metadata`、`translate_article_to_english` 共同使用。

## 6.5 七大核心功能详解

### 6.5.1 `expand_query` --- 查询扩展

**目的：** 将用户的单条搜索查询改写为 2--4 个语义近似的替代查询，提升检索召回率。

**调用路径：** LangChain（`_invoke_prompt` -> 配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）

**完整 Prompt 模板：**

```
You rewrite search queries for a Chinese business knowledge base.
Return 2 to 4 short alternative queries in the same language as the original query.
Do not broaden the meaning, do not explain, and return JSON only.

User query:
{query}

JSON:
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `query` | `str` | 用户原始搜索词 |

**返回值：** `list[str]`，最多 4 条去重后的替代查询。

**后处理逻辑：**
1. 调用 `_parse_json_payload` 解析 JSON 数组。
2. 过滤非字符串元素。
3. 去除与原始 `query` 完全相同的条目。
4. 用 `set` 去重，保留插入顺序。
5. 截断到最多 4 条。

**降级：** AI 不可用或解析异常时返回空列表 `[]`。

### 6.5.2 `summarize_article` --- 文章摘要

**目的：** 将一篇文章压缩为 6--10 个高密度要点的 Markdown 简报。

**调用路径：** LangChain（`_invoke_prompt` -> 配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）

**完整 Prompt 模板：**

```
You are an editor for a Chinese business knowledge product.
Summarize the article into a high-density Markdown brief.
Keep the original logic, do not invent facts, avoid filler, and answer in the same language as the article.
Target 6 to 10 short sections or bullets.

Title:
{title}

Article:
{content}

Summary:
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `title` | `str` | 文章标题 |
| `content` | `str` | 文章正文，截断到前 18,000 字符 |

**返回值：** `str`，Markdown 格式摘要。

**降级：** AI 不可用或调用异常时，退回 `build_extractive_summary`，该函数简单提取正文前 6 个非空段落拼接。

### 6.5.3 `answer_with_sources` --- 带引用的知识库问答

**目的：** 根据检索到的知识库材料，用 `[1]`、`[2]` 等编号引用回答用户问题，是 RAG 问答的核心生成环节。

**调用路径：** LangChain（`_invoke_prompt` -> 配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）

**完整 Prompt 模板：**

```
You are a business knowledge assistant.
Answer only from the supplied materials and never invent facts.
If the evidence is insufficient, say that clearly.
Respond in Markdown and cite sources with [1], [2], etc.
{language_instruction}

Conversation history:
{history}

User question:
{question}

Materials:
{context_blocks}

Answer:
```

其中 `{language_instruction}` 根据 `response_language` 参数动态切换：

| `response_language` | 指令文本 |
|---------------------|---------|
| `"zh"` | `Respond in concise, professional Simplified Chinese.` |
| `"en"` | `Respond in concise, polished English.` |
| `"auto"`（默认） | `Use the same language as the user question.` |

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `question` | `str` | 用户当前问题 |
| `history` | `str` | 对话历史文本，为空时填 `"None"` |
| `context_blocks` | `str` | 由 `_build_context_blocks` 拼装的检索材料 |
| `response_language` | `str` | 响应语言，`"auto"` / `"zh"` / `"en"` |

**返回值：** `str`，带有编号引用的 Markdown 回答。

**降级：** 该函数本身不做降级；由调用方 `_answer_from_sources` 在异常时退回 `_fallback_answer`（纯阅读路径建议列表）。

### 6.5.4 `rerank_search_results` --- 搜索结果重排

**目的：** 对检索引擎返回的候选文章列表，让 LLM 按与查询的相关性打 0--10 分，用于 smart 模式的结果重排。

**调用路径：** LangChain（`_invoke_prompt` -> 配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）

**完整 Prompt 模板：**

```
You rerank search results for a business knowledge base.
Score each candidate from 0 to 10 based only on relevance to the user query.
Return JSON only in this format: [{"id": 1, "score": 8.6}].

User query:
{query}

Candidates:
{candidates}

JSON:
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `query` | `str` | 用户查询 |
| `candidates` | `list[dict]` | 候选文章列表，序列化为 JSON 字符串传入 Prompt |

**返回值：** `dict[int, float]`，键为候选 ID，值为 0.0--10.0 的相关性分数（已做 clamp）。

**后处理逻辑：**
1. `_parse_json_payload` 解析 JSON 数组。
2. 遍历每个 `{"id": int, "score": float}` 对象。
3. 用 `max(0.0, min(float(score), 10.0))` 钳制到合法范围。

**降级：** AI 不可用、候选列表为空或解析异常时返回空字典 `{}`。

### 6.5.5 `suggest_editorial_metadata` --- 编辑元数据建议

**目的：** 自动为文章生成摘要（excerpt）、文章类型、主题标签、栏目归属等编辑元数据，减轻人工录入负担。

**调用路径：** LangChain（`_invoke_prompt` -> 配置名 `gemini-3-flash`，当前运行时映射到 `gemini-3-flash-preview`）

**完整 Prompt 模板：**

```
You are helping an editorial CMS for a Chinese business knowledge site.
Read the article title and content, then return strict JSON with this shape:
{
  "excerpt": "80-140 Chinese characters summary",
  "article_type": "short type label",
  "main_topic": "primary topic label",
  "column_slug": "insights|industry|research|deans-view",
  "tags": [{"name": "tag name", "category": "topic|industry|type|entity", "confidence": 0.0}]
}
Use concise Chinese labels when possible, do not invent facts, return 4 to 8 tags, and return JSON only.

Title:
{title}

Content:
{content}

JSON:
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `title` | `str` | 文章标题 |
| `content` | `str` | 文章正文，截断到前 12,000 字符 |

**返回值：** `dict`，包含 `excerpt`、`article_type`、`main_topic`、`column_slug`、`tags` 字段。

**返回示例：**

```json
{
  "excerpt": "复旦大学管理学院发布最新研究报告，探讨生成式AI在金融行业的落地路径...",
  "article_type": "深度分析",
  "main_topic": "AI与金融科技",
  "column_slug": "research",
  "tags": [
    {"name": "生成式AI", "category": "topic", "confidence": 0.95},
    {"name": "金融科技", "category": "industry", "confidence": 0.88}
  ]
}
```

**降级：** AI 不可用或解析异常时返回空字典 `{}`。

### 6.5.6 `auto_format_editorial_markdown` --- 自动编辑排版

**目的：** 将原始 Markdown 稿件重排为微信公众号级别的出版就绪格式，支持 4 种布局模式。

**调用路径：** REST 直连（`_request_gemini_text` -> `gemini-3-flash-preview`）

**4 种布局模式（`layout_mode`）：**

| 模式 | 布局引导词 |
|------|-----------|
| `auto`（默认） | Default to a polished WeChat-style long-form layout with a short lead, clear H2/H3 hierarchy, and dense but readable information blocks. |
| `insight` | Favor deeper analysis, explicit argument structure, section transitions, and slightly longer analytical paragraphs. |
| `briefing` | Favor rapid scanning with short sections, bullet lists, key takeaways, and compact summaries. |
| `interview` | Preserve speaker turns and Q&A logic while adding a stronger title, lead, and readable section breaks. |

**完整 Prompt 模板：**

```
You are the formatting engine for a top-tier Chinese WeChat official-account editorial CMS.
Rewrite the source into publication-ready Markdown only.
Hard requirements:
1. Use model-side judgment to create a readable, professional Chinese long-form layout similar to the automatic layout mode used by high-end WeChat publishing tools.
2. Keep every important fact, number, company name, quote, and conclusion. Do not invent facts.
3. The first heading must be a single H1 title.
4. Add a short lead paragraph after the title when useful.
5. Use clean H2/H3 hierarchy, short paragraphs, lists, and quotes where appropriate.
6. Preserve interview, dialogue, and speech structure when present.
7. Do not output HTML. Do not output code fences. Output Markdown only.
8. Do not mention the model, prompt, or formatting rules in the answer.

Layout mode guidance: {layout_guidance}
Additional editorial notes: {formatting_notes}

Metadata:
- Title: {normalized_title}
- Excerpt hint: {excerpt}
- Main topic: {main_topic}
- Article type: {article_type}
- Organization: {organization}
- Tags: {tags_text}

Source content:
{normalized_source}
```

**参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | `str` | 是 | 文章标题，为空时默认 `"未命名文稿"` |
| `source_markdown` | `str` | 是 | 原始 Markdown 内容，不可为空 |
| `excerpt` | `str` | 是 | 文章摘要提示 |
| `main_topic` | `str` | 是 | 主题标签 |
| `article_type` | `str` | 是 | 文章类型 |
| `organization` | `str` | 是 | 来源机构 |
| `tags` | `list[str]` | 否 | 标签列表 |
| `layout_mode` | `str` | 否 | 布局模式，默认 `"auto"` |
| `formatting_notes` | `str` | 否 | 额外编辑备注 |

**返回值：** `dict[str, str]`，包含 `markdown`（排版后的 Markdown）和 `model`（使用的模型名称）。

**后处理：**
1. `_strip_code_fence` 去除可能的代码围栏包裹。
2. 若输出不以 `#` 开头，自动补上 `# {title}` 作为一级标题。

**降级：** 无静默降级；源内容为空或 API 返回为空时直接抛出 `RuntimeError`。

### 6.5.7 `translate_article_to_english` --- 全文英译

**目的：** 将中文商业文章整篇翻译为专业英文，同时保留原文结构。

**调用路径：** LangChain（`_invoke_prompt` -> `gemini-2.5-flash`），通过 `llm=flash_llm` 参数显式指定 Flash 模型。

**完整 Prompt 模板：**

```
You are translating a Chinese business article for an English-language knowledge product.
Translate the title, short deck, summary, and full visible content into polished professional English.
Preserve structure, headings, lists, quotations, and paragraph breaks.
Do not invent facts and do not omit visible source content.
Return strict JSON only with this shape:
{
  "title": "English title",
  "excerpt": "English deck or short intro",
  "summary": "Markdown summary in English",
  "content": "Full translated content in Markdown"
}

Source title:
{title}

Source excerpt:
{excerpt}

Source content:
{content}

JSON:
```

**参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| `title` | `str` | 中文标题 |
| `excerpt` | `str` | 中文摘要 |
| `content` | `str` | 中文正文 |

**返回值：** `dict[str, str]`，包含 `title`、`excerpt`、`summary`、`content`、`model` 五个字段。

**后处理逻辑：**
1. `_parse_json_payload` 解析 JSON 对象。
2. 各字段做空值兜底：标题、摘要、内容若为空则回退到原文。
3. `summary` 若为空，退回到 `excerpt` 或正文第一行。
4. 附加 `model` 字段记录使用的模型名。

**降级：** Flash 模型未配置时抛出 `RuntimeError`。

## 6.6 `is_ai_enabled()` 降级策略

```python
def is_ai_enabled() -> bool:
    return bool(get_gemini_api_keys())
```

这是整个 AI 服务体系的**总开关**。当没有配置任何有效的 API Key 时，返回 `False`。各功能对降级的处理方式汇总如下：

| 功能 | 降级行为 | 用户感知 |
|------|---------|---------|
| `expand_query` | 返回 `[]` | 搜索不做查询扩展，仅用原始关键词检索 |
| `summarize_article` | 返回 `build_extractive_summary` 的结果（前 6 段原文） | 摘要质量下降但仍可阅读 |
| `answer_with_sources` | 不做内部降级，由 `_answer_from_sources` 捕获异常后退回 `_fallback_answer` | 返回阅读路径建议列表替代生成式回答 |
| `rerank_search_results` | 返回 `{}` | 搜索结果不做 LLM 重排，仅依赖向量/BM25 分数 |
| `suggest_editorial_metadata` | 返回 `{}` | 编辑界面不显示 AI 建议，需人工填写 |
| `auto_format_editorial_markdown` | 抛出 `RuntimeError` | 前端提示排版服务不可用 |
| `translate_article_to_english` | 抛出 `RuntimeError` | 前端提示翻译服务不可用 |

**设计原则：** 查询侧功能（扩展、摘要、重排）做静默降级以保持搜索可用；生成侧功能（排版、翻译）做显式报错以避免返回低质量内容。

## 6.7 辅助函数清单

| 函数名 | 职责 |
|--------|------|
| `_strip_code_fence(text)` | 移除 LLM 返回内容外层的 Markdown 代码围栏 |
| `_parse_json_payload(raw)` | 多策略 JSON 提取（直接解析 -> 最外层数组 -> 最外层对象） |
| `_extract_gemini_text(response_payload)` | 从 Gemini REST 响应体中提取拼接文本 |
| `build_extractive_summary(content)` | 无 AI 时的文章摘要降级方案，提取前 6 个非空段落 |
