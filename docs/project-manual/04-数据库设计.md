# 04 — 数据库设计

本文档详细说明复旦商业知识库平台的数据库架构，包括技术选型、所有表结构定义、表间关系、迁移机制及种子数据初始化策略。

对应代码文件：`backend/database.py`、`backend/config.py`、`backend/scripts/build_business_db.py`

---

## 4.1 技术选型

### 4.1.1 为什么选择 SQLite

| 考量维度 | SQLite 的优势 |
|----------|---------------|
| 部署简单 | 无须独立数据库进程，数据库即单个 `.db` 文件，随应用一起部署即可 |
| 读多写少 | 商业知识库以文章检索、阅读为主，写入操作（浏览计数、点赞、收藏等）频次远低于读取，SQLite 的单写者模型完全胜任 |
| 文件级备份 | 可直接拷贝 `.db` 文件实现冷备份，也可通过 WAL 日志实现热备，运维成本极低 |
| 零配置 | 不需要管理用户权限、网络端口、连接池等基础设施，适合中小型内容平台 |

### 4.1.2 连接参数

连接逻辑集中在 `backend/database.py` 的 `get_connection()` 函数中：

```python
def get_connection() -> sqlite3.Connection:
    connection = sqlite3.connect(SQLITE_DB_PATH, timeout=60)
    connection.row_factory = sqlite3.Row
    connection.execute("PRAGMA busy_timeout = 60000")
    return connection
```

| 参数 | 值 | 说明 |
|------|----|------|
| `timeout` | `60`（秒） | `sqlite3.connect()` 的 Python 层超时，当数据库被锁时最多等待 60 秒再抛出 `OperationalError` |
| `row_factory` | `sqlite3.Row` | 查询结果以类字典对象返回，支持 `row["column_name"]` 方式访问字段，避免依赖列索引 |
| `PRAGMA busy_timeout` | `60000`（毫秒） | SQLite 引擎层的忙等待超时，当 WAL 写锁被占用时，最多等待 60 秒后再报 `SQLITE_BUSY`。与 Python `timeout` 参数配合，提供双层超时保护 |

此外，数据库构建阶段通过 `PRAGMA journal_mode=WAL` 启用 WAL（Write-Ahead Logging）模式，允许读写并发，进一步减少锁竞争。

### 4.1.3 连接管理

使用 `connection_scope()` 上下文管理器统一管理连接的创建与关闭：

```python
@contextmanager
def connection_scope():
    connection = get_connection()
    try:
        yield connection
    finally:
        connection.close()
```

所有业务代码均通过 `with connection_scope() as connection:` 获取连接，确保连接在异常情况下也能正确关闭，避免连接泄漏。

## 4.2 核心表（Core Tables）

核心表在数据库构建阶段（`backend/scripts/build_business_db.py` 的 `create_schema()` 函数）创建，构建时采用 `DROP TABLE IF EXISTS` + `CREATE TABLE` 的方式全量重建。这些表构成平台数据的骨架。

代码中使用 `REQUIRED_TABLES` 集合来校验数据库是否就绪：

```python
REQUIRED_TABLES = {
    "articles", "tags", "article_tags", "columns", "article_columns",
    "topics", "topic_articles", "topic_tags", "featured_articles",
    "chat_sessions", "chat_messages",
}
```

### 4.2.1 meta — 元数据表

存储构建时的全局键值对信息。

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `key` | TEXT | PRIMARY KEY | 元数据键名 |
| `value` | TEXT | NOT NULL | 元数据值 |

当前存储的键值包括 `business_article_count`（文章总数）和 `updated_at`（最近构建时间）。

| `source` | TEXT | NOT NULL | 数据来源，固定为 `"business"` |
| `source_mode` | TEXT | | 来源模式（如 `business`、`wechat` 等） |
| `title` | TEXT | NOT NULL | 文章标题 |
| `publish_date` | TEXT | NOT NULL | 发布日期（ISO 8601 格式，如 `2024-03-15`） |
| `link` | TEXT | | 原始来源链接 |
| `content` | TEXT | NOT NULL | 文章正文全文 |
| `excerpt` | TEXT | | 摘要，从正文前 180 字符截取 |
| `main_topic` | TEXT | | 主题分类（由 Gemini 模型分析得出） |
| `article_type` | TEXT | | 文章类型：专访、对谈、案例报道、研究/论文解读、活动/预告、书摘/书评、评论/观点 |
| `series_or_column` | TEXT | | 系列/专栏归属 |
| `primary_org_name` | TEXT | | 主要关联机构名称 |
| `tag_text` | TEXT | DEFAULT '' | 标签文本（以 ` \| ` 分隔），用于搜索 |
| `people_text` | TEXT | DEFAULT '' | 关联人物文本（以 ` \| ` 分隔），用于搜索 |

| `access_level` | TEXT | NOT NULL DEFAULT 'public' | 访问级别：`public`（公开）、`member`（会员）、`paid`（付费）— 通过迁移添加 |
| `created_at` | TEXT | | 记录创建时间（ISO 8601） |
| `updated_at` | TEXT | | 记录更新时间（ISO 8601） |

**索引**：
- `idx_articles_date` — `publish_date DESC`
- `idx_articles_view_count` — `view_count DESC`

### 4.4.1 核心内容关系

```
articles (1) ──┬── (*) article_tags (*) ──── (1) tags
               ├── (*) article_columns (*) ── (1) columns
               ├── (*) topic_articles (*) ─── (1) topics

               └── (*) topic_tags (*) ────── (1) tags

- **columns** 通过 `article_columns` 与文章实现多对多关联。
- **topics** 通过 `topic_articles` 关联文章，通过 `topic_tags` 关联标签。

```
business_users (1) ──┬── user_memberships  (通过 user_id 逻辑关联)

- `business_users` 是用户中心表，与 `user_memberships` 通过 `user_id` 逻辑关联（非显式外键）。

articles (1) ──── (*) editorial_articles (通过 article_id / source_article_id)

## 4.5 迁移机制

### 4.5.1 设计原则

本项目不使用独立的迁移框架（如 Alembic），而是在 `ensure_runtime_tables()` 函数中实现幂等的增量迁移。核心思路：

1. **表级别**：使用 `CREATE TABLE IF NOT EXISTS` 确保表存在，不会重复创建。
2. **列级别**：使用 `_table_has_column()` 检查列是否已存在，仅在缺失时执行 `ALTER TABLE ADD COLUMN`。
3. **数据修补**：使用 `UPDATE ... WHERE ... IS NULL OR ... = ''` 对空值字段进行默认值回填。

### 4.5.2 幂等实现

`_table_has_column()` 是迁移的核心辅助函数：

```python
def _table_has_column(connection, table_name, column_name) -> bool:
    rows = connection.execute(f"PRAGMA table_info({table_name})").fetchall()
    return any(row["name"] == column_name for row in rows)
```

每次迁移都遵循以下模式：

```python
if not _table_has_column(connection, "表名", "新列名"):
    connection.execute("ALTER TABLE 表名 ADD COLUMN 新列名 类型 约束")
```

这保证了：
- **首次运行**：列不存在，执行 `ALTER TABLE`，添加新列。
- **重复运行**：列已存在，跳过 `ALTER TABLE`，无副作用。
- **安全性**：SQLite 的 `ALTER TABLE ADD COLUMN` 只支持在表末尾追加列，不支持删除或修改已有列，这一限制在此机制下反而成为优势——不会因为迁移冲突破坏已有数据。

### 4.5.3 当前迁移清单

以下列是通过迁移机制添加的（按表分组）：

**articles 表**：
- `access_level` — TEXT NOT NULL DEFAULT 'public'

**editorial_articles 表**：
- `access_level`, `source_article_id`, `workflow_status`, `review_note`, `scheduled_publish_at`, `submitted_at`, `approved_at`, `ai_synced_at`, `source_markdown`, `layout_mode`, `formatting_notes`, `formatter_model`, `last_formatted_at`

**media_items 表**：
- `series_name`, `episode_number`, `media_url`, `preview_url`, `transcript_markdown`, `chapter_payload_json`

**business_users 表**：
- `description`

**article_ai_outputs 表**：
- `translation_title_en`, `translation_excerpt_en`, `translation_summary_en`, `summary_html_zh`, `summary_html_en`, `translation_content_en`, `summary_status`, `format_status`, `translation_status`, `summary_error`, `format_error`, `translation_error`

**chat_messages 表**：
- `follow_ups_json`

### 4.5.4 数据修补逻辑

迁移完成后，`ensure_runtime_tables()` 还会执行一系列数据修补 `UPDATE` 语句：

1. 将 `articles.access_level` 和 `editorial_articles.access_level` 的空值回填为 `'public'`。
2. 将 `editorial_articles.source_markdown` 的空值回填为 `content_markdown` 的内容（`COALESCE(NULLIF(source_markdown, ''), content_markdown)`）。
3. 将 `editorial_articles.layout_mode` 的空值回填为 `'auto'`。
4. 将 `editorial_articles.workflow_status` 与 `status` 同步：已发布的文章设为 `'published'`，其余默认 `'draft'`。
5. 将 `media_items` 的 `series_name`（按 kind 映射为"管理者音频"或"管理者视频"）、`episode_number`（至少为 1）、`preview_url`（回退为 `source_url`）、`transcript_markdown`（回填空串）、`chapter_payload_json`（回填 `'[]'`）回填默认值。

## 4.7 就绪检查

`database_is_ready()` 函数在应用启动时被调用，验证数据库文件存在且包含所有核心表：

```python
def database_is_ready() -> bool:
    if not SQLITE_DB_PATH.exists():
        return False
    with connection_scope() as connection:
        rows = connection.execute(
            "SELECT name FROM sqlite_master WHERE type IN ('table', 'view')"
        ).fetchall()
    existing = {row["name"] for row in rows}
    return REQUIRED_TABLES.issubset(existing)
```

若检查不通过，`ensure_database_ready()` 会触发全量重建（`rebuild_database()`），该过程包括：

1. 解析 `business_data/` 目录下的原始文本文件
2. 调用 Gemini AI 分析结果构建文档对象（标题、主题、标签、关键人物等）
3. 创建表结构（`create_schema()`）
4. 插入所有数据（`insert_documents()`）
5. 重建专题（`rebuild_topics()`）
6. 刷新搜索缓存（`refresh_search_cache()`）
