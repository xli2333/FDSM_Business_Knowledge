# 10 — 前端架构

> 本文档覆盖"复旦管院智识库"前端（React 18 + Vite）的完整架构实现，
> 包括应用启动链路、Provider 层级、路由设计、认证体系、API 客户端、
> 访客追踪、角色体验系统、国际化、布局组件和共享组件。
>
> 源码入口：`frontend/src/main.jsx`

## 10.1 Provider 栈

应用采用严格的 Provider 嵌套顺序，每一层向下层提供特定的 Context：

```
StrictMode
  └── BrowserRouter            ← react-router-dom，提供路由能力
        └── LanguageProvider   ← 国际化上下文（语言、翻译函数 t()）
              └── AuthProvider ← 认证上下文（用户、会话、角色、权限标志）
                    └── App   ← 路由表 + Suspense 懒加载
```

### 10.1.1 各层职责

| 层级 | 组件 | 来源文件 | 职责 |
|------|------|----------|------|
| 1 | `StrictMode` | React 内置 | 开发模式下双渲染检测副作用 |
| 2 | `BrowserRouter` | `react-router-dom` | 提供 `useNavigate`、`useLocation`、`Routes` 等路由原语 |
| 3 | `LanguageProvider` | `src/i18n/LanguageProvider.jsx` | 管理当前语言状态（`zh` / `en`），暴露 `t(key)` 翻译函数、`isEnglish` 标志、`setLanguage` / `toggleLanguage` 切换方法 |
| 4 | `AuthProvider` | `src/auth/AuthProvider.jsx` | 管理认证状态（登录/登出）、会员信息（membership）、商务档案（businessProfile）、角色首页路径（roleHomePath）、权限标志集（isAdmin / isPaidMember / canUseAiAssistant 等） |
| 5 | `App` | `src/App.jsx` | 使用 `Suspense` + `lazy()` 实现路由级代码分割，定义全部路由及守卫规则 |

### 10.1.2 嵌套顺序的设计约束

- **LanguageProvider 在 AuthProvider 之上**：AuthProvider 内部需要调用 `useLanguage()` 获取 `isEnglish` 以生成本地化错误提示（如"邮箱或密码不正确"vs."Incorrect email or password."），因此语言层必须先于认证层。
- **BrowserRouter 在最外层**：LanguageProvider 在初始化时需要读取 `window.location.search` 中的 `?lang=en` 参数，因此 BrowserRouter 必须先于 LanguageProvider 挂载。
- **AuthProvider 在 App 之上**：App 中的 `ProtectedRoute` 需要消费认证上下文中的权限标志来做路由守卫。

## 10.2 路由设计

所有路由定义在 `App.jsx` 中，使用 `react-router-dom` 的 `<Routes>` + `<Route>` 声明式路由。

### 10.2.1 懒加载策略

所有页面组件均通过 `React.lazy()` + 动态 `import()` 加载，在路由匹配时才会触发对应 chunk 的下载。
全局 `<Suspense>` 的 fallback 使用 `<PageFallback />`，显示一行翻译后的"加载中"提示文字。

```jsx
const HomePage = lazy(() => import('./pages/HomePage.jsx'))
// ... 其他 22 个页面同理
```

- **动态段**（`:id`、`:slug`）：通过 `useParams()` 在页面组件内获取。
- **查询参数**：SearchPage 使用 `useSearchParams()` 解析 `q`、`mode`、`sort`、`start_date`、`end_date`、`tags`、`columns` 等参数。LoginPage 读取 `redirect` 参数以在登录后跳转回原页面。
- **Props 传参**：MediaHubPage 通过 JSX prop `kind="audio"` / `kind="video"` 区分音频和视频两种模式，复用同一个页面组件。

认证系统由四个文件构成：

### 10.3.1 AuthProvider 状态管理

AuthProvider 维护以下核心状态：

| 状态字段 | 类型 | 初始值 | 说明 |

### 10.3.4 派生权限标志

AuthProvider 向下游暴露以下派生标志（通过 `useAuth()` 获取）：

| 标志 | 计算逻辑 | 用途 |
|------|----------|------|
| `isAuthenticated` | `Boolean(user)` | 是否已登录 |
| `isGuest` | `!user` | 是否为访客 |
| `isMember` | `Boolean(membership?.can_access_member)` | 是否具有会员访问权限 |
| `isPaidMember` | `Boolean(membership?.can_access_paid)` | 是否为付费会员 |
| `isAdmin` | `Boolean(membership?.is_admin)` | 是否为管理员 |
| `canUseAiAssistant` | `Boolean(user) && (can_access_paid \|\| is_admin)` | 是否可使用 AI 助理 |
| `authEnabled` | `backendAuthEnabled \|\| loading` | 认证系统是否可用（loading 期间假定可用） |

### 10.3.5 signInWithPassword 登录流程

```
用户提交 email + password
  ↓
调用 loginWithPassword({ email, password })  →  POST /api/auth/login
  ↓
成功响应 payload 包含:
  - payload.user (id, email)
  - payload.membership (tier, benefits, etc.)
  - payload.business_profile (display_name, role_home_path, etc.)
  - payload.role_home_path
  ↓
saveDebugAuth(account)          ← 将账号写入 localStorage
setUser(buildLocalUser(account))
setSession(buildLocalSession(account))
setMembership(payload.membership)
setBusinessProfile(payload.business_profile)
setRoleHomePath(payload.role_home_path)
  ↓
syncStatus()                    ← 再次调用 GET /api/auth/status 同步最新状态
setDialogOpen(false)            ← 关闭登录对话框
```

**错误处理**：
- HTTP 401 → 显示"邮箱或密码不正确"
- 其他错误 → 显示"登录暂不可用"

### 10.3.6 signOut 登出流程

```
clearDebugAuth()        ← 清除 localStorage 中的调试认证数据
setSession(null)
setUser(null)
setMembership(GUEST_MEMBERSHIP)
setBusinessProfile(GUEST_BUSINESS_PROFILE)
setRoleHomePath('/')
setAuthMessage('')
setAuthError('')
```

### 10.3.7 初始化流程

应用启动时（`useEffect([], ...)`）：

1. 从 localStorage 加载 `loadDebugAuth()`，若存在则恢复用户和会话状态
2. 调用 `syncStatus()`（`GET /api/auth/status`）获取后端认证和会员状态
3. 无论成功或失败，将 `loading` 设为 `false`

### 10.3.8 AuthDialog 弹窗登录

AuthProvider 内渲染了 `<AuthDialog>` 组件，任何页面组件可通过 `useAuth().openAuthDialog()` 打开弹窗式登录，无需跳转到 `/login` 页面。

## 10.4 ProtectedRoute 路由守卫

`ProtectedRoute` 支持三种独立的守卫模式，可组合使用：

### 10.4.1 守卫判定流程

```
loading?
  → 显示 "Loading access..."
  ↓
isAuthenticated?
  → false: 重定向到 /login?redirect=<当前路径>
  ↓
requireAdmin && !isAdmin?
  → 重定向到 fallbackPath（默认 "/"）
  ↓
requirePaid && !isPaidMember?
  → 重定向到 fallbackPath（默认 "/"）
  ↓
requireAiAssistant && !canUseAiAssistant?
  → 重定向到 fallbackPath（默认 "/"）
  ↓
渲染 children
```

### 10.4.2 三种守卫 Props

| Prop | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `requireAdmin` | `boolean` | `false` | 要求管理员权限 |
| `requirePaid` | `boolean` | `false` | 要求付费会员权限 |
| `requireAiAssistant` | `boolean` | `false` | 要求 AI 助理权限（付费会员或管理员） |
| `fallbackPath` | `string` | `'/'` | 权限不足时的回退路径 |

### 10.4.3 实际使用统计

| 守卫类型 | 使用路由 | fallbackPath |
|----------|----------|--------------|
| `requireAdmin` | `/analytics`、`/admin`、`/admin/memberships`、`/commercial/leads`、`/editorial`、`/media-studio` | `/`、`/admin` |
| `requireAiAssistant` | `/chat` | `/membership` |
| `requirePaid` | （当前未使用，作为扩展预留） | — |

### 10.4.4 重定向机制

未登录用户被守卫拦截时，会被重定向到 `/login?redirect=<原路径>`。
`buildLoginPath()` 函数负责构建含 redirect 参数的登录路径，保留完整的 `pathname` + `search`。

## 10.5 API 客户端

API 客户端定义在 `src/api/index.js`，为整个前端提供统一的后端通信层。

### 10.5.1 基础配置

```js
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL || 'http://127.0.0.1:8000/api'
const API_ORIGIN = API_BASE_URL.replace(/\/api\/?$/, '')
```

- `API_BASE_URL`：所有 API 请求的基础路径，通过 Vite 环境变量 `VITE_API_BASE_URL` 配置，默认指向本地开发服务器。
- `API_ORIGIN`：去掉 `/api` 后缀的服务器源地址，用于 `resolveApiUrl()` 拼接相对路径的资源 URL（如图片、音频文件）。

### 10.5.2 核心 request() 函数

```js
async function request(path, options = {}) { ... }
```

**自动注入的 Headers**：

| Header | 条件 | 来源 | 说明 |
|--------|------|------|------|
| `Content-Type: application/json` | 请求体不是 FormData 时 | 自动 | JSON 请求内容类型 |
| `X-Visitor-Id` | 浏览器环境下 | `getOrCreateVisitorId()` | 访客追踪 UUID |
| `Authorization: Bearer <token>` | options.authToken 有值时 | 调用方传入 | JWT 认证令牌 |
| `X-Debug-User-Id` | 无 authToken 且有调试认证数据时 | `getDebugAuthHeaders()` | 调试模式用户 ID |
| `X-Debug-User-Email` | 同上且有 email 时 | `getDebugAuthHeaders()` | 调试模式用户邮箱 |

**FormData 特殊处理**：当 `options.body` 为 `FormData` 实例时，不设置 `Content-Type` header，让浏览器自动添加带 boundary 的 `multipart/form-data`。

**错误处理**：非 2xx 响应时抛出 `Error`，并在 error 对象上附加 `status` 属性，便于上层根据 HTTP 状态码做差异化处理。

### 10.5.3 resolveApiUrl() 辅助函数

```js
export function resolveApiUrl(url) {
  if (!url) return null
  if (url.startsWith('http://') || url.startsWith('https://')) return url
  return `${API_ORIGIN}${url}`
}
```

用于将后端返回的相对路径（如 `/uploads/cover/123.jpg`）解析为完整的可访问 URL。如果已经是绝对 URL，直接返回。

## 10.6 访客追踪

文件：`src/lib/visitor.js`

### 10.6.1 UUID 生成策略

```js
function generateVisitorId() {
  if (typeof crypto !== 'undefined' && typeof crypto.randomUUID === 'function') {
    return `visitor-${crypto.randomUUID()}`
  }
  return `visitor-${Date.now()}-${Math.random().toString(36).slice(2, 10)}`
}
```

- 优先使用 `crypto.randomUUID()` 生成标准 UUID v4
- 降级方案：时间戳 + 随机字符串

### 10.6.2 存储与获取

| 函数 | 行为 |
|------|------|
| `getOrCreateVisitorId()` | 从 localStorage 读取 `fdsm-visitor-id`；若不存在则生成新 ID 并存入 localStorage，返回该 ID |
| `getStoredVisitorId()` | 仅读取 localStorage，不创建新 ID |

### 10.6.3 使用方式

`request()` 函数在每次 API 调用时都会通过 `getOrCreateVisitorId()` 获取访客 ID，并作为 `X-Visitor-Id` header 发送给后端。这使得后端可以追踪匿名访客的行为（浏览量统计等），而无需用户登录。

export function resolveRoleTier({ membership, isAuthenticated = false } = {}) {
  if (!isAuthenticated) return 'guest'

### 10.8.1 语言检测优先级
