# Round 100：专题选择排序与发布反馈补强

## 对齐结论
- [x] 编辑后台里的“专题选择”不能只停留在可加可删，必须能明确排序，发布后按同样顺序写入专题关系。
- [x] 编辑稿详情应自带一批可选专题候选，避免每次进页面都从空状态开始搜索。
- [x] 发布成功反馈里要直接告诉管理员同步到了哪些专题，不能只显示文章编号。
- [x] 本轮继续沿用 Round 99 的截图验收脚本，确保补强后的专题顺序和发布反馈能被真实看见。

## 原子任务
- [x] P0 | 后端 | 在编辑稿详情返回结构里补充 `topic_candidates`，优先给出和当前标题/主题相关的已发布专题候选。
- [x] P0 | 后端 | 在发布返回结果里补充 `selected_topics`，保持和实际写入 `topic_articles.sort_order` 一致。
- [x] P0 | 后端测试 | 覆盖详情包含 `topic_candidates`，以及发布返回里带有按顺序同步的专题列表。
- [x] P0 | 前端 | 把“专题选择”从纯标签展示补成可排序列表，至少支持上移、下移、移除，并显式展示当前顺序。
- [x] P0 | 前端 | 进入编辑稿时优先显示后端返回的专题候选，不再默认空白。
- [x] P0 | 前端 | 发布成功提示补充专题同步结果，让管理员立即知道进入了哪些专题。
- [x] P0 | 验收脚本 | 更新截图种子和验收脚本，让截图里能看到多专题已选状态和排序控件。
