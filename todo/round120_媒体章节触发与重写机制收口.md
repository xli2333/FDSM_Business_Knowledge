# Round 120：媒体章节触发与重写机制收口

## 对齐结论
- [x] 当前媒体章节的触发链路虽然能工作，但规则不够清晰：上传文本会自动识别，`生成文案` 也会刷新章节，而普通保存因为总会带上转录/脚本文本字段，容易把章节反复重写。
- [x] 本轮目标是把章节触发机制收口成四条固定规则：上传文本立即识别；保存时只有文本源变化才自动重算；点击 `生成文案` 强制按最新文本重写章节；另外提供单独的 `重写章节` 入口，不必为了改目录重跑整套摘要和简介。
- [x] 本轮会同时补齐后端触发判定、独立重写接口、前端按钮与提示文案，并把真实验收更新到新规则。
- [x] 全部 todo 与代码改动保持 UTF-8 中文，不引入中文编码错误或 JSX / Python 语法错误。

## 原子任务
- [x] P0 | 后端触发 | 收口草稿保存时的章节重算条件，只有转录/脚本文本真正变化时才自动重识别。
- [x] P0 | 后端重写 | 新增媒体草稿 `重写章节` 独立接口，使用最新转录/脚本强制刷新章节，但不重跑摘要与节目简介。
- [x] P0 | 后端生成 | 明确 `生成文案` 会强制重写章节，保证它和章节识别的触发语义一致。
- [x] P0 | 前端工作台 | 在媒体后台增加 `重写章节` 按钮，并更新提示文案，明确四条触发规则。
- [x] P0 | 前端交互 | `重写章节` 点击前先落当前草稿，再调用独立重写接口，回填最新章节并给出成功反馈。
- [x] P0 | 测试 | 补后端测试，覆盖“普通保存不乱重写”“文本变化时自动重算”“重写章节独立生效”“生成文案强制刷新章节”。
- [x] P0 | 验收 | 更新 `frontend/scripts/media_acceptance_round106.mjs`，覆盖上传后自动识别和独立重写章节入口。
- [x] P0 | 验证 | 跑通 `pytest backend/tests/test_admin_editorial_media_features.py -q`、`npm run build`、`npm run media:acceptance:round106`。

## 结束标准
- [x] 上传文本后章节会立刻出现。
- [x] 普通保存不会在文本没变化时把章节无意义重写。
- [x] 管理员可以单独点击 `重写章节` 强制刷新目录。
- [x] 点击 `生成文案` 时也会按最新文本强制重写章节。
- [x] `pytest backend/tests/test_admin_editorial_media_features.py -q`、`npm run build`、`npm run media:acceptance:round106` 通过。

## 备注
