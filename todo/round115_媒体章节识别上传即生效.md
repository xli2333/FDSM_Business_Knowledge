# Round 115：媒体章节识别上传即生效

## 对齐结论
- [x] 当前媒体工作台里的章节识别并不是没实现，而是触发时机过晚：只有在“生成文案”或“发布前兜底”时才会抽取章节，上传转录或脚本文本后不会立即写回 `chapters`。
- [x] 本轮目标是把章节识别前移到文本上传和草稿保存链路，让管理员一上传转录 / 脚本就能看到识别结果，而不是先看到误导性的空态提示。
- [x] 本轮同时收口空态提示：区分“还没上传文本”和“已上传文本但未识别到可用时间戳”两种状态。
- [x] 本轮继续保持轻量实现，不新增复杂章节编辑器；重点是让已有识别能力真正落到用户可见链路。
- [x] 全部 todo 文档与代码改动保持 UTF-8 中文，不引入中文编码错误或 JSX / Python 语法错误。

## 原子任务
- [x] P0 | 后端上传 | 在媒体转录 / 脚本文本上传成功后，立即从文本中抽取时间戳章节并写入草稿。
- [x] P0 | 后端保存 | 在草稿保存链路里，当转录 / 脚本文本变更且未显式传入章节时，自动重新抽取章节，避免章节识别只依赖上传入口。
- [x] P0 | 后端兼容 | 保留发布前章节兜底逻辑，确保旧草稿和历史数据不回归。
- [x] P0 | 前端空态 | 在 `MediaStudioPage.jsx` 里区分“未上传文本”和“已上传文本但未识别到时间戳”的章节空态文案。
- [x] P0 | 测试 | 更新后端媒体测试，覆盖“上传脚本后立即出现章节”与“无时间戳文本不会伪造章节”。
- [x] P0 | 验收 | 更新 `frontend/scripts/media_acceptance_round106.mjs`，把真实验收补到“上传脚本后，不点生成文案也能立即看到章节列表”。

## 结束标准
- [x] 上传转录或脚本文本后，媒体工作台会立即显示识别章节。
- [x] 已上传文本但无有效时间戳时，会显示更准确的提示，而不是继续提示先上传文本。
- [x] 旧的生成文案 / 发布前章节兜底逻辑继续可用。
- [x] `pytest backend/tests/test_admin_editorial_media_features.py -q`、`npm run build`、`npm run media:acceptance:round106` 通过。
