# Round 91：已发布稿移出草稿箱与正文打回重编

## 对齐规则
- [x] 本轮先不直接改业务代码，先把“已发布稿移出草稿箱”和“正文打回草稿箱重新修改”拆成原子级 todo。
- [x] “移出草稿箱”不是删除正式文章，只是把该编辑记录从编辑台默认草稿列表隐藏。
- [x] “打回草稿箱重新修改”是管理员把线上正文重新拉回编辑台继续改稿，不等于下架，也不应影响当前线上已发布内容。
- [x] 已存在关联编辑稿时优先复用，不重复新建多份同一正式文章的并行编辑稿。
- [x] 全部文案与 todo 文件保持 UTF-8 中文，避免再次出现中文乱码。

## 待推进
- [x] P0 | 数据模型 | 为 `editorial_articles` 增加“是否显示在草稿箱/编辑台”的独立状态字段，例如 `draft_box_state`，默认值为 `active`。
- [x] P0 | 数据迁移 | 为历史 `editorial_articles` 数据补默认值，保证升级后旧草稿、旧已发布稿都不会因为空值导致列表异常。
- [x] P0 | 规则 | 明确三种操作语义并落到代码常量：`删除草稿`、`移出草稿箱`、`打回草稿箱重编`，避免继续混用“删除”。
- [x] P0 | 后端列表 | 改造 `GET /api/editorial/articles`，默认仅返回 `draft_box_state = active` 的记录，已移出的记录不再继续堆在草稿列表里。
- [x] P0 | 后端列表 | 为编辑台列表接口预留按 `draft_box_state` 过滤的能力，后续可支持查看已移出的记录，不把查询能力写死。
- [x] P0 | 后端服务 | 新增“移出草稿箱”服务函数，只允许对已发布且已关联正式文章的编辑记录执行，禁止误删数据库记录。
- [x] P0 | 后端路由 | 新增独立接口，例如 `POST /api/editorial/articles/{id}/archive-draft-box`，语义明确为“从草稿箱移出”。
- [x] P0 | 后端约束 | “移出草稿箱”执行后保留 `article_id`、`published_final_html`、标签、栏目、手工稿备份，不得破坏线上文章和重编能力。
- [x] P0 | 前端草稿列表 | 为已发布稿显示“移出草稿箱”动作，不再维持“没有任何操作按钮”的状态。
- [x] P0 | 前端主操作区 | 当前选中的是已发布稿时，将红色危险操作从“删除草稿”切换为“移出草稿箱”，未发布稿仍保留“删除草稿”。
- [x] P0 | 前端交互 | “移出草稿箱”前增加二次确认，明确提示“只会从编辑台隐藏，不会删除正式文章”。
- [x] P0 | 后端重编 | 新增“正文打回草稿箱”服务：输入正式文章 `article_id`，优先查找已关联的编辑记录并重新激活到草稿箱。
- [x] P0 | 后端重编 | 若该正式文章还没有关联编辑记录，则基于当前正式文章内容新建一条编辑记录，并自动挂上 `article_id` 关联。
- [x] P0 | 后端重编 | 新建的重编记录需要回填标题、摘要、作者、机构、发布日期、访问级别、标签、栏目、当前线上 HTML 与正文纯文本，避免打回后还是空白稿。
- [x] P0 | 后端重编 | 对“正文打回草稿箱”做幂等控制：同一篇正式文章重复点击时优先复用最近一条关联编辑记录，不无限新增重复稿。
- [x] P0 | 后端重编 | 被重新激活的编辑记录要重新出现在草稿箱顶部，便于管理员立即进入继续修改。
- [x] P0 | 后端路由 | 新增正式文章重编入口，例如 `POST /api/editorial/source-articles/{article_id}/reopen-draft` 或等价语义接口。
