# Round 106：音视频上传按文章草稿工作台对齐

## 对齐结论
- [x] 文章上传当前采用“导入原稿 -> 生成 `editorial_articles` 草稿 -> 自动生成摘要/标签/排版 -> 发布时写入 `articles` 正式表 -> 发布后自动离开草稿箱”的双轨模式。
- [x] 媒体后台原先采用“上传文件 -> 直接创建或更新 `media_items`”的单轨模式，没有独立媒体草稿表，也没有与文章一致的草稿箱、回编、发布收口逻辑。
- [x] 原媒体上传路由虽然没有显式大小校验，但使用 `await file.read()` 整包进内存，不算真正的无应用层大小限制。
- [x] 前台音视频播放已经支持在受限场景下直接基于主媒体 URL 做限时试看/试听，因此 `preview_url` 不是新后台流程的必要主链路。
- [x] 本轮目标是把音视频改造成更接近文章编辑台的“媒体草稿 -> 自动生成文字素材 -> 发布成正式媒体”的工作流。

## 原子任务
- [x] P0 | 数据模型 | 新增独立 `media_drafts` 表，承载 `media_item_id` 关联、媒体元数据、主媒体 URL、转录/脚本、摘要、节目简介、章节、状态、草稿箱状态、工作流状态、发布时间与文案更新时间等字段。
- [x] P0 | 数据模型 | 为媒体草稿补齐“工作稿 / 线上稿”双轨字段，区分当前编辑态与已发布版本，避免草稿直接覆盖线上媒体。
- [x] P0 | 数据迁移 | 把旧 `media_items` 里的 `draft` 记录迁入 `media_drafts`，保留旧 `published` 记录继续作为正式表。
- [x] P0 | 后端上传 | 重写主媒体上传链路，改为分块流式落盘，不再整包 `read()` 到应用层内存，也不新增应用层文件大小上限。
- [x] P0 | 后端上传 | 主媒体上传后回填草稿所需的基础元数据，包括存储 URL、原始文件名衍生标题、时长与来源地址。
- [x] P0 | 后端上传 | 废弃 `usage=preview`，改为支持 `transcript` 与 `script` 两类文本上传。
- [x] P0 | 后端上传 | 复用文章上传已有的文本抽取逻辑，支持从 `.md/.txt/.html/.htm/.docx` 抽取转录或脚本文本。
- [x] P0 | 后端草稿 | 新增“从上传创建媒体草稿”链路，上传主媒体文件后自动创建或绑定媒体草稿。
- [x] P0 | 后端草稿 | 新增媒体草稿列表、详情、新建、更新、删除接口，默认只返回草稿箱里的活跃草稿。
- [x] P0 | 后端草稿 | 未发布媒体草稿允许删除；已发布且已关联正式媒体的草稿禁止从草稿箱直接删除。
- [x] P0 | AI 素材 | 基于标题 + 转录/脚本自动生成 `summary` 与 `body_markdown` 节目简介等文字素材，减少人工首填负担。
- [x] P0 | AI 素材 | 为媒体文案定义独立 contract：只输出正文素材，不写“以下是”之类前置说明，不虚构事实。
- [x] P0 | AI 素材 | 生成后的文字素材写回草稿表，并保留模型、更新时间与人工覆盖痕迹。
- [x] P1 | AI 素材 | 当转录/脚本包含明确时间戳时自动生成章节；信号不足时保持为空，不伪造时间点。
- [x] P0 | 发布链路 | 新增“发布媒体草稿”服务，将草稿同步写入或更新 `media_items` 正式表，并冻结当前线上版本快照。
- [x] P0 | 发布链路 | 媒体发布成功后，草稿默认自动离开草稿箱；前台 `/audio` 与 `/video` 始终只消费正式 `published` 内容。

- [x] P0 | 前端页面 | 重构 [MediaStudioPage.jsx](C:/Users/LXG/fdsmarticles/frontend/src/pages/MediaStudioPage.jsx)，形成“草稿箱 + 编辑区 + 操作区”的文章式工作台布局。
- [x] P0 | 前端页面 | 将“上传试看文件”与 `preview_url` 主编辑入口替换为“上传转录 / 上传脚本”，并保留文本区可继续人工编辑。

## 结束标准
- [x] 音频 / 视频上传不再依赖整包读入内存，应用层不再设置人为大小上限。
- [x] 媒体后台不再使用“上传试看文件”作为主编辑入口，而是改为上传转录 / 脚本并驱动文字素材生成。
- [x] 管理员可以像文章工作台一样管理媒体草稿、保存、自动生成素材、发布、回编。
- [x] 已发布媒体默认不继续堆在草稿箱里；需要修改时可重新打回草稿箱。
- [x] 前台 `/audio` 与 `/video` 的播放、试看/试听、权限分层逻辑保持可用且不回归。
