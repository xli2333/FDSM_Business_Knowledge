# Round 101：栏目自动更新与摘要成品统一

## 对齐结论
- [x] 栏目链路现在没有真正做到“最新发布自动顶上来”，因为栏目页和首页栏目预览都把 `article_columns.is_featured` 放在 `publish_date` 前面排序，导致新发布文章即使已经写入栏目关系，也可能长期被旧精选文章压住。
- [x] 编辑台当前没有独立的“AI 摘要成品区”，只有短 `excerpt` 和正文最终 HTML；这不满足“自动生成摘要、可见、可编辑、可发布”的要求。
- [x] 前台正文页和编辑台当前不是同一套成品契约：正文页摘要走独立 `summary_preview_service`，编辑台右侧正文走 `RichPreviewEditor`，正式正文页再走文章详情页 iframe 壳，三套展示壳并不统一。
- [x] 下一轮目标不是再补一个局部按钮，而是把“栏目自动更新”“摘要可编辑”“前后台成品一致”三条链路打通成一套发布契约。

## 当前问题核对
- [x] P0 | 审计 | 核对栏目页查询 `get_column_articles` 当前排序：`ac.is_featured DESC, a.publish_date DESC, a.view_count DESC`，确认这会压住新发布稿件。
- [x] P0 | 审计 | 核对首页栏目预览 `column_previews` 当前排序：`ac.is_featured DESC, a.publish_date DESC, a.id DESC`，确认这同样会压住新发布稿件。
- [x] P0 | 审计 | 核对编辑稿发布链路确实已经写入 `article_columns`，说明问题不在“没写栏目关系”，而在“栏目展示排序策略不对”。
- [x] P0 | 审计 | 核对正文页摘要当前来自 `summary_preview_service.render_summary_preview_html()`，并不是编辑台生成或保存的摘要成品 HTML。
- [x] P0 | 审计 | 核对正文页正文当前优先读 `published_final_html/final_html/html_wechat`，但摘要和正文不是同一套成品存储和渲染来源。
- [x] P0 | 审计 | 核对编辑台右侧 `RichPreviewEditor` 与正文页 `AutoHeightPreviewFrame` 使用的展示壳和容器样式并不统一。

## 栏目自动更新修复
- [x] P0 | 后端 | 重构栏目页文章排序，改为默认按 `publish_date DESC, id DESC` 自动倒序，不再让 `is_featured` 永久压住最新发布稿件。
- [x] P0 | 后端 | 重构首页栏目预览排序，默认展示栏目内最新发布文章，确保新发布稿件在首页栏目区下一次请求即可出现。
- [x] P0 | 后端 | 如果仍需保留“精选”能力，要拆成独立显式配置位，不能继续混在自动更新流里影响倒序展示。
- [x] P0 | 后端 | 发布编辑稿后，确保栏目关系写入后无需其他补数据动作，栏目页和首页栏目预览的下一次请求即可直接读到新稿。
- [x] P0 | 前端 | 校对栏目页和首页栏目区文案，明确这是“最新栏目文章流”，避免用户误解为手动运营位。

## AI 摘要成品链路
- [x] P0 | 数据层 | 给 `editorial_articles` 增加摘要相关字段，至少包含：`summary_markdown`、`summary_html`、`summary_editor_document_json`、`summary_model`、`summary_updated_at`、`published_summary_html`。
- [x] P0 | 数据层 | 运行时自动补列和旧库兼容补齐，不能要求删库重建。
- [x] P0 | 后端 | 新增“自动生成 AI 摘要”接口，输入原稿/正文，输出结构化摘要 markdown + 成品 HTML + 编辑文档。
- [x] P0 | 后端 | 自动摘要不能只生成一句 `excerpt`，而要生成可单独展示的摘要成品块，并保留模型、更新时间和回退字段。
- [x] P0 | 后端 | 编辑稿详情返回里补充摘要字段，至少包含：`summary_markdown`、`summary_html`、`summary_editor_document`、`summary_model`、`summary_updated_at`。
- [x] P0 | 后端 | 保存编辑稿时支持单独保存摘要人工修改，不影响正文成品。
- [x] P0 | 后端 | 发布编辑稿时把“当前工作摘要成品”固化到 `published_summary_html`，避免再次编辑时提前污染线上展示。
- [x] P0 | 后端 | 已发布文章打回草稿箱后，摘要也要跟正文一样走“工作稿 / 线上稿”双轨，不在再次发布前覆盖线上摘要。

## 编辑台摘要工作区
- [x] P0 | 前端 | 在编辑台新增独立“AI 摘要”工作区，位置放在正文成品区前，符合先看摘要再看正文的编辑流。
- [x] P0 | 前端 | 摘要工作区提供：自动生成摘要、编辑摘要、预览摘要、保存摘要、恢复摘要备份。
- [x] P0 | 前端 | 摘要工作区使用和正文一致的可编辑成品模式，不再只是纯文本 textarea。
- [x] P0 | 前端 | 摘要状态要明确区分：未生成、AI 已生成、人工已修改、线上已发布、待重新发布。
- [x] P0 | 前端 | 发布前校验补充摘要校验项：未生成摘要或摘要成品为空时，按配置决定是否阻止发布。

## 前后台成品统一
- [x] P0 | 后端 | 建立统一的“成品 HTML 契约”：编辑台预览、正式正文页、正式摘要页都优先消费同一份已发布 HTML，而不是各自重新套壳。
- [x] P0 | 后端 | 正文页摘要改为优先读取 `published_summary_html`，不再默认用 `summary_preview_service` 重新生成另一套壳。
- [x] P0 | 后端 | 若摘要尚未有已发布成品，再明确走降级逻辑，而不是静默套另一套默认模板。
- [x] P0 | 前端 | 正文页摘要 iframe 与编辑台摘要预览使用同一份 HTML 数据源，保证管理员看到什么，用户就看到什么。
- [x] P0 | 前端 | 正文页正文 iframe 与编辑台正文预览继续收口到同一份已发布 HTML，去掉不必要的二次包装差异。
- [x] P0 | 前端 | 统一摘要区和正文区的外层容器宽度、留白、背景和滚动策略，减少“后台像一个模板、前台像另一个模板”的割裂感。
- [x] P0 | 前端 | 对旧稿没有 `published_summary_html` 的情况提供清晰降级展示，不能白屏，也不能把旧摘要样式和新摘要样式混在一起。
