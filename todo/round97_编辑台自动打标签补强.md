# Round 97：编辑台自动打标签补强
## 对齐规则
- [x] 当前编辑台自动打标签过弱，不能再接受“只打出一个低信号标签”这种结果。
- [x] 本轮要先查清现状，再直接补强实现，不停留在说明层。
- [x] 标签补强要同时覆盖三层：AI 生成、无 AI/AI 失败时的本地兜底、低信号脏标签过滤。
- [x] 继续保持 UTF-8 中文，不引入中文编码或 JSX 问题。

## 本轮待办
- [x] P0 | 审计 | 核实现有编辑台自动打标签链路是“弱启发式 + `suggest_editorial_metadata` 一次调用”；而旧实现里 AI 失败会被静默吞掉，随后只剩默认机构名等极弱兜底。
- [x] P0 | AI 调用 | 将编辑台元数据建议改成更稳定的 Gemini JSON 直连链路，避免旧的 langchain 路径失败后直接回空结果。
- [x] P0 | Prompt | 强化标签 prompt，明确要求输出更具体的主题、行业和实体标签，避免“商业/企业”这类泛标签。
- [x] P0 | 规则兜底 | 增加编辑台长文的本地信号抽取，覆盖数据治理、数据安全、隐私保护、数据泄露、基因检测、生物科技、医疗健康，以及正文中的关键实体名。
- [x] P0 | 过滤 | 拦截单个字母、作者缩写等低信号脏标签，避免再次出现 `L` 这种结果。
- [x] P0 | 测试 | 新增自动化测试，覆盖 AI 成功解析、低信号脏标签过滤、AI 失败时本地兜底三条路径。
- [x] P0 | 验证 | 用 23andMe 这篇稿子的真实文本跑了一遍：AI 路径已返回 `23andMe / 基因检测 / 数据泄露 / 隐私保护 / GDPR / 数据治理 / 网络安全 / 葛兰素史克` 等多维标签；fallback 也能返回 `基因检测 / 医疗健康 / 数据治理 / 数据安全 / 隐私保护 / 网络安全 / 23andMe / GDPR / GSK / ICO / OPC`。
- [x] P0 | 验收 | `pytest backend/tests/test_admin_editorial_media_features.py -q`
- [x] P0 | 验收 | `python -m compileall backend`

## 结束标准
- [x] 23andMe 这篇稿子能产出多于 1 个高信号标签。
- [x] 标签中同时能看到主题/行业/实体，而不是只剩机构名或作者缩写。
- [x] AI 失败时也不会退化成只有一个无意义标签。

## 结论
- [x] Round 97 已把“AI 静默失败 + 本地兜底过弱 + 脏标签不过滤”三处核心缺口补齐，当前没有明确的 P0 阻塞项。
