# Round 87：编辑台预览区可编辑能力拆解

## 已完成对齐
- [x] P0 | 盘点 | 确认正式站当前编辑台右侧只有 `iframe + final_html` 预览，没有富文本编辑能力。
- [x] P0 | 盘点 | 确认旧“公众号排版”工程只有 Markdown 编辑、本地预览、AI 反馈重排、微信草稿回读，没有可直接搬运的公众号式所见即所得编辑器。
- [x] P0 | 盘点 | 确认当前后端只持久化 `source_markdown / content_markdown / final_html / published_final_html / tags / 栏目`，没有“结构化编辑文档”字段。
- [x] P0 | 决策 | 预览区可编辑能力不采用手搓 `contenteditable + execCommand`，而是引入成熟富文本编辑内核承接 HTML 编辑与导出。

## 目标对齐
- 左边继续保留原稿区，作为原始输入与 AI 自动排版的来源。
- 右边不再只是只读预览，而是“可编辑的最终成品区”，默认承接自动排版后的 HTML。
- 管理员先自动排版，再在右侧做基础微调，再保存，再发布。
- 发布永远以右侧最终成品 HTML 为准，不直接把左侧原稿发布出去。
- 标签、栏目、发布前校验都要围绕“当前右侧最终稿”重新计算和展示。

## 实施顺序
1. 先定编辑器内核和数据真源，再动数据库和接口。
2. 先打通 HTML 导入导出，再做右侧画布和工具栏。
3. 先做基础命令闭环和保存发布，再做表格、表情、图片细调这类增强项。
4. 先确保旧稿兼容可打开，再处理历史版本、版本对比这类二轮体验。
