# Round 108：媒体草稿箱按删除收口与单入口文本上传

## 对齐结论
- [x] 媒体草稿箱不应保留已发布稿件；发布成功后，草稿应自动从草稿箱删除，而不是保留为已发布草稿。
- [x] 音频与视频的正式内容仍需支持打回草稿箱，但这个动作属于正式内容区，不属于草稿箱区。
- [x] 转录与脚本在后台是同一类“文本素材上传”能力，前端不应再分两个按钮；音频显示“上传转录”，视频显示“上传脚本”即可。
- [x] 后台无需预览或编辑原始转录 / 脚本文本，只保留 AI 生成后的摘要、节目简介与目录章节作为可见结果。

## 原子任务
- [x] P0 | 后端发布 | 调整媒体发布逻辑：发布成功后删除当前草稿记录，而不是归档为已发布草稿。
- [x] P0 | 后端删除 | 调整草稿删除语义：草稿箱里的删除只删除草稿，不影响已经发布的正式媒体内容。
- [x] P0 | 后端回编 | 保留正式媒体打回草稿箱能力，但新建 / 复用的草稿记录应回到纯草稿状态，不再伪装成已发布草稿。
- [x] P0 | 前端页面 | 调整媒体草稿箱状态文案与计数，不再把已发布稿件视为草稿箱成员。
- [x] P0 | 前端页面 | 把“上传转录 / 上传脚本”收敛成单按钮，并按媒体类型切换按钮文案与上传用途。
- [x] P0 | 前端页面 | 移除原始转录 / 脚本文本预览区，只保留 AI 生成的摘要、节目简介与章节展示。
- [x] P0 | 前端页面 | 正式内容区继续保留“打回草稿箱”动作，草稿箱区只保留删除。
- [x] P0 | 测试 | 更新后端媒体工作流测试，覆盖“发布即删除草稿”“删除草稿不删正式内容”“正式内容打回草稿箱”。
- [x] P0 | 验收 | 更新 `media_acceptance_round106.mjs` 验证新交互，并重新跑后端测试、媒体验收脚本与前端构建。

## 结束标准
- [x] 媒体草稿发布后不再留在草稿箱里。
- [x] 草稿箱仅承担草稿编辑与删除职责；正式内容区承担打回草稿箱职责。
- [x] 后台只提供一个文本素材上传入口，且按音频 / 视频显示对应文案。
- [x] 后台不再显示原始转录 / 脚本文本，只显示 AI 生成摘要、节目简介与章节。
- [x] `pytest`、`npm run build`、`npm run media:acceptance:round106` 全部通过。
