# 大数据资产交接方案

生成日期：2026-06-02

本文给出当前项目最完整的交接方式，重点说明本地大数据如何交给接手同事。结论是：代码走 GitHub，真实配置和数据不要走 GitHub；运行数据应通过单独数据包、移动硬盘、内网文件服务器、SFTP/rsync 或对象存储交接，并做校验。

## 1. 交接总口径

完整交接拆成四类资产：

| 类别 | 推荐交接方式 | 是否进 GitHub | 说明 |
| --- | --- | --- | --- |
| 代码 | GitHub 仓库 | 是 | 前端、后端、Docker、部署脚本、测试、文档 |
| 运行数据 | 单独大文件渠道 | 否 | `data/` 整目录，数据库、原始文章、图片、上传媒体、FAISS 索引 |
| 真实密钥 | 安全渠道 | 否 | `.env.production`、Gemini key、CAS 信息、Redis 密码、备份凭据 |
| 历史归档 | 按需交接 | 否 | `archive/`、`backups/` 历史备份、旧副本、QA/报告 |

不要把本地整个项目目录压缩给同事。这样会混入 `.env`、`.venv`、`node_modules`、旧归档、历史备份、发布包副本和不必要数据。

## 2. 当前大数据资产

当前 Docker 主线通过 `docker-compose.yml` 将宿主机 `./data` 挂载到容器 `/data`。后端配置中：

```text
FDSM_DATA_DIR=/data
SQLITE_DB_PATH=/data/fudan_knowledge_base.db
FAISS_DB_DIR=/data/faiss_index_business
UPLOADS_DIR=/data/uploads
AUDIO_DIR=/data/audio
```

所以生产或同事本地复现时，核心是迁移 `data/`。

当前盘点如下：

| 路径 | 当前大小/数量 | 是否必须交接 | 用途 |
| --- | ---: | --- | --- |
| `data/` | 约 5.02GB，约 20256 个文件 | 是 | 当前 Docker 运行数据总目录 |
| `data/fudan_knowledge_base.db` | 约 703.8MB | 是 | SQLite 主库 |
| `data/Fudan_Business_Knowledge_Data/` | 约 3.80GB，约 20220 个文件 | 是 | 原始文章、图片、批处理知识库材料 |
| `data/faiss_index_business/` | 约 155.3MB，2 个文件 | 是 | FAISS 业务向量索引 |
| `data/uploads/` | 约 239.8MB，约 29 个文件 | 是 | 后台上传的图片、音频、视频、转录稿 |
| `data/audio/` | 约 29.2MB，4 个文件 | 是 | 本地音频文件 |
| `backups/fudan_knowledge_base.handoff_20260602-142839.db.gz` | 约 223.3MB | 建议随包附带 | 最新数据库交接压缩备份，`quick_check=ok` |
| `archive/legacy_pre_knowledge_base/local_runtime_duplicates_20260602/` | 约 5.33GB | 否，除非要历史副本 | 根目录旧数据库和同名素材副本归档，不是当前运行数据 |

`data/`、`backups/`、`archive/legacy_pre_knowledge_base/` 都被 `.gitignore` 排除，不会进入 GitHub。

## 3. 推荐交接方案

### 方案 A：完整运行数据交接，推荐

给同事以下内容：

```text
fdsmarticles/
  data/
    fudan_knowledge_base.db
    Fudan_Business_Knowledge_Data/
    faiss_index_business/
    uploads/
    audio/
  backups/
    fudan_knowledge_base.handoff_20260602-142839.db.gz
  数据校验清单
```

优点：

- 同事拿到后可以完整复现 Docker 当前效果。
- 文章、图片、视频、音频、RAG 检索都在。
- 数据库备份可用于恢复或比对。

适用：

- 正式交接。
- 服务器迁移。
- 同事要继续开发、测试、部署。

### 方案 B：数据库轻量交接，只适合临时查看

只给：

```text
backups/fudan_knowledge_base.handoff_20260602-142839.db.gz
```

适用：

- 同事只想看表结构、文章记录、栏目/专题数据。
- 不需要跑完整前端媒体和 RAG。

限制：

- 没有文章图片原始素材。
- 没有上传视频/音频。
- 没有 FAISS 索引。
- 页面中引用媒体时可能缺资源。
- AI/RAG 检索效果不能完整验证。

### 方案 C：历史归档交接，按需

如果同事需要追溯历史副本，再额外给：

```text
archive/legacy_pre_knowledge_base/local_runtime_duplicates_20260602/
```

这不是生产运行数据，只用于审计、追溯或比对。

## 4. 打包前准备

正式打包前，建议先停止会写入数据库或上传目录的容器，避免边拷边写：

```powershell
docker compose stop backend-web backend-worker backend-housekeeping backup frontend
```

如果只是生成 SQLite 备份，可以不停服务，因为 SQLite backup API 能拿一致快照；但如果要复制整个 `data/` 目录，建议停服务。

打包前做一次数据库检查：

```powershell
@'
import sqlite3
from pathlib import Path

p = Path("data/fudan_knowledge_base.db")
con = sqlite3.connect(f"file:{p}?mode=ro", uri=True)
print("quick_check =", con.execute("PRAGMA quick_check").fetchone()[0])
print("articles =", con.execute("SELECT COUNT(*) FROM articles").fetchone()[0])
print("chunks =", con.execute("SELECT COUNT(*) FROM article_chunks").fetchone()[0])
print("embeddings =", con.execute("SELECT COUNT(*) FROM article_chunk_embeddings").fetchone()[0])
print("business_article_count =", con.execute("SELECT value FROM meta WHERE key='business_article_count'").fetchone()[0])
con.close()
'@ | python -
```

当前应看到的基准：

```text
quick_check = ok
articles = 2146
chunks = 8900
embeddings = 8900
business_article_count = 2142
```

## 5. 大数据传输方式

### 方式 1：移动硬盘或内网共享，最稳

Windows 到 Windows 推荐 `robocopy`，不需要把 5GB、2 万多个文件先压缩成一个大包：

```powershell
$target = "E:\fdsmarticles-handoff"
New-Item -ItemType Directory -Force -Path $target
robocopy .\data "$target\data" /MIR /COPY:DAT /DCOPY:DAT /R:2 /W:2 /MT:8
New-Item -ItemType Directory -Force -Path "$target\backups"
Copy-Item .\backups\fudan_knowledge_base.handoff_20260602-142839.db.gz "$target\backups\"
```

优点：

- 断点和重试能力更好。
- 适合大量图片、小文件和媒体文件。
- 不会因为单个压缩包损坏导致全部不可用。

### 方式 2：SFTP/rsync 到服务器

Linux/服务器迁移推荐 `rsync`：

```bash
rsync -a --info=progress2 --partial data/ user@server:/srv/fdsmarticles/data/
rsync -a --info=progress2 backups/fudan_knowledge_base.handoff_20260602-142839.db.gz user@server:/srv/fdsmarticles/backups/
```

如果只能用 `scp`：

```bash
scp -r data user@server:/srv/fdsmarticles/
scp backups/fudan_knowledge_base.handoff_20260602-142839.db.gz user@server:/srv/fdsmarticles/backups/
```

`rsync` 优先于 `scp`，因为中断后可以续传，且可重复执行校正差异。

### 方式 3：压缩包上传网盘或对象存储

如果必须压缩，优先用 7-Zip 分卷，避免网盘或文件系统单文件限制：

```powershell
7z a -t7z -mx=3 -v1900m fdsmarticles-runtime-data_20260602.7z .\data .\backups\fudan_knowledge_base.handoff_20260602-142839.db.gz
```

同事解压：

```powershell
7z x fdsmarticles-runtime-data_20260602.7z.001
```

如果没有 7-Zip，可用 Windows 自带 `tar`：

```powershell
tar -czf fdsmarticles-runtime-data_20260602.tar.gz data backups/fudan_knowledge_base.handoff_20260602-142839.db.gz
```

注意：图片、视频、音频已经是压缩格式，压缩率不会很高。压缩包适合网盘交接，不是本地拷贝的首选。

## 6. 校验方式

### 生成数据清单

交出数据前，在本机生成清单：

```powershell
$manifest = "fdsmarticles-data-manifest_20260602.tsv"
"sha256`tbytes`tpath" | Set-Content -Encoding UTF8 $manifest
Get-ChildItem -LiteralPath data -Recurse -File -Force |
  Sort-Object FullName |
  ForEach-Object {
    $hash = Get-FileHash -LiteralPath $_.FullName -Algorithm SHA256
    $relative = $_.FullName.Substring((Resolve-Path ".").Path.Length + 1)
    "$($hash.Hash)`t$($_.Length)`t$relative"
  } | Add-Content -Encoding UTF8 $manifest
Get-FileHash -LiteralPath backups\fudan_knowledge_base.handoff_20260602-142839.db.gz -Algorithm SHA256 |
  ForEach-Object {
    "$($_.Hash)`t$((Get-Item backups\fudan_knowledge_base.handoff_20260602-142839.db.gz).Length)`tbackups/fudan_knowledge_base.handoff_20260602-142839.db.gz"
  } | Add-Content -Encoding UTF8 $manifest
```

这会遍历约 2 万个文件，耗时正常。清单要跟数据一起给同事。

### 同事侧复核清单

同事拿到数据后，至少检查目录数量和关键文件哈希：

```powershell
Get-ChildItem -LiteralPath data -Recurse -File -Force | Measure-Object
Get-ChildItem -LiteralPath data -Recurse -File -Force | Measure-Object -Property Length -Sum
Get-FileHash -LiteralPath backups\fudan_knowledge_base.handoff_20260602-142839.db.gz -Algorithm SHA256
Get-FileHash -LiteralPath data\fudan_knowledge_base.db -Algorithm SHA256
Get-FileHash -LiteralPath data\faiss_index_business\index.faiss -Algorithm SHA256
Get-FileHash -LiteralPath data\faiss_index_business\index.pkl -Algorithm SHA256
```

如果需要逐文件完整校验，可用清单逐行比对；重点是确保：

- `data/` 文件数约 20256。
- `data/` 总大小约 5.02GB。
- `data/fudan_knowledge_base.db` 存在且约 703.8MB。
- `data/faiss_index_business/index.faiss` 和 `index.pkl` 存在。
- `data/uploads/` 和 `data/audio/` 存在。

## 7. 同事恢复到本地 Docker

同事拿到代码仓库后：

```powershell
git clone <repo-url> fdsmarticles
cd fdsmarticles
```

把交接来的 `data/` 放到仓库根目录：

```text
fdsmarticles/data/fudan_knowledge_base.db
fdsmarticles/data/Fudan_Business_Knowledge_Data/
fdsmarticles/data/faiss_index_business/
fdsmarticles/data/uploads/
fdsmarticles/data/audio/
```

准备本地环境文件：

```powershell
Copy-Item .env.docker.example .env.docker
```

然后由负责人把真实本地测试 key、Redis 密码、AI key 等填入 `.env.docker`。不要把你的本机 `.env.docker` 原样发给同事，除非走安全渠道并确认 key 可以共享。

启动：

```powershell
docker compose --env-file .env.docker up -d --build
docker compose ps
```

验收：

```powershell
docker compose exec backend-web python -m backend.scripts.db_diagnostics
```

浏览器访问：

```text
http://localhost:8080
```

## 8. 同事恢复到服务器

服务器建议目录：

```text
/srv/fdsmarticles/app/      # 代码
/srv/fdsmarticles/data/     # 运行数据
/srv/fdsmarticles/backups/  # 备份
```

迁移后确认：

```bash
ls -lh /srv/fdsmarticles/data/fudan_knowledge_base.db
ls -lh /srv/fdsmarticles/data/faiss_index_business/
find /srv/fdsmarticles/data/uploads -type f | wc -l
```

生产环境启动前必须准备 `.env.production`，并确认：

```bash
chmod 600 .env.production
docker compose --env-file .env.production -f docker-compose.prod.yml up -d --build
docker compose --env-file .env.production -f docker-compose.prod.yml ps
docker compose --env-file .env.production -f docker-compose.prod.yml exec backend-web python -m backend.scripts.db_diagnostics
```

如果服务器是 ARM64，FAISS 索引必须额外验证。验证失败就不要强行使用 Windows/x86 旧索引，应在服务器环境重建。

## 9. 不能交接或不能上传的内容

以下内容不要进 GitHub，也不要混在普通代码压缩包里：

- `.env`
- `.env.docker`
- `.env.production`
- `data/`
- `backups/`
- `archive/legacy_pre_knowledge_base/`
- `_publish_clean/`
- `.venv/`
- `frontend/node_modules/`
- `node_modules/`
- `logs/`
- 任何包含真实 Gemini key、CAS 凭据、Redis 密码、Sentry DSN、rclone 凭据的文件

如果这些内容必须交接，应走安全渠道，并在交接记录里写清楚接收人、用途、有效期和是否需要轮换。

## 10. 签收清单

交出方应确认：

- [ ] GitHub 代码仓库已推送到约定分支。
- [ ] `data/` 已通过单独渠道交出。
- [ ] 最新数据库备份 `backups/fudan_knowledge_base.handoff_20260602-142839.db.gz` 已随数据交出。
- [ ] 数据校验清单已随数据交出。
- [ ] `.env.production` 或生产配置已通过安全渠道交出，不在 GitHub。
- [ ] Gemini key、CAS、Redis、备份凭据已明确负责人。
- [ ] 说明 `archive/legacy_pre_knowledge_base/local_runtime_duplicates_20260602/` 不是生产运行数据。

接收方应确认：

- [ ] `data/` 文件数量和体积基本一致。
- [ ] 数据库 `PRAGMA quick_check` 为 `ok`。
- [ ] `articles=2146`，`business_article_count=2142`，`article_chunks=8900`，`article_chunk_embeddings=8900`。
- [ ] Docker 能启动，`backend-web`、`frontend`、`redis` 为 healthy。
- [ ] 首页、栏目、专题、搜索、AI/RAG、媒体访问均可用。
- [ ] 生产服务器备份策略已重新配置，不依赖本地历史备份。

## 11. 最终建议

最完整、风险最低的交接方式是：

1. GitHub 只交代码和交接文档。
2. 用移动硬盘、内网共享、SFTP/rsync 或对象存储交接 `data/` 整目录。
3. 随包附带 `backups/fudan_knowledge_base.handoff_20260602-142839.db.gz`。
4. 随包附带数据校验清单。
5. `.env.production`、Gemini key、CAS、Redis、备份凭据走单独安全渠道。
6. 同事恢复后按本文第 7 或第 8 节做 Docker 和数据库验收。
