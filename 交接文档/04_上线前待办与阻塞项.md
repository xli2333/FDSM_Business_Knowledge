# 上线前待办与阻塞项

生成日期：2026-06-01

## 总体判断

当前项目已经具备较完整的业务系统结构和生产化骨架，但不能直接上线开放。接下来要做的不是继续补一个简单功能，而是完成“生产环境落地闭环”。

上线前工作可以分为三类：

- P0：未处理则不能正式开放。
- P1：不完成会有明显运营、安全、稳定性风险。
- P2：不阻塞首发，但影响后续维护质量。

## P0：上线硬阻塞

### 0. 发布基准冻结

正式部署前必须先冻结发布基准，否则服务器上到底部署哪一版无法追溯：

- 记录发布分支、commit hash、是否包含 dirty tree。
- 明确当前工作区未提交改动是否纳入发布。
- 生成并保存发布包 manifest。
- 记录发布包生成时间和数据迁移批次。
- 不要把本地整个目录原样上传替代发布包流程。

### 1. 生产服务器架构确认

必须确认服务器真实架构：

```bash
uname -m
```

判断：

- `aarch64`：华为 Kunpeng/ARM64，当前方案可推进，但必须 ARM64 实机构建和验证。
- `x86_64`：可按普通 Linux Docker 方式推进。
- `loongarch64` 或其他：当前方案不能直接承诺可用，需要重新评估 Docker 基础镜像和依赖 wheel。

### 2. ARM64 镜像构建

如果服务器是 ARM64：

- 不能使用本地 Windows/x86 机器构建出的 amd64 镜像。
- 需要在 ARM64 服务器上 `docker compose build`，或使用 buildx 构建 `linux/arm64`。
- 需要确认后端镜像中的 Python 3.12、Node 20、`faiss-cpu==1.13.2`、`numpy`、`cryptography` 等均安装成功。

### 3. FAISS 索引验证或重建

当前 `data/faiss_index_business` 包含：

- `index.faiss`，约 142 MB。
- `index.pkl`，约 20 MB。

上线前必须在 ARM64 容器里验证：

```bash
python - <<'PY'
import pickle
import faiss
from pathlib import Path
p = Path('/data/faiss_index_business')
idx = faiss.read_index(str(p / 'index.faiss'))
with (p / 'index.pkl').open('rb') as f:
    meta = pickle.load(f)
print('faiss_loaded', idx.ntotal, type(meta).__name__)
PY
```

如果失败，需要在 ARM64 环境重建该目录。原因是 FAISS 索引是二进制索引，跨平台兼容不能只凭文件存在判断。当前 RAG 主路径配置为 `local_chunk`，FAISS 属于旧/辅助索引资产；这意味着 FAISS 失败时不一定让系统完全启动失败，但仍会影响相关检索路径，所以必须单独显式验收。

### 4. `.env.production` 生产配置重做

当前本地 `.env.production` 不可直接用于正式服务器。它存在本地域名、CAS 空值、Gemini 空值、缺 Redis 密码和 metrics token 等问题；本轮差异检查还发现它相对 `.env.production.example` 缺少 57 个键，包括资源限制、备份加密/异地目标、housekeeping、日志轮转、媒体上传限制、metrics、Redis、Sentry 等配置。

另外，本地 `.env` 和 `.env.docker` 已检出存在非空 Gemini key。交接、打包、截图或外发材料前必须确认这些 key 的安全边界，必要时轮换或撤销，不能把本地真实 key 当成生产交接方式。

必须根据 `.env.production.example` 在服务器重新创建：

- `APP_ENV=production`
- `SITE_BASE_URL=https://正式域名`
- `ALLOWED_ORIGINS=https://正式域名`
- `AUTH_BACKEND=cas`
- `CAS_ENABLED=1`
- `CAS_SERVER_URL=学校 CAS 地址`
- `CAS_SERVICE_BASE_URL=https://正式域名`
- `CAS_SERVICE_URL=https://正式域名/api/auth/cas/callback`
- `REDIS_PASSWORD=真实强密码`
- `METRICS_TOKEN=真实强 token`
- `GOOGLE_API_KEY` 或 `GEMINI_API_KEYS`
- `BACKUP_OFFSITE_TARGET`
- `BACKUP_ENCRYPTION_RECIPIENT`
- `SENTRY_DSN`，如果要求 Sentry 必开

### 5. 正式域名、HTTPS、CAS 白名单

需要外部系统配合：

- 申请或确认正式域名。
- 配置 DNS 指向服务器。
- 签发 HTTPS 证书。
- 将 CAS service URL 加入学校 CAS 白名单。
- CAS 回调必须与 `.env.production` 中 `CAS_SERVICE_URL` 完全一致。

没有正式域名和 CAS 白名单，生产登录不能完成。

### 6. 数据迁移

必须单独迁移：

- `data/fudan_knowledge_base.db`
- `data/Fudan_Business_Knowledge_Data/`
- `data/faiss_index_business/`
- `data/uploads/`
- `data/audio/`

不要把整个本地项目原样上传，因为本地项目包含缓存、历史日志、报告、旧材料、环境文件和本地数据库备份。代码走发布包，数据走数据迁移。

### 7. 生产预检

服务器上必须运行：

```bash
bash deploy/preflight.sh
```

这个脚本会拒绝：

- 非 production 环境。
- 非 HTTPS `SITE_BASE_URL`。
- `knowledge.example.edu` 占位域名。
- CORS 通配。
- debug auth 开启。
- metrics token 空或占位。
- Redis 密码空、弱或占位。
- CAS 配置缺失。
- 备份异地目标缺失。
- 备份加密收件人缺失。
- `.env.production` 权限过宽。
- 缺少关键数据目录。
- Compose 配置不合法。

### 8. 生产启动后验收

启动后必须运行：

```bash
bash deploy/acceptance_check.sh
```

至少覆盖：

- Docker 服务状态。
- `/healthz`。
- `/api/health?details=1`。
- `/api/home/feed?language=zh`。
- 生产拒绝 debug header。
- CAS login endpoint。
- DB diagnostics。
- worker healthcheck。

### 9. 发布包和 Docker 构建自包含性确认

本轮已处理一个自包含性风险：

- 原 `frontend/src/lib/pretextRichInline.js` 从 `../../../pretext-main/src/rich-inline.ts` 再导出函数。
- 该桥接文件未被当前主应用入口引用。
- `pretext/`、`pretext-main/` 和该桥接文件已移入 `unused_project_assets/`。
- `.dockerignore` 和 `deploy/create_release_package.ps1` 已排除 `unused_project_assets/`。

如果后续业务需要 Pretext 能力，不要直接从 `unused_project_assets/pretext-main/` 跨目录引用；应把所需代码 vendoring 到 `frontend/src/lib/`，或作为正式 npm 依赖纳入 `frontend/package.json`。

## P1：上线强建议完成

### 1. 前端 lint 收口

已知 `npm --prefix frontend run lint` 当前失败，结果为 50 problems（49 errors、1 warning）。它不一定阻止构建，但会降低代码质量和长期维护可控性。建议上线前收口到可通过，至少确认错误不影响生产路径。

已知类别包括：

- React hooks memoization/refs/render 阶段 setState 问题。
- Markdown 组件 unused node。
- Chat markdown unnecessary escapes。
- 页面 effect 中 setState 警告。

### 2. 全量后端测试和 AI key 测试

后端全量测试曾被 Gemini key 无效阻断。上线前应使用可用测试 key，或通过 monkeypatch/mock 隔离 Gemini 外部调用后重跑全量测试，避免 CI 依赖真实生产 key。当前 `backend/tests` 有 19 个测试文件，正式上线前不应只跑两个 focused tests。

重点确认：

- CAS/auth。
- 会员权限。
- 媒体 stream 鉴权。
- 编辑 CMS 发布链路。
- AI 摘要/翻译/排版。
- RAG 搜索。
- metrics。
- 数据库迁移。

### 3. 备份恢复演练

项目有备份脚本，但生产上线前必须完成真实演练：

- 生成一次备份。
- 备份压缩。
- GPG 加密。
- 异地上传。
- 下载或读取备份。
- restore 到临时目标。
- 执行 quick_check。
- 记录恢复耗时。

只有备份文件存在，不代表可恢复。当前 compose 中 backup 服务负责循环运行 `backup_sqlite.py --loop`，但没有配置 Docker healthcheck，仓库内也没有外部告警闭环；生产需要额外配置备份任务成败告警。

### 4. 监控告警

仓库提供 `/api/metrics`、健康检查和结构化日志，但没有完整 Prometheus/Grafana/Alertmanager 或等价外部告警配置。生产需要补：

- 容器健康告警。
- Nginx 4xx/5xx 告警。
- CPU/内存/磁盘/inode 告警。
- SQLite WAL 文件增长告警。
- Redis 健康告警。
- backup 成败告警。
- CAS 错误率告警。
- Gemini 错误率告警。
- async dead-letter queue 告警。
- 证书过期告警。

### 5. 媒体上传大小配置

后端默认允许：

- 音频 512 MB。
- 视频 2 GB。
- 文本/图片 20 MB。

但 `deploy/nginx/default.conf` 和 `deploy/nginx/fdsmarticles-https.conf` 当前都是 `client_max_body_size 200m`。如果生产要上传大视频，需要同步调整前端容器 Nginx 和宿主机 HTTPS 反代配置，否则会出现前端/后端配置允许但 Nginx 提前拒绝的问题。

### 6. 支付边界确认

当前计费模块是订单/订阅/checkout intent 骨架，`PAYMENT_PROVIDER` 默认 `mock`。如果上线范围包括真实收费，必须补：

- 支付网关。
- 回调验签。
- 订单状态机。
- 退款/取消/续订。
- 对账。
- 发票或财务接口。

如果首期只做内容权限和人工会员开通，需要在项目范围里明确“不开放自动支付”。

### 7. 发布版本冻结

当前工作区存在未提交修改和未跟踪文档。该项虽然列在 P1，但在正式对外开放前应按 P0 标准处理，否则发布版本不可追溯。上线前需要明确：

- 哪些代码改动纳入发布。
- 哪些只是本地临时改动。
- 发布包从哪个 commit/branch 生成。
- 发布包 manifest 留档。

当前可见未提交修改包括：

- `backend/services/ai_service.py`
- `backend/wechat_runtime/wechatOfficialPublisherService.mjs`
- `frontend/src/components/editorial/RichPreviewEditor.jsx`

还存在未跟踪文档材料。正式交接时应保留状态说明，避免后续接手者误以为仓库是干净发布态。

## P2：不阻塞首发但建议规划

### 1. PostgreSQL 迁移评估

当前主库仍是 SQLite。SQLite 在中小流量、读多写少场景可以工作，但如果用户量、并发写、媒体 CMS 操作、AI 异步任务显著增加，应评估 PostgreSQL。

不能只填 `DATABASE_URL` 就完成迁移，因为服务层大量 SQL 当前按 SQLite 直接执行。

### 2. CI 覆盖增强

`.github/workflows/cloud-release-qa.yml` 已经包含：

- 后端依赖安装。
- 依赖锁检查。
- compileall。
- 部分后端 focused tests。
- 前端 npm audit。
- 前端 build。
- Compose config 检查。
- shell 语法检查。

建议补：

- 前端 lint。
- 全量后端测试，并隔离外部 AI key 依赖。
- Docker build。
- ARM64 buildx 验证。
- release package manifest 检查。
- 部署脚本 dry run。

### 3. 文档归档整理

`todo/`、`docs/deployment_plan/`、`reports/` 历史记录很多。交接后建议归档成：

- “上线必读”。
- “历史任务记录”。
- “验收报告”。
- “废弃/仅参考”。

避免接手者被大量 round 文件淹没。

### 4. 品牌和页面细节

前端 `index.html` 默认 title 仍需人工确认是否替换为正式品牌。Logo、favicon、正式文案、版权和备案信息也需要上线前由业务方确认。

## 接下来推荐执行顺序

1. 确认服务器架构、系统版本、Docker/Compose 是否可用。
2. 明确正式域名、HTTPS、CAS 白名单、Gemini key、备份目标。
3. 冻结一个发布 commit，生成发布包。
4. 单独迁移 `data/`。
5. 在 ARM64 服务器构建镜像。
6. 跑 `preflight.sh`。
7. 启动生产栈。
8. 显式验证 FAISS。
9. 跑 `acceptance_check.sh`。
10. 跑 CAS、AI/RAG、媒体、会员、CMS、备份恢复专项验收。
11. 配置监控告警。
12. 小范围试用，再开放正式用户。
