# 测试验收运维与回滚清单

生成日期：2026-06-01

## 验收分层

正式上线验收应分四层：

1. 代码质量：构建、lint、单元测试、依赖锁。
2. 部署可用：Docker build、preflight、compose 启动、health check。
3. 业务可用：CAS、首页、搜索、文章、AI/RAG、CMS、媒体、会员、后台。
4. 运维可控：监控、日志、备份、恢复、回滚、告警。

## Docker/Compose 检查

本地或服务器：

```bash

docker compose --env-file .env.production.example -f docker-compose.prod.yml config >/dev/null
```

服务器真实生产配置：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml config >/dev/null
```

注意：真实 `.env.production` 缺 `REDIS_PASSWORD` 等变量时，生产 compose config 会失败，这是预期保护。正式服务器必须从 `.env.production.example` 重新生成并逐项补齐，不能沿用当前本地占位文件。

## 备份

生产至少每日备份：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml exec backup \
  python /ops/backup_sqlite.py
```

需要确认：

- 备份包含 SQLite 主库一致性快照。
- 压缩成功。
- 加密成功。
- 异地同步成功。
- 本地保留策略生效。
- 备份失败有告警。

注意：当前 backup 服务没有 Docker healthcheck，备份失败告警需要外部监控或任务系统补足，不能只依赖容器 `restart`。

## 恢复演练

恢复不要直接覆盖生产库。先恢复到临时目录：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml exec backup \
  python /ops/restore_sqlite_backup.py --help
```

实际命令按脚本参数执行，目标是：

- 解密备份。
- 解压备份。
- 恢复到临时 SQLite 文件。
- 执行 `PRAGMA quick_check`。
- 抽样查 `articles`、`media_items`、`article_chunks`。
- 记录恢复耗时。

## 日志和敏感信息检查

检查：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml logs backend-web | grep -i token
docker compose --env-file .env.production -f docker-compose.prod.yml logs frontend | grep -i token
```

目标：

- 不记录 CAS token。
- 不记录 `Authorization` header。
- 不记录 Gemini key。
- 不记录 Redis 密码。
- 媒体 stream URL 不在 Nginx access log 中暴露敏感 token。

## 回滚策略

### 普通代码回滚

适用于不涉及数据库结构变更的版本。

```bash
bash deploy/rollback_release.sh
```

回滚前确认：

- 上一个发布包还在。
- 上一个 `.env.production` 兼容。
- 数据库结构兼容。
- 当前版本备份已完成。

### 蓝绿发布

非 schema 变更可以使用：

```bash
bash deploy/blue_green_deploy.sh
```

如果需要切换宿主机 Nginx：

```bash
sudo SWITCH_NGINX=1 bash deploy/blue_green_deploy.sh
```

### 数据库结构变更

涉及以下内容时，不建议普通蓝绿：

- `backend/database.py`
- DDL。
- `ensure_runtime_tables`。
- 核心表 schema。
- 数据迁移脚本。

建议流程：

1. 公告维护窗口。
2. 停止写入。
3. 完整备份。
4. 执行迁移。
5. quick_check。
6. 启动新版本。
7. 验收。
8. 保留回滚方案。
