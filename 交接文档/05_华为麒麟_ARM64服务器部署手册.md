# 华为麒麟 ARM64 服务器部署手册

生成日期：2026-06-01

## 适用范围

本文适用于华为服务器、麒麟系统、Kunpeng/ARM64 架构的部署。核心前提是服务器架构为 `aarch64`。如果服务器不是 ARM64，而是 LoongArch64 或其他架构，需要重新评估。

## 先确认服务器信息

在服务器执行：

```bash
uname -a
uname -m
cat /etc/os-release
docker version
docker compose version
```

判断：

- `uname -m` 输出 `aarch64`：按本文继续。
- 输出 `x86_64`：按普通 Linux Docker 部署。
- 输出 `loongarch64`：不要按本文直接上线，依赖兼容风险明显更高。

## ARM64 的核心风险

### 1. 镜像架构风险

本地 Windows 电脑上的 Docker 镜像通常是 amd64。ARM64 服务器不能直接稳定运行 amd64 镜像。即使用 QEMU 模拟，也不建议作为生产方案。

正确方式：

- 在 ARM64 服务器上执行 `docker compose build`。
- 或在支持 buildx 的机器上构建 `linux/arm64` 镜像。

### 2. Python wheel 风险

后端依赖包括：

- `faiss-cpu==1.13.2`
- `numpy==2.4.4`
- `cryptography==46.0.7`
- `pillow==12.2.0`
- `google-genai==1.73.1`
- `langchain` 系列

这些依赖需要在 ARM64 Linux 下成功安装。不能只看本地 Windows 是否能跑。

### 3. FAISS 二进制索引风险

当前已有 `data/faiss_index_business/index.faiss` 和 `index.pkl`。即便 `faiss-cpu` 安装成功，也要确认旧索引能在 ARM64 加载。

如果加载失败，需要在 ARM64 环境重建索引。

当前业务 RAG 主路径配置为 `local_chunk`，FAISS 属于旧/辅助索引资产；所以页面和部分检索能跑不等于 FAISS 已经可用。ARM64 部署仍要把 FAISS 加载验证作为单独验收项。

### 4. 公众号 Node 运行时风险

后端镜像同时包含 `backend/wechat_runtime/`，其中 `package.json` 依赖 `@google/genai`、`sharp`、`undici`。`sharp` 是原生依赖，虽然 `package-lock.json` 中包含 linux arm64 相关包，但仍要在华为 ARM64 容器内实际跑 `npm ci`/镜像构建，并验证公众号排版渲染链路。不能只看 Python 依赖是否安装成功。

### 5. 性能和内存风险

SQLite、FAISS、Gemini 并发、媒体上传、前端 Nginx、Redis worker 会共同占用资源。ARM64 服务器性能不能只按“能启动”评估，还要做压测和资源监控。

## 本地准备

### 1. 生成代码发布包

在本地项目根目录：

```powershell
powershell -ExecutionPolicy Bypass -File .\deploy\create_release_package.ps1
```

产物：

- `_publish_clean/fdsmarticles-cloud-release.zip`
- `_publish_clean/fdsmarticles-cloud-release.manifest.txt`

发布包不包含 `data/`、`.env.production`、数据库、上传文件、`qa/`、`reports/`、`project_materials/`、`unused_project_assets/` 等。当前脚本没有排除 `docs/`、`todo/`、`交接文档/`；这些历史文档和交接文档本轮保留原位，如果服务器只需要最小运行包，出包前要调整脚本或检查 zip 内容。

### 2. 准备数据迁移

需要单独迁移：

```text
data/fudan_knowledge_base.db
data/Fudan_Business_Knowledge_Data/
data/faiss_index_business/
data/uploads/
data/audio/
```

数据约 5.02 GB，实际大小会随 SQLite WAL/SHM、上传文件和本地缓存变化。建议用 `rsync`、SFTP、对象存储、内网传输或运维指定方式。不要通过 Git 上传。

## 服务器初始化

### 1. 安装 Docker 和 Compose

由服务器运维按麒麟系统标准安装。安装后确认：

```bash
docker version
docker compose version
```

### 2. 创建目录

```bash
sudo mkdir -p /srv/fdsm/app
sudo mkdir -p /srv/fdsm/app/data /srv/fdsm/app/backups /srv/fdsm/app/logs
sudo chown -R "$USER":"$USER" /srv/fdsm/app
```

### 3. 解压代码包

```bash
cd /srv/fdsm/app
unzip /path/to/fdsmarticles-cloud-release.zip
```

确认存在：

```bash
ls Dockerfile.backend Dockerfile.backup docker-compose.prod.yml
ls backend frontend deploy
```

### 4. 上传数据

最终应满足：

```bash
ls /srv/fdsm/app/data/fudan_knowledge_base.db
ls /srv/fdsm/app/data/Fudan_Business_Knowledge_Data
ls /srv/fdsm/app/data/faiss_index_business/index.faiss
ls /srv/fdsm/app/data/faiss_index_business/index.pkl
ls /srv/fdsm/app/data/uploads
ls /srv/fdsm/app/data/audio
```

```bash
cd /srv/fdsm/app
cp .env.production.example .env.production
chmod 600 .env.production
```

必须修改：

```env
APP_ENV=production
APP_PORT=8080
SITE_BASE_URL=https://正式域名
ALLOWED_ORIGINS=https://正式域名
AUTH_BACKEND=cas
CAS_ENABLED=1
CAS_SERVER_URL=https://学校CAS地址
CAS_SERVICE_BASE_URL=https://正式域名
CAS_SERVICE_URL=https://正式域名/api/auth/cas/callback
REDIS_PASSWORD=真实强密码
METRICS_TOKEN=真实强token

GEMINI_API_KEYS=可选多个key
```

如果暂时没有 Sentry，但 `.env.production.example` 中 `SENTRY_REQUIRED=1`，要么配置 `SENTRY_DSN`，要么明确调整 `SENTRY_REQUIRED=0` 并记录风险。

## ARM64 构建

在服务器执行：

```bash
cd /srv/fdsm/app
docker compose --env-file .env.production -f docker-compose.prod.yml build
```

如果构建失败，重点看：

- `pip install -r requirements.lock.txt` 是否卡在 `faiss-cpu`、`numpy`、`cryptography`。
- Node 20 基础镜像是否支持当前 ARM64 系统。
- `backend/wechat_runtime` 的 `npm ci` 是否卡在 `sharp`、`@img/sharp-linux-arm64` 或网络拉包。
- 网络是否能拉取 Docker Hub 镜像。
- 是否需要设置国内镜像源。
- 前端构建是否引用了发布包没有包含的外部目录。本轮已将未使用的 `pretextRichInline.js` 桥接文件和外部 `pretext-main/` 参考工程隔离到 `unused_project_assets/`；正式构建前仍应用 `rg "pretext-main|pretextRichInline" backend frontend Dockerfile.backend frontend/Dockerfile docker-compose.yml docker-compose.prod.yml` 和 `npm --prefix frontend run build` 复核。`deploy/create_release_package.ps1` 中仅保留排除名单命中。

## FAISS 验证

构建后先不急着对外开放。启动后进入后端容器验证：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml up -d redis
docker compose --env-file .env.production -f docker-compose.prod.yml run --rm backend-web python - <<'PY'
import pickle
import faiss
from pathlib import Path
p = Path('/data/faiss_index_business')
print('exists', (p / 'index.faiss').exists(), (p / 'index.pkl').exists())
idx = faiss.read_index(str(p / 'index.faiss'))
with (p / 'index.pkl').open('rb') as f:
    meta = pickle.load(f)
print('faiss_loaded', idx.ntotal, type(meta).__name__)
PY
```

如果失败：

1. 不要忽略。
2. 先确认 `import faiss` 是否失败，还是索引读取失败。
3. 如果是索引读取失败，在 ARM64 环境重建 `data/faiss_index_business`。
4. 重建后重新跑上面的验证。

## HTTPS 和宿主机 Nginx

如果使用脚本：

```bash
sudo DOMAIN=正式域名 EMAIL=运维邮箱 APP_PORT=8080 bash deploy/bootstrap_https.sh
```

如果手工配置：

- 使用 `deploy/nginx/fdsmarticles-https.conf`。
- 替换 `knowledge.example.edu`。
- 替换证书路径。
- 确认 upstream 指向 `127.0.0.1:8080` 或实际 `APP_PORT`。
- 执行：

```bash
sudo nginx -t
sudo systemctl reload nginx
```

## 启动后验收

```bash
bash deploy/acceptance_check.sh
```

再补专项：

```bash
curl -fsS https://正式域名/healthz
curl -fsS https://正式域名/api/health?details=1
curl -fsS https://正式域名/api/home/feed?language=zh
```

CAS：

- 打开 `https://正式域名/login`。
- 点击 CAS 登录。
- 确认跳转学校 CAS。
- 登录后回到 `/login/cas-callback`。
- 确认 token 保存后用户状态正常。

AI/RAG：

- 访问搜索页，用业务关键词检索。
- 访问 `/chat`，提问一条能命中文章的问题。
- 后台查看 RAG 资产。
- 后端日志确认没有 Gemini key 错误。

媒体：

- 访问公开音频/视频。
- 验证付费媒体未登录不可播放。
- 验证有权限用户可播放。
- 确认 Nginx/backend 日志无 token 泄漏。

备份：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml exec backup python /ops/backup_sqlite.py
```

再用 `restore_sqlite_backup.py` 做临时恢复验证。

## 回滚

普通回滚：

```bash
bash deploy/rollback_release.sh
```

如果涉及数据库结构变更：

- 不建议两个版本并行共享 SQLite。
- 需要维护窗口。
- 先备份。
- 停旧版本。
- 恢复或迁移数据库。
- 再启动目标版本。
